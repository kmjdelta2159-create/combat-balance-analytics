# 검증 마일스톤 — 포켓몬式 JRPG (정답지 + 측정 계획)

## 0. 이 실험이 무엇인가

도구(전투로그 역설계 + 전문가 개입 → 복제 → 최적화)가 **실제 게임에서 작동하는가**를
처음으로 검증한다. 지금까지 7개 Phase는 합성 데이터 하나로만 왔다.

방식 — **known-answer 통제 실험**: 포켓몬 Gen6+ 전투 규칙을 충실히 참조 구현
(`pkmn_ref.py`)하고 거기서 로그를 생성했다. 도구는 *로그만* 보고 규칙을 역설계해야
한다. 정답(아래)을 우리가 알고 있으므로 회수 충실도를 정밀 채점할 수 있다.

이 실험의 결과가 다음 결정을 **데이터로** 답한다: L3 Plugin 레이어 구축 vs 표현력
보강 vs descope.

## 1. 정답 (Ground Truth) — 도구가 회수해야 할 규칙

정확한 코드 정의는 `pkmn_ref.py`. 요약:

### 1-1. 데미지 공식

```
base   = floor( ((2·Level/5 + 2) · Power · A / D) / 50 + 2 )      (Level=50)
damage = floor( base · STAB · TypeEff · Crit · Random )           (immune이면 0, 그 외 최소 1)
```

- **A / D** — 무브가 Physical이면 공격자 Attack / 방어자 Defense, Special이면 SpAtk / SpDef
- **Power** — 무브별 위력 (40~120). 캐릭터 스탯이 아니라 **무브 속성**
- **STAB** — 1.5 (무브 타입이 공격자 타입 중 하나와 일치할 때), 아니면 1.0
- **TypeEff** — 18타입 상성표의 곱. 방어 타입 1~2개 각각에 {0, 0.5, 1, 2} → 종합 {0, 0.25, 0.5, 1, 2, 4}
- **Crit** — 1.5 (확률 1/24), 아니면 1.0
- **Random** — 균등분포 0.85 ~ 1.00

### 1-2. 스탯 공식 (Lv50, EV=0)

```
HP   = floor((2·Base + IV)·50/100) + 50 + 10
Stat = floor( (floor((2·Base + IV)·50/100) + 5) · Nature )      Nature ∈ {0.9, 1.0, 1.1}
```

### 1-3. 그 외 규칙

- **턴순** — Speed 높은 쪽 먼저 (동점 랜덤)
- **승패** — HP ≤ 0 = 기절 = 패배 (1v1). 50턴 초과 시 HP% 높은 쪽 승
- **AI** — 그리디 (기대 데미지 최대 무브 선택)
- **타입 상성표** — 18×18 전체는 `pkmn_ref.py`의 `_SUPER`/`_NOTVERY`/`_IMMUNE` 참조

## 3. 실행 절차 (도구에서 할 일)

### Run A — Phase 7 SR (데미지 공식 역설계)
1. `pkmn_attack_log.csv` 업로드 → Step 2
2. 숫자 스탯으로 `Atk_Attack, Atk_SpAtk, Def_Defense, Def_SpDef, Move_Power` 매핑
3. gplearn 설치돼 있으면 "🔮 공식 자동 추측 실행" → 산출 후보를 **1-1 base 공식과 대조**
4. 화면 캡처해서 공유

### Run B — ML + 시뮬레이션 + Validation
1. `pkmn_battle_log.csv` 업로드 → Step 2
2. 스탯 `HP, Attack, Defense, SpAtk, SpDef, Speed` / 기믹 `Type1, Type2` / 타겟 `Is_Victorious`
3. 데미지 공식을 직접 입력 시도 — **포켓몬 공식을 엔진 공식창에 표현할 수 있는가?**
4. Step 2 프로파일링(ML 계수) → Step 6 시뮬레이션 → Validation Loop
5. 각 단계 화면 캡처 공유

## 5. 사전 예측 (실행 전 기록 — 나중에 실제와 대조)

정직하게, 실행 전 예상을 박아둔다. 예측 vs 실제의 차이가 진짜 발견이다.

- **Phase 7 SR (Run A)** — attack_log엔 Atk_Attack/Atk_SpAtk/Def_Defense/Def_SpDef 4개
  스탯이 있고 어느 쌍을 쓸지는 `Move_Category`가 결정한다. SR은 숫자 컬럼만 보므로
  category 조건 분기를 모른 채 4개를 섞어 적합 → **base 공식의 곱/나눗셈 구조는 부분
  회수하나 R²는 중간(STAB·타입·크리·난수가 미모델링 분산으로 남음)**. 예상 R² 0.5~0.8.
- **타입 상성 (C)** — 엔진 `element_chart`는 6속성·1.5/0.5 고정. 포켓몬은 18타입·
  {0,¼,½,1,2,4}. → **18타입 상성표는 표현 불가** (가장 명확한 표현력 구멍).
- **무브 시스템 (C)** — 데미지가 무브별 Power·Type·Category에 의존. 엔진의 단일
  `global_damage_formula`는 캐릭터 스탯 식이라 "무브마다 다른 위력/타입"을 한 식에
  못 담음. → **표현 불가 후보 2순위**.
- **Physical/Special 라우팅 (C)** — 한 공격이 category에 따라 다른 방어 스탯을
  때리는 이중 라우팅. 엔진에 이 개념이 있는지 불명 → 검증 포인트.
- **STAB (C)** — "무브 타입 == 공격자 타입이면 ×1.5". 조건부 자기참조 배율 →
  엔진 표현 가능성 낮음.
- **ML (A)** — 스탯 6개 → 승률. HP·방어·공격이 양(+)의 중요도로 나올 것. 단 ML은
  선형 계수라 데미지 공식 자체는 회수 못 함 (그건 Phase 7 SR 몫).
- **Validation 일치율 (D)** — 위 표현 불가 항목들 때문에 도구 시뮬이 포켓몬을
  충실히 재현 못 함 → 일치율 중간 이하 예상.

**예측 종합:** 도구는 *스탯→승률 경향*과 *base 데미지 구조*는 부분 회수하지만,
포켓몬을 정의하는 핵심(18타입 상성·무브별 위력·STAB)은 표현 못 한다. 즉 이 실험은
"L3/표현력 보강이 시급하다"를 데이터로 보여줄 가능성이 높다 — 단, 실제로 돌려봐야
확정된다.

---

## 6. 결과 → 결정

- 표현 불가 목록(C)이 길고 핵심적 → **L3 Plugin + 표현력 보강이 다음 작업**
- 도구가 대부분 회수하고 개입(B)이 가벼움 → descope(모듈 도구 착지) 검토 가능
- Validation 일치율(D)이 의외로 높으면 → 근사 충실도가 쓸 만하다는 강한 신호

도구를 돌리고 각 단계 화면을 공유해 주시면 채점표를 함께 채우겠습니다.
