# 검증 마일스톤 — 포켓몬式 JRPG · 결과 채점표

`검증_정답지_측정계획.md`의 측정 계획에 대한 실측 결과. Run A(SR)는 사용자 라이브 실행,
Run B(시뮬 충실도)는 클린룸 검증 harness로 측정했다.

---

## 0. 한 줄 요약

도구의 best-effort 충실도는 **전투별 승자 예측 71.6%**다. 완벽 메커니즘 시뮬레이터의
이론적 천장은 **95.8%**(나머지 4.2%는 불가피한 난수). 그 **24.2%p 격차 중 20.5%p가
타입상성 + STAB** 한 묶음에서 나온다. 즉 도구가 포켓몬을 재현 못 하는 이유의 85%는
"무브가 1급 엔티티가 아니라서"로 환원된다. → **L3/표현력 보강 결정을 데이터가 지지한다.**

---

## 1. 실행 메타 & 방법론

| 항목 | 내용 |
|---|---|
| 입력 로그 | `pkmn_battle_log.csv` 5000행 · `pkmn_attack_log.csv` 8000행 (Mono 토큰 재생성본) |
| Run A (SR) | 사용자 라이브 — Phase 7 기호 회귀 |
| Run B (충실도) | 클린룸 harness — 도구의 `modules/engine.py`(무수정)로 로그의 2500전투를 전부 재시뮬 |
| ML 회수 | 클린룸 numpy 로지스틱 회귀 (샌드박스 sklearn 부재 → 대체 측정) |
| 천장(Ceiling) | `pkmn_ref.py`를 난수 제거 결정론 모드로 재실행 → RNG 로그와 대조 |

### 1-1. 발견 — 도구의 Validation Loop는 전투별 채점을 못 한다

`modules/validation.py`의 `calculate_validation_score`는 **집계 통계만** 비교한다:
평균 데미지, 전체 승률 vs `target_col` 평균, 속성, 버프. `battle_log`엔 데미지·속성·버프
컬럼이 없으므로 overall 점수는 **"MC 1매치업 승률이 0.5에 얼마나 가까운가"** 단일 수치로
붕괴한다. 정답지 D열이 의도한 "전투별 승패 일치율"을 이 도구는 **구조적으로 측정 못 한다**.
(1차 측정 26%는 이 한계 + placeholder 공식의 복합 산물.) → 진짜 D열은 harness로 측정함.
**부수 로드맵 항목**: 도구에 per-battle backtest 검증 모드가 없다.

## 2. 채점표

### A — 역설계 자동 회수

| 경로 | 결과 | 판정 |
|---|---|---|
| Phase 7 SR | 후보 R² 0.20 / 0.13 / 0.09, 복잡도 313. base 공식 구조(`Power·A/D`) 미회수 | **실패** |
| ML 계수 (LR) | 6개 스탯 전부 양(+)의 계수. Speed +0.44 / Def +0.40 / HP +0.36 / SpAtk +0.35 / Atk +0.30 / SpDef +0.30. 테스트 정확도 66.9% | **부분 성공** |

SR 실패 원인: 포켓몬 데미지 분산은 범주형 곱셈 modifier(타입상성 0~4×·STAB·크리·난수)가
지배 → 숫자전용 gplearn이 ~20%만 설명. ML은 정성적으로 옳다 — 모든 부호가 맞고, **대체
가능한 공격 스탯(Attack/SpAtk)을 정확히 낮게 가중**한다(둘 중 하나만 쓰므로 개별 신호가 약함).
단 ML은 캐릭터-per-행이라 상대/매치업을 못 봐서 "평균 강함"만 학습 → ~67% 천장.

### B — 개입 필요량 (전문가가 손으로 넣어야 한 것)

| 개입 | 정도 | 비고 |
|---|---|---|
| 컬럼 매핑 (스탯/기믹/타겟) | 경 | 모든 게임 공통 |
| 데미지 공식 직접 작성 | 중 | SR이 자동 회수 실패 → 손으로 `22·P·A/D/50+2` 작성 |
| Physical/Special 라우팅 | 중 | 엔진 단일 공식 → `(attack+spatk)/(target_defense+target_spdef)` 평균으로 붕괴 |
| 대표 무브 위력 상수 P | 중 | 무브별 위력을 한 상수로 — 보정 필요 (단 정확도는 P에 거의 무감, §3 참조) |
| 타입 상성표 | **불가** | 아무리 개입해도 입력 불가 (C 참조) |

슬라이딩 스케일 상 포켓몬은 **고개입 끝단**. 게다가 최대 개입을 해도 충실도가 71.6%에서
막힌다 — "config가 많이 필요한" 문제가 아니라 **엔진 표현력 자체가 부족한** 문제다.

### C — 표현 불가 (★ L3/표현력 우선순위의 핵심 증거)

| # | 항목 | 왜 불가능한가 | 충실도 비용 |
|---|---|---|---|
| 1 | 18타입 상성표 | `element_chart`는 6속성·{1.5/0.5} 하드코딩. 게다가 element 모듈은 이름에 'element'가 든 컬럼만 읽음 → `Type1/Type2`는 picked up조차 안 됨 | 타입+STAB 합산 **20.5%p** |
| 2 | STAB | "무브 타입 == 공격자 타입 → ×1.5". 조건부 자기참조 배율, 무브 타입 개념 없음 | (위에 포함) |
| 3 | 무브별 위력/타입/카테고리 | 엔진의 단일 `global_damage_formula`는 캐릭터 스탯 식 — per-move 데이터를 담을 슬롯이 없음 | 위력분 **3.7%p** |
| 4 | Physical/Special 라우팅 | 공격 카테고리별로 다른 방어 스탯을 때리는 이중 라우팅 — 엔진엔 한 공식뿐 | (3에 포함) |

**공통 뿌리**: 1~4는 전부 **"무브/어빌리티가 1급 엔티티가 아니다"**로 환원된다. 무브가
타입·위력·카테고리를 갖는 객체가 되면 → 타입표(데이터 테이블)·STAB(조건부)·P/S 라우팅이
한꺼번에 표현 가능해진다. **이것이 아키텍처 키스톤이다.**

### D — Validation 일치율 (충실도)

| 측정 | 값 |
|---|---|
| 도구 best-effort (harness, avg 라우팅) | **71.6%** (1790/2500) |
| ─ phys 라우팅 / spec 라우팅 | 65.5% / 67.2% |
| ─ P 민감도 (×0.5~×2.0) | 70.0~72.7% — **거의 무감** |
| 이론 천장 (완벽 메커니즘 결정론) | **95.8%** |
| 천장 中 typeless (무브·P/S 있음, 타입·STAB 없음) | 75.3% |
| 불가피한 난수 (100 − 95.8) | 4.2% |
| 참고: 잘못된 공식 `attack - target_defense` | 42.6% (코인플립 이하) |

P 민감도가 거의 없다는 건 결정적이다 — **충실도를 만드는 건 데미지 크기가 아니라 A/D 비율
구조 + 속도**다. 공식 보정을 아무리 해도 71.6% 근처가 한계.

## 3. 핵심 발견 — 충실도 사다리 분해

```
 71.6% ──────────── 도구 best-effort (현재 표현력)
   │  +3.7%p  ← 엔진 자체 근사 (상수 위력, 평균 P/S 라우팅)
 75.3% ──────────── typeless 천장 (무브 위력·P/S 있다고 가정)
   │  +20.5%p ← 타입상성 + STAB  ★ 격차의 85%
 95.8% ──────────── 완벽 메커니즘 천장
   │  +4.2%p  ← 불가피한 난수 (크리 1/24 · 데미지 롤 0.85~1.0)
100.0%
```

24.2%p의 회복 가능 격차 중 **20.5%p(85%)가 타입상성+STAB** 한 묶음, 3.7%p가 무브
위력/라우팅. 즉 표현력 보강의 ROI가 압도적으로 한 곳(무브 엔티티 + 타입 테이블)에 몰려 있다.

## 5. 결정 — 데이터가 가리키는 방향

정답지 §6 분기 판정:

- **표현 불가 목록(C)이 길고 핵심적이다** → "L3 Plugin + 표현력 보강이 다음 작업" 분기 채택.
- descope 분기 기각 — 개입(B)이 가볍지 않고, 최대 개입으로도 71.6% 천장.
- Validation이 의외로 높지도 않음 — 71.6%는 typeless 천장(75.3%)에 거의 붙어 있어
  "엔진이 typeless 한계까지는 잘 쓰지만 그 위로 못 올라간다"를 뜻함.

**다음 작업의 키스톤**: 무브/어빌리티 1급 엔티티화. 이것이 풀리면 타입표·STAB·P/S
라우팅이 동시에 표현 가능 → 충실도 71.6% → ~95% 권역으로 도약. 우선순위 ROI가 한 곳에
명확히 몰려 있다는 게 이번 검증의 최대 수확이다.

남은 설계 질문: 무브 엔티티 + 타입 테이블을 (a) L2 엔진 표현력으로 직접 넣을지, (b) L3
Plugin 추상화를 먼저 세우고 그 안에 넣을지 — 다음 세션 설계 논의 대상.

## 6. 선택 — 라이브 UI 확인 (결정에는 불필요)

핵심 데이터는 위로 확정됐다. 도구 UI 자체를 한 번 더 보고 싶다면:

1. `pkmn_battle_log.csv` 업로드 → Step 2, 스탯 6개 / 기믹 `Type1,Type2` / 타겟 `Is_Victorious`
   매핑 → ML 프로파일링 계수가 위 LR 결과(전부 양, Speed·Def 상위)와 일치하는지 확인.
2. 공식창에 `22*182.8*(attack+spatk)/(target_defense+target_spdef)/50+2` 입력 → Step 6
   시뮬 → Validation Loop. 단 Loop는 §1-1 한계로 "MC 승률 vs 0.5" 단일 수치만 나옴.

이건 UI 배선 확인용일 뿐 — 채점표는 이미 완성됐다.
