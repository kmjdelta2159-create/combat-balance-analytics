# 무브 우선도 결정 프리패스 설계안 — P-트랙 부활

## 0. 이 문서의 위치

교체 모델(S1~S5) + 우선도 P1(무브 priority 필드)·P2(교체-우선 프리소트) 완료 후,
사용자가 "항상 최종목표 기준" 지침으로 §5의 **갈래 1(무브 우선도 결정 프리패스)** 을
선택했다. 최종목표는 범용 턴제 전투 시스템을 사용자 개입으로 복제하는 것이고, 우선도
무브(속도를 우회해 먼저 나가는 행동)는 Pokemon 전용이 아니라 이니셔티브·행동 우선
티어·ATB 등 턴제 전반의 일반 프리미티브다.

문제 정의(핸드오프 §5 정정). P1으로 무브 dict에 `priority` 필드가 들어갔지만 **잠복
상태다** — P2 예측기(`_switch_action_priority`)가 "이번 턴 교체 여부"만 정렬에 반영하고,
비-교체 유닛에는 0을 반환해 무브 우선도를 정렬에 안 넣는다. 그래서 우선도 무브를 골라도
행동 순서가 안 바뀐다. 본 설계는 그 클로저가 비-교체 유닛의 *예측된 무브의 우선도*를
반환하게 만들어 P1 필드를 살린다. 본 보조자는 코드를 직접 수정하지 않는다 — 정찰 →
설계안 → 사용자 검토 → 작은 PR 분할 → Antigravity 프롬프트 산출 → 검증 순으로 간다.

## 1. 현재 구조 (정찰 결과, Read 기준)

행동 순서 정렬(turn_manager.py L112~125). `SequentialTurnManager.run`은 라운드마다
participants를 속도 정렬하고, on_field 유닛 스냅샷 `acting_units`를 만든 뒤 다음 한 줄로
재정렬한다:

```
if self._action_priority is not None:
    acting_units.sort(key=lambda x: -self._action_priority(x))
```

이건 **단일 스칼라 안정정렬**이다. acting_units는 직전에 이미 속도순이므로, 동일
`action_priority` 값 안에서는 속도순이 보존된다(stable sort). 즉 `action_priority`가
하나의 숫자로 (교체 티어, 무브 우선도)를 모두 인코딩하면 추가 정렬 단계 없이 한 번의
재정렬로 "교체 먼저 → 우선도 무브 → 속도순"이 떨어진다. **turn_manager 변경 불필요.**

정렬 키 클로저(engine.py L1171~1174, run_simulation 스코프):

```
def _switch_action_priority(unit):
    if _will_voluntary_switch(unit, participants, game_config):
        return int((game_config or {}).get("switch_priority", 6))
    return 0          # ← 비-교체 유닛은 항상 0. 무브 우선도가 여기서 사라진다.
```

이 클로저는 `participants`·`game_config`·`build_ctx`(같은 스코프)에 접근 가능하다.

`_will_voluntary_switch`(L297)는 이미 **부작용 없는 read-only 교체 예측기**다 —
`_maybe_voluntary_switch`(L318, 실제 교체 수행·로그·on_field 변경)와 판정 조건이 일치한다.

결정 액션 두 개에 부작용 셸과 순수 코어가 섞여 있다.
- `_act_target_select`(L357): (셸) 비전투 트리거 스킵 + `_maybe_voluntary_switch` 실행 +
  battle_over 설정 + 로그 / (순수 코어) 상대 on_field 후보 → 사거리 필터 → target 태그
  정규화로 `ctx["targets"]` 결정(L374~413).
- `_act_move_select`(L451): (순수) setup_first 정책 분기 + 그리디 기대데미지 최대 무브
  선택. 입력은 char·current_target·sys_stats·game_config·formula_str, 출력은
  `ctx["current_move"]`. **상태 변경 없음**(읽기만).

`build_ctx`(L1104~1161)는 char gimmicks를 읽어 ctx dict를 구성할 뿐 **부작용이 없다**
(participants 비변경). turn 인자는 로그 문자열에만 쓰인다. → 예측기가 안전하게 재사용 가능.

무브 dict는 P1으로 `priority` 필드(int, 기본 0)를 갖는다(move_extraction.extract_moves L74).
movepool의 각 무브에 실려 있어 `move.get("priority", 0)`로 읽으면 된다.

## 2. 핵심 통찰 — 위험을 두 곳에 가둔다

(1) **순서 결합은 공짜다.** 스칼라 안정정렬이 이미 결합 티어를 처리하므로, 클로저가
비-교체 유닛에 무브 우선도를 반환하게만 바꾸면 된다. 교체 티어(기본 6) ≫ 통상 무브
우선도(작은 정수)라 교체가 여전히 앞선다. turn_manager 무변경.

(2) **유일한 실질 위험은 "예측 = 실행" 보장이다.** 예측기가 비-교체 유닛의 무브를 미리
골라 그 우선도를 읽어야 하는데, 예측이 고른 무브와 실제 실행이 고르는 무브가 어긋나면
정렬이 틀린다. 어긋남을 원천 차단하려면 예측과 실행이 **같은 선택 코드**를 호출해야 한다.
그래서 그리디 무브 선택·타겟 선택의 순수 코어를 함수로 추출하고, 실제 액션과 예측기가
둘 다 그 함수를 호출한다(코드 1벌 → 발산 불가).

## 3. 설계 (목표 구조)

세 조각이다. 모두 engine.py 안. turn_manager·move_extraction·step2 무변경.

(A) **순수 코어 추출 — 부작용 0.**
- `_select_targets_pure(char, participants, target_val, spatial_module, attack_range)` →
  targets 리스트. `_act_target_select`의 L374~413(상대 on_field 후보 → 사거리 → 태그
  정규화) 로직을 그대로 옮긴다. **switch·battle_over·로그는 넣지 않는다**(셸의 몫).
- `_select_move_pure(char, target, sys_stats, game_config, formula_str)` → 무브 dict 또는
  None. `_act_move_select`의 setup_first 분기 + 그리디 본체(L459~501)를 그대로 옮긴다.
  ctx 의존을 인자로 받게 바꾼 것 외 로직 동일.

(B) **두 액션을 코어 호출로 리팩터 — 행동 byte-동일.**
- `_act_target_select`: 트리거 스킵 + `_maybe_voluntary_switch` + battle_over + 로그 셸은
  유지하고, 후보·사거리·태그 부분만 `_select_targets_pure` 호출로 치환.
- `_act_move_select`: movepool 가드 + `ctx["current_move"]=...` 대입 셸만 남기고 선택
  본체를 `_select_move_pure` 호출로 치환.
  → 이 리팩터는 **동작을 바꾸지 않는다.** 하니스로 구·신 출력 동일성을 증명한다.

(C) **예측기 + 클로저 교체.**
- `_predict_action_priority(unit)`(클로저, run_simulation 스코프):
  1. `_will_voluntary_switch(unit, participants, game_config)`면 → switch_priority 티어 반환
     (현행 동일).
  2. 아니면 `build_ctx(unit, 0, participants)`로 unit의 target_val·spatial·attack_range·
     sys_stats·formula_str를 얻고, `_select_targets_pure`로 타겟 예측. 타겟 없으면 0.
  3. 타겟 있으면 `_select_move_pure(unit, targets[0], …)`로 무브 예측. 무브 None이면 0,
     아니면 `int(move.get("priority", 0))` 반환.
- `manager` 생성부의 `action_priority=_switch_action_priority`를 `_predict_action_priority`로
  교체(이름도 일반화). turn_manager는 그대로 이 스칼라를 안정정렬한다.

(turn=0은 build_ctx에서 로그에만 쓰여 예측에 영향 없음. 비전투 트리거 유닛은 예측 단계
타겟이 비어 0 — 행동도 안 하므로 정렬 영향 무의미.)

## 4. 회귀 0 증명 전략

- **미설정 경로.** 어떤 무브도 priority 컬럼이 없으면(전부 0) + switch_policy 미설정이면
  → 예측기가 전원 0 반환 → 정렬 키 전부 0 → 안정정렬이 속도순 보존 → **현행과 완전 동일.**
  (현행 클로저가 비-교체 시 0 반환하던 것과 같다.)
- **리팩터 동일성.** `_act_target_select`/`_act_move_select`의 구·신 버전을 outputs/ 하니스에서
  동일 입력(여러 케이스: movepool 유무·setup_first·사거리·AoE/Lowest/Single·복수 후보)에
  대해 출력 일치 비교 → byte-동일 증명.
- **우선도 역전 케이스.** 느린 유닛에 priority 큰 무브, 빠른 유닛에 priority 0을 줘서
  느린 쪽이 먼저 행동함을 하니스로 확인. 교체 유닛이 둘 다보다 앞섬도 확인.
- **예측=실행(싱글).** 상대 액티브 1마리면 그리디·타겟이 결정적이라 예측 무브 = 실행 무브.
  같은 순수 함수를 부르므로 구조적으로 일치. 합성 백테스트로 교차 확인.
- 그리고 라운드트립 MD5 + 클린룸 컴파일(파편 wrap) + 납품 후 Read 정독 + Grep 마커.

## 5. 정직한 한계

(a) **멀티 액티브(더블+).** 상대 액티브가 둘 이상이면 한 유닛의 그리디 타겟·무브가 같은
라운드 앞선 해결로 바뀔 수 있어 예측이 실행과 어긋날 수 있다. 싱글에서만 정확하다. 검증
범위·외부 PS replay 모두 싱글이라 현재 무관하나, 더블+ 충실도가 필요해지면 설계안
§2~3의 결정-후-해결 깊은 재구조화로 들어가야 한다(연기, 폐기 아님).

(b) **ON_SWITCH 진입 타이밍.** 본 작업은 진입 hook 타이밍을 안 건드린다 — 여전히 "다음
턴 시작" 근사다(갈래 2의 몫). 우선도와 독립이다.

(c) setup_first가 켜진 경우, 예측기도 같은 정책 분기를 타므로 setup 무브의 priority를 읽어
일관되게 정렬한다(특수 케이스 아님).

## 6. PR 분할

PR-P3. **결정 프리패스(코어 추출 + 예측기).** 위 §3 (A)(B)(C)를 한 PR로. 순수 코어 2개
추출 + 두 액션 리팩터(행동 동일) + 예측기 클로저 + manager 배선 교체. engine.py만.
회귀 0은 §4로 증명. turn_manager·move_extraction·step2 무변경.

PR-P4(후속, 선택). **우선도 입력 UI.** step2에서 priority 컬럼 매핑 + switch_priority
설정 노출(S5 동일 양식). 현재 priority 컬럼 자동 탐지는 P1에 있으나, 사용자가 명시
매핑·switch_priority를 조정하는 경로를 연다. P3와 독립이라 분리한다.

(P3가 한 PR로 다소 크면 코어 추출/리팩터(P3a)와 예측기 배선(P3b) 둘로 더 쪼갤 수 있다 —
P3a는 순수 리팩터라 회귀 0 자체 증명, P3b가 동작 변화를 도입. 사용자 선호에 따라.)
