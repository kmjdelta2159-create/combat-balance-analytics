## 0. 한 줄 요약 (현재 위치)

이번 세션은 "쓸 수 있을 정도 = 포켓몬 복제 기능 + 가독 UI"로 목표를 재정의하고(사용자 확정), 남은 작업을 **9개 PR 단위**(꼬리 F1~F5 + UI U1~U4)로 잡았다. 검증 표면 기준으로 **엔진 PR은 점근적(셀 귀속이 검증 방법)·UI는 배칭(앱 캡처가 비쌈)**으로 순서를 고정했다. 첫 PR **F1(Seismic Toss `fixed_damage`)을 적용·실측검증 완료** — Clefable의 Seismic Toss 3회가 엔진에서 dmg=100을 정확히 내고, 해당 턴 HP가 로그와 정확히 일치. 골든 회귀0 유지. 남은 건 held-out 구조적 12셀의 분류(F2 해저드 타이밍·F3 단건·신규 스탯스테이지 후보)와 UI 배칭이다.

## 1. 이번 세션 완료물

### (가) 목표 재정의 — "쓸 수 있을 정도"의 정의 확정
사용자 확정: **"1차목표인 포켓몬 복제가 가능한 수준의 기능 구현 + 가독성 있는 UI"**. 외부 셀프서브가 아니라, 이미 검증한 복제 루프가 *기능적으로 완결*되고 *읽히는* 상태. 결정 기준은 최종목표(임의 턴제 게임을 슬라이더로 복제)이고, 그 과정에서 현재 복제 대상(포켓몬)에 필요한 기능을 먼저 구현하는 것.

### (나) 작업 9단위 + 순서 고정
- 꼬리(엔진/데이터, 클린룸·run_b4 회귀0로 검증): **F1** Seismic Toss `fixed_damage` / **F2** 해저드 타이밍(T-1 윈도우 + forced 교체) / **F3** 단건 노이즈(Brave Bird recoil·Ice Shard lockedmove·반올림) / **F4** 검출기 별칭 1줄(`mechanism_detect._norm` sandstorm→sand) / **F5** 세트 per-corpus 스코핑(다전투 견고성).
- UI(Streamlit, 앱 캡처로만 검증): **U1** 트레이스-리플레이를 앱 뷰로(키스톤) / **U2** gen5 프리셋 로더 / **U3** step2 expander 가독성(프롬프트 이미 작성됨, 미적용) / **U4** Step3 복제 일치율 헤드라인.
- **순서 원칙**: 엔진 데미지 PR은 *같은 divergence 리포트*에 쓰고 서로 상호작용 → 점근적(F1→F2→F3, 각 적용 후 셀 귀속 확인). F4는 표면이 달라(쇼핑리스트) 직교 → 아무 때나. UI는 통째로 배칭(앱 캡처 한 번). 최소 묶음 = U1+U2+U3+F1, 완전 = +F2~F5·U4.

### (다) ★ F1 적용·실측검증 완료 — 1차목표의 "유일한 진짜 언어 부채" 닫힘
프롬프트 `꼬리_PR-F1_SeismicToss_fixeddamage_프롬프트.md`. 3파일 적용 확인(Read/Grep):
- `reference_gen5.py` L71 `'Seismic Toss': (0, 'phys', 'Fighting')`(타입=면역판정용) + L210 `FIXED_DAMAGE_MOVES = {'Seismic Toss': 100}`(CONTACT_MOVES 패턴의 무브명→부가속성 레이어; 확장: Night Shade=레벨, Dragon Rage=40, Sonic Boom=20).
- `battle_setup.py` L127 무브 dict에 `"fixed_damage": getattr(ref,"FIXED_DAMAGE_MOVES",{}).get(e["move"])`.
- `engine.py` L614-618 `_act_damage_calc`에서 fixed_damage override(raw_dmg=dmg=int(_fd), `ctx["fixed_damage"]=True`) + L636 분산 가드 + L694 `_act_element_mult`에서 효과배율 무시·면역(×0)만 존중.

설계 핵심: `fixed_damage`를 **무브 속성**으로 일반화(고정데미지 일가 전부 커버). 상성 배율·STAB는 무시하되 타입 면역은 존중(Ghost에 0).

**실측검증(`run_f1diag.py`, 앱사이드, 레지스트리 몽키패치로 원본 무수정 런타임 관찰):**
- [B] build_trace_actions: Seismic Toss 3건에 `fixed_damage=100` 부착, 키 `(20,'Clefable')→Skarmory`·`(21,'Clefable')→Skarmory`·`(22,'Clefable')→Scrafty`. actor 'Clefable'은 참가자 id에 존재(키 매칭 OK).
- [D] 런타임: 세 턴 전부 `[DMG] dmg=100 flag=True` — `_act_damage_calc` 도달·override 발동 확인.
- [E] HP 일치: **Skarmory T20 log=3 eng=3·T21 log=0 eng=0(기절)·Scrafty T22 log=59 eng=59 — 전부 정확히 일치.** F1이 레벨고정 100을 정확히 적용·반영.
- 골든 `run_b4` 회귀0 유지(골든엔 Seismic Toss 없음).

### (라) 이번 세션의 교훈 — 오판 기록
한동안 "F1 적용했는데 run_xval 출력이 안 바뀜 → 미발동"으로 진단했으나 **오판**이었다. 원인: 비교 대상으로 삼은 "베이스라인" run_xval이 사실 *이미 post-F1*이었다 — 이 세션의 run_xval은 전부 F1 적용 후였고, pre-F1 런이 존재하지 않아 "안 바뀜"은 비교 대상이 없었던 것일 뿐. 교훈: **"출력 불변 = 미발동"은 pre/post 런이 실제로 갈릴 때만 성립.** 진단 스크립트(런타임 훅 + HP 표면)가 이 오판을 깨고 F1 정상을 확증했다.

## 2. 남은 작업 — held-out 구조적 12셀의 정체

F1로 Seismic Toss 턴은 닫혔고(이미 일치), 남은 **구조적 12셀은 Seismic Toss가 아닌 다른 메커니즘**이다. `run_f1diag.py` [E] 및 직전 run_xval에서 본 단서:
- **Scrafty T24 log=81 eng=67(엔진 14 *과*데미지)** — Scrafty가 턴22 **Bulk Up(방어+1)**을 썼는데 엔진이 그 방어 상승을 트레이스-리플레이로 안 들고 가면 이후 물리타가 과하게 박힘. **신규 후보: 스탯 스테이지 추적**(F2/F3 어디에도 없던 항목).
- 큰 갭들(노이즈 아님 가능성): **Garchomp T10 log=21 eng=82(61차)**·**Abomasnow T9 log=63 eng=100(37차)**·**Reuniclus T29 log=34 eng=67(33차)**. 평결 §6의 "Ice Shard 과소(Garchomp lockedmove)"가 T10일 가능성. 나머지는 미상.
- 작은 갭: Skarmory T16/T18(엔진 ~12 과소).

**그래서 다음 권고 = 12셀 먼저 분류**(진단 스크립트 1회 확장: 각 셀의 그 턴 사건/관련 무브를 코퍼스에서 붙여 어느 게 해저드 타이밍 F2·lockedmove F3·스탯스테이지 신규인지 크기 순 정렬). "점근 꼬리=작은 노이즈" 전제가 일부 틀릴 수 있어, 분류 후 우선순위를 재확정한다. 분류 없이 계획대로 F2(해저드 타이밍)부터 가는 선택지도 유효.
