## 1. 정답 (Ground Truth) — 도구가 회수해야 할 규칙

정확한 코드 정의는 `ff7_ref.py`. 요약:

### 1-1. 물리 데미지 공식

```
PhysAtk = Strength + WeaponAttack
Base    = floor( (PhysAtk * (PhysAtk + Level)) / 32 )
Damage  = floor( Base * (512 - TargetVitality) / 512 * Power / 16 )
```

- **PhysAtk** — 캐릭터 Strength + 무기 Attack (장비 보너스)
- **Power** — 무브별 위력 (1~200). 무기 일반 공격은 Power=16, 특수 기술은 64~200
- **(512 - Vit) / 512** — 방어자 Vitality가 클수록 데미지 감쇠 (Vit=512면 0)
- **/ 16** — Power 정규화 (Power=16이면 1배, Power=32면 2배)

크리·난수 추가:
```
Final = floor( Damage * Crit * Random )
- Crit = 2.0 (확률 Luck/64), 아니면 1.0
- Random ∈ [0.9375, 1.0625]  (±~6.25%, 단순화)
```

### 1-2. 마법 데미지 공식

```
MagAtk  = Magic + WeaponMagicBonus
Base    = floor( 6 * (MagAtk + Level) )
Damage  = floor( Base * (512 - TargetSpirit) / 512 * SpellPower / 16 )
ElemMult ∈ {0 (무효), 0.5 (저항), 1.0, 2.0 (약점), -1.0 (흡수)}
Final   = floor( Damage * ElemMult * Random )
```

- **MagAtk** — 캐릭터 Magic + 무기 Magic 보너스
- **SpellPower** — 마법별 위력. Fire 1=14, Fire 2=42, Fire 3=120 같은 등급
- **(512 - Spr) / 512** — 방어자 Spirit이 클수록 마법 감쇠
- **ElemMult** — 약점 2x, 저항 0.5x, 흡수 -1x(회복), 무효 0

### 1-3. 스탯 구성

```
HP      = Base + (Level - 1) * HpGrowth      (Level 50 → HP 약 1500~4000)
MP      = MpBase + (Level - 1) * MpGrowth
Strength, Magic, Vitality, Spirit, Dexterity, Luck ∈ [1, 99]
```

장비 보너스 (별도 컬럼):
```
WeaponAttack ∈ [10, 100]
WeaponMagic  ∈ [0, 60]
WeaponElement ∈ {None, Fire, Ice, Lightning, Earth, ...}  (무기 부여 속성)
ArmorDefense ∈ [10, 80]
ArmorSpirit  ∈ [0, 30]
```

### 1-4. 그 외 규칙

- **턴순** — Dexterity 높은 쪽 먼저 (동점 랜덤). ATB의 단순화
- **승패** — HP ≤ 0 = 기절 = 패배 (1v1). 60턴 초과 시 HP% 높은 쪽 승
- **AI** — 그리디 (기대 데미지 최대 액션 선택). MP 부족 시 물리만 선택
- **속성표** — 9속성. 캐릭터의 ArmorElement·종족 약점이 곱연산
- **Limit Break** — 받은 데미지 누적이 max HP에 도달하면 게이지 full. 다음 일반 공격이
  자동으로 1.5x ~ 2.0x 강력 기술로 변환

## 5. 사전 예측 (실행 전 기록 — 나중에 실제와 대조)

- **Phase 7 SR (Run A)** — attack_log는 9개 숫자 스탯이 있고 어느 쌍을 쓸지는
  `Move_Category`(Physical/Magic)가 결정. SR은 category 조건 분기를 모른 채 9개를
  섞어 적합 → **base 공식의 (Atk·(Atk+L))/32 꼴은 부분 회수, R² 0.5~0.75**.
  Pokemon 시 R² 와 비슷.
- **Physical/Magic 라우팅 (C)** — 한 공격이 category에 따라 다른 공·방 스탯을 때리는
  이중 라우팅. Pokemon에서 검증 포인트였던 부분. **FF7도 동일 한계 예상**.
- **9속성표 (C)** — 엔진 `element_chart`는 6속성 + 1.5/0.5 고정. FF7은 9속성 + {0,
  0.5, 1, 2, -1}. → **9속성표 표현 불가 (특히 흡수 -1)**. Phase 8b의 type_table로
  9속성은 표현 가능하지만 **흡수(-1)는 양의 배율만 받는 구조라 표현 불가**.
- **Limit Break (C)** — "받은 데미지 누적 → 게이지 full → 다음 공격 강화". 조건부
  resource + state machine. **Phase 11 영역, 본 reference에서 표현 불가 예상**.
- **WeaponAttack 보너스 (B)** — Strength + WeaponAttack 합산. 캐릭터 스탯과 장비
  스탯의 합산. ML이 둘을 따로 학습 vs 합산해서 학습하는지 검증.
- **ML (A)** — 스탯 12개 → 승률. HP·Strength·WeaponAttack·Magic이 양의 중요도로
  나올 것. 단 ML은 선형 계수라 비선형 데미지 공식은 회수 못 함.
- **Validation 일치율 (D)** — 표현 불가 항목들(9속성 흡수, Limit Break,
  Physical/Magic 라우팅) 때문에 시뮬이 FF7 충실히 재현 못 함 → **per-battle backtest
  55~65% 예상** (Pokemon 66.2% 대비 약간 낮음, FF7 메커니즘 다양성 때문).

**예측 종합:** 도구는 *스탯→승률 경향*과 *base 데미지 구조*는 부분 회수. Pokemon과
유사한 패턴으로 Physical/Magic 라우팅이 표현 불가 핵심 한계. 9속성 흡수와 Limit
Break는 새로운 표현 불가 후보. 슬라이더 원리(사용자 개입으로 보강 가능)는 채널 매핑
+ type_table로 일부 시연 가능.

## 7. 단순화 가정 (FF7 원작 대비)

본 reference는 다음을 단순화했다:

1. **ATB → 순수 턴제**: 원작은 Dex 기반 ATB 게이지. 본 reference는 Dex 정렬 후 1턴씩.
2. **3v3 → 1v1**: 원작은 3인 파티. 본 reference는 1v1 (Pokemon format 일관성).
3. **MP 제한 단순화**: 원작은 spell마다 MP 비용. 본 reference는 매 턴 MP 회복 가정.
4. **Materia 시스템 단순화**: 원작은 무기/방어구의 Materia 슬롯으로 마법·속성 부여.
   본 reference는 캐릭터에 미리 일부 마법풀 + WeaponElement 1개.
5. **상태이상 제외**: Poison/Sleep/Silence 등은 Phase 11 영역.
6. **소환수 제외**: Summon은 단발 강력 기술이지만 일반 공격 패턴으로 추상화.
7. **Limit Break는 게이지·발동만**: 원작 캐릭터별 고유 Limit 기술은 generic 강화 공격으로.

이 단순화는 의도적. 100% FF7 시뮬레이터가 목적이 아니라, **도구가 JRPG 클래스 메커니즘
(속성 시스템 + Physical/Magic 라우팅 + 자원 게이지)을 reverse-engineer 할 수 있는가**를
known-answer로 검증하는 게 목적.

검증의 타당성은 단순화의 충실도가 아니라 known-answer가 명확히 정의되었는가에 달려 있다.
