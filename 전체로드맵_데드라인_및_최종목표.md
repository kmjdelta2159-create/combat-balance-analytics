# 전체 로드맵 — 데드라인부터 최종 목표까지

## 0. 이 문서의 위치

오늘은 2026년 5월 27일이고, 과제 제출 데드라인은 6월 9일이다. 13일 남았다. 데드라인 이후의 작업은 데드라인이 없지만, 최종 목표(이 세상에 없는 게임이라도 턴제 구조면 사용자 개입 비중 조절로 복제 가능한 도구)까지 도달하기 위한 trajectory가 필요하다.

이 로드맵은 두 구간으로 나뉜다. 첫째는 데드라인까지의 13일이며, 시연 가능한 vertical slice를 만든다. 둘째는 데드라인 이후의 4~5개월이며, vertical slice에 살을 붙여 최종 목표에 도달한다.

핵심 architecture는 사용자가 제안한 "가중치 기반 동적 상태 주도 UI"이다. 엔진은 순수 데이터를 방출하고, UI 레지스트리가 컴포넌트를 등록하며, 사용자 가중치 슬라이더가 비중을 조절하고, 동적 레이아웃 매니저가 화면을 즉석에서 조립한다. 이 architecture를 6/9까지 골격으로 세우고, 이후엔 레지스트리에 컴포넌트만 추가하면서 살을 붙인다.

## 1. 데드라인까지 (5/27 ~ 6/9, 13일)

### 1-1. 목표

시연 가능한 vertical slice를 완성한다. 구체적으로는 Pokemon, Final Fantasy 7, Hearthstone 세 게임의 전투 로그를 도구가 자동 분석하고, 각 로그의 내용에 따라 UI 컴포넌트가 가변되는 모습을 라이브로 시연한다. 사용자는 사이드바 슬라이더로 컴포넌트 가중치를 조절할 수 있다.

이 vertical slice가 데드라인 이후 trajectory의 골격이 된다. 6/9까지 만든 모든 코드는 영구 자산이며, 이후 phase에서 재작업되지 않는다.

### 1-3. 13일 동안 만들지 않는 것

다음 항목은 데드라인 이후로 미룬다. 13일에 욱여넣으면 검증이 압축되어 silent regression이 생긴다.

상태이상 (Status Effects) 같은 메커니즘별 살은 데드라인 이후 첫 작업으로 미룬다. 가중치 기반 UI 골격이 세워지면 상태이상은 컴포넌트 1개 추가일 뿐이라 무리하지 않는다.

밸런스 진단 엔진, 캐릭터 빌드 추천, 팀 빌드 메타 분석도 미룬다. 이들은 후속 phase의 본격 작업이다.

Python plugin escape hatch는 비대중 게임을 위한 최후 수단이며 데드라인 이후 trajectory의 마지막 단계다.

### 1-4. 13일 plan의 위험

5/31~6/1 사이의 Antigravity 검증 사이클이 가장 위험하다. step6_dashboard.py가 1321줄이며 부분 refactor 분량이 Phase 9b 수준(195줄 블록 교체)에 준한다. 검증에서 결함이 발견되면 6/7까지 fallback 여유가 있지만 빠듯하다.

5/29까지의 reference 게임 작업이 늦어지면 가중치 UI 작업이 밀린다. 이 경우 Hearthstone을 단순화하거나 (HP와 공격력만, 카드는 placeholder) 6/2의 컴포넌트 등록을 2게임만 우선 처리한다.

자동 감지 휴리스틱이 6/2~3에 작동하지 않으면 게임별 default preset만 적용하여 사용자가 게임 선택 후 가중치 수동 조절하는 모드로 fallback한다. 시연 가치는 다소 약해지지만 데모는 가능하다.

## 2. 데드라인 이후 (6/10 ~ 약 6개월)

### 2-1. trajectory 원칙

데드라인 이후의 모든 작업은 가중치 기반 UI 골격에 컴포넌트를 추가하는 형식이다. 메인 코드 수정이 0이고 registry 등록 한 줄과 컴포넌트 함수 하나를 추가하는 것으로 새 기능이 들어간다. 이게 OCP가 보장하는 trajectory의 본질이다.

각 후속 phase는 독립적이며 우선순위는 사용자 결정에 따라 조정 가능하다. 다만 아래 순서가 의존성과 가치 측면에서 합리적이라고 본다.

### 2-2. 후속 phase 목록

**Phase 11.1 — 상태이상 (Status Effects) 컴포넌트 (예상 2~3주)**

상태이상은 슬라이더 원리에서 가장 갈증이 큰 영역이며 후속 첫 작업으로 적합하다. game_config dict에 status_effects 키를 추가하고, step2에 "상태이상 추가" 폼을 만들며, engine의 PASSIVE_START와 ON_HIT 단계에서 처리하는 로직을 추가한다. UI 레지스트리에 상태이상 시각화 컴포넌트를 등록한다.

reference 구현으로 Pokemon의 poison, burn, sleep, paralysis, freeze를 다룬다. 도구의 known-answer 충실도가 92.6%에서 95% 이상으로 올라갈 가능성이 있다.

**Phase 11.2 — 무브 우선도 (Move Priority) 컴포넌트 (예상 1주)**

무브 dict에 priority 필드(정수, 기본 0)를 추가하고 turn_manager의 정렬 로직을 확장한다. UI 컴포넌트는 턴 순서 시각화이다.

상태이상보다 작은 작업이며 빠르게 끝낸다.

**Phase 11.3 — 다중 hit 및 명중률 (Multi-hit / Accuracy) 컴포넌트 (예상 1주)**

무브 dict에 hit_count와 accuracy 필드를 추가하고 stochasticity 모듈을 확장한다. engine의 MOVE_SELECT에서 accuracy roll을 처리하고 APPLY_DAMAGE를 hit_count 루프로 바꾼다. UI 컴포넌트는 명중률 그래프와 hit 분포 히스토그램이다.

**Phase 11.4 — 날씨 및 필드 (Weather/Field) 컴포넌트 (예상 1.5주)**

BattleState에 weather와 field 슬롯을 추가하고, 모든 액션에 적용되는 modifier hook을 구현한다. Pokemon의 rain/sun/sandstorm/hail을 reference로 구현한다. UI 컴포넌트는 날씨 indicator와 영향 받는 액션 표시이다.

여기까지 끝나면 4 메커니즘이 모두 1급 시민이 되며, Pokemon 천장(95.8%)을 재돌파할 가능성이 있다.

**Phase 11.5 — 자동 감지 알고리즘 강화 (예상 2주)**

데드라인까지 만든 자동 감지는 컬럼명 기반 단순 휴리스틱이다. 이 phase에서는 로그 패턴 마이닝을 통해 더 깊은 감지를 구현한다. HP가 매 턴 일정량 감소하는 패턴을 보면 DOT 후보를, 속도가 turn-to-turn 변동하면 paralysis 후보를, 턴이 skip되는 패턴을 보면 sleep 후보를 자동 제시한다.

이 phase가 슬라이더의 역설계 측면을 진짜로 강화한다. 사용자에게 "이 메커니즘이 있을 가능성"을 제시하고 사용자가 confirm/reject한다.

**Phase 11.6 — 밸런스 진단 컴포넌트 (예상 2~3주)**

도구의 세 번째 목표인 "복제본을 밸런스 분석용으로 최적화"의 본격 작업이다. overpowered 캐릭터 자동 감지, dominant strategy 마이닝, 구체적 밸런스 제안 생성을 다룬다. UI 컴포넌트는 진단 리포트와 제안 카드이다.

기존 Phase 9b의 스탯 배분 최적화 위에 빌드된다.

**Phase 11.7 — 결과 export 컴포넌트 (예상 1주)**

PDF나 HTML로 분석 결과를 출력하는 컴포넌트이다. 발표 자료 자동 생성도 포함한다. 평가자나 외부 협업자에게 분석 결과를 들고 갈 수 있게 한다.

**Phase 11.8 — Python plugin escape hatch (예상 1주)**

표준 라이브러리로 표현 불가능한 game-specific 메커니즘을 위한 최후 수단이다. typing.Protocol 기반의 작은 인터페이스이며, 사용자가 자기 게임 고유 메커니즘을 Python 클래스로 작성해 등록한다.

이 phase는 작지만 trajectory의 "이 세상에 없는 게임도 다룰 수 있다" 조건의 마지막 안전망이다.

**Phase 11.9 — 비대중 장르 reference 추가 (게임당 1~2주)**

SRPG (Fire Emblem 또는 Tactics), 가챠 (FGO 또는 HSR), 추가 덱빌더 (Slay the Spire) 등의 reference를 추가한다. 데드라인까지의 3게임(Pokemon, FF7, HS)은 대중성이 우선이라 메커니즘 다양성이 부족하다. 이 phase에서 도구의 장르 커버리지를 확장한다.

게임당 1~2주이며, 모두 다 추가하면 6~10주가 추가된다. 사용자 우선순위에 따라 선별한다.

### 2-3. 후속 phase 우선순위

순서는 위와 같이 11.1부터 11.9까지가 합리적이지만, 사용자 필요에 따라 재정렬 가능하다. 예를 들어 사용자 본인 게임이 SRPG라면 11.9의 SRPG 추가를 11.4 직후에 두는 것이 자연스럽다.

밸런스 진단(11.6)을 더 일찍 당기는 것도 가능하다. 다만 4 메커니즘이 1급 시민이 되기 전에 밸런스 진단을 만들면 진단 대상이 좁아진다.

### 2-4. 총 기간 추정

데드라인 이후 약 4~5개월이며, 비대중 장르 reference를 다 추가하면 5~6개월이다. 매주 약 20시간 작업 가정이다. 작업 강도가 낮아지면 비례 연장된다.

## 3. 최종 도달 모습

### 3-1. 6개월 후의 도구

4 Layer가 모두 작동한다. Layer 1 역설계 엔진은 데미지 공식, 스탯 상관, 4 메커니즘, 격자/위치 메커니즘, 자원 메커니즘을 모두 자동 감지하거나 사용자 폼으로 받는다. Layer 2 개입 엔진은 dict pattern 기반의 메커니즘 추가가 form one-click이며 표준 preset 라이브러리가 풍부하다. Layer 3 시뮬레이션 엔진은 전투 시뮬, 스탯 최적화, 캐릭터 빌드 추천, 밸런스 진단, 팀 빌드 메타 분석을 모두 다룬다. Layer 4 UI는 가중치 기반 동적 상태 주도 architecture로 작동한다.

장르 커버리지는 JRPG, 가챠, SRPG, 덱빌더 모두 다루며, 각 장르에 최소 한 개의 reference 게임이 정량 검증된다.

### 3-2. 슬라이더 원리의 실현

사용자가 도구에 임의 턴제 게임의 로그를 업로드하면 두 가지가 일어난다. 첫째는 도구가 로그에서 메커니즘을 자동 감지하여 사용자에게 후보 제시한다. 사용자는 confirm/reject한다. 둘째는 도구가 표현 불가능한 메커니즘은 사용자가 form으로 직접 입력한다. form으로도 불가능한 극단적 메커니즘은 Python plugin으로 작성한다.

이게 슬라이더의 양 끝이며, 도구는 그 사이 어디든 사용자가 선택할 수 있게 한다. 대중적인 게임일수록 자동 감지 비중이 높고, 비대중적인 게임일수록 사용자 개입 비중이 높다. 양 끝이 연속이라 카테고리 점프가 없다.

### 3-3. 최종 완성도

이 로드맵을 다 마치면 도구는 약 75~85% 완성도에 도달한다고 추정한다. 100%가 아닌 이유는 다음이 여전히 부족하기 때문이다.

검증 다양성에서 reference 게임은 5~6개에 도달하지만, 도구를 일반 공개하려면 더 많은 reference가 필요하다. AI 행동 모델링은 그리디 휴리스틱 수준이며, 실제 게임의 AI 의사결정을 정확히 복제하려면 별도의 학습 인프라가 필요하다. 메타 진행과 멀티배틀 시나리오는 다루지 않는다 (단일 전투 단위).

이 부족분은 사용자의 사용 패턴에 따라 우선순위가 결정되며, 현재 시점에서 후속 phase를 미리 정의하지 않는다.

## 4. 위험과 가정

### 4-1. 위험

가장 큰 위험은 데드라인까지의 13일 plan에서 5/31~6/1의 Antigravity 검증 사이클이다. 결함 발견 시 6/7까지 fallback이 있지만, 결함이 architecture 수준이면 plan B (수동 refactor)가 필요하다.

두 번째 위험은 데드라인 이후 사용자의 작업 시간 가용성이다. 매주 20시간이 안 되면 6개월이 늘어난다. 다만 trajectory가 정해져 있어 phase 단위로 끊어 진행 가능하다.

세 번째 위험은 도구가 향후 사용자의 실제 작업에 충분히 활용되지 못해서 trajectory가 도중에 폐기되는 경우다. 이는 6/9 데모 평가 결과와 사용자 본인 게임 분석 시도 결과에 따라 결정된다.

### 4-2. 가정

데드라인까지의 작업은 보조자(나)가 spec과 Antigravity 프롬프트를 작성하고 사용자가 Antigravity에 전달해 납품 받는 패턴을 유지한다. 검증은 보조자가 수행한다.

데드라인 이후의 후속 phase도 동일 패턴이 유지된다고 가정한다. 사용자가 직접 코드를 짜는 모드로 전환하면 trajectory 속도가 달라진다.

가중치 기반 UI architecture는 5/30 설계 문서에서 unaddressed corners를 흡수한 형태로 정의되며, 이후 phase에서 이 architecture가 깨지지 않는다고 가정한다.
