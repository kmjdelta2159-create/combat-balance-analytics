# 1차목표: 포켓몬 복제 커버리지 종료판정

최종목표: DB 로그와 전문가 개입을 통해 스탯 기반 턴제 전투 게임을 범용적으로 복제하는 시뮬레이터.
1차목표: 그 최종목표로 가는 첫 실증 대상으로 포켓몬 전투를 복제할 수 있는 기능군과 검증 루프를 확보하는 것.

> **주의:**
> 포켓몬은 최종 대상이 아니라 첫 실증 대상입니다. DB-log IR/backtest는 특정 로그 하나를 맞추기 위함이 아니라, 스탯 기반 전투의 복제 기능을 검증하는 generic한 구조를 보장하기 위한 도구입니다.

## 3. 남은 작업

기능 누락이 아니라 코퍼스 처리 및 제품화 관점에서 분리된 남은 단계들입니다.

### 1차 종료 전/직후 정리
- 실제 DB 로그 코퍼스가 들어오면 manifest case 추가
- mismatch/outcome triage 결과를 사용자 개입 후보로 연결
- 종료판정 문서와 데모 설명 정리

### 1차 이후 제품화/2차
- Streamlit 구조 분리 또는 대체 UI 이주 (현재 Streamlit은 검증 콘솔용)
- 프로젝트/코퍼스/스키마 파일 관리
- 장기 실행 job 관리
- 대규모 corpus dashboard
- AI/controller policy, full teambuild inference, doubles/multi-target 등
- PP / move availability, ability suppression/change, terrain/room 등

## 4. UI 표현력 판정

**질문:** 지금 부족한 게 UI 표현력인가, 복제쪽 결함인가?

1. **복제 Core 결함 여부**: 
   복제 core의 큰 축(DB 로그 매핑, 페이즈별 메커니즘 개입, HP/비-HP 자원 검증 시스템, 정규화 루프 등)은 닫혔습니다.
2. **UI는 "정의/수정/검수 가능한 Surface" 문제**: 
   현재 요구되는 UI는 단순 미관 작업이 아니라, **사용자가 엔진의 검증 결과와 미모델 내역을 보고 직접 개입할 수 있는 통로(Surface)를 연결**하는 작업입니다.

이 루프는 이제 단순 기능 목록이 아니라 아래 형태의 일관된 계약으로 연결되어 있다:
  Step6 DB mapping/export
  → schema JSON
  → run_db_corpus_backtest.py
  → manifest fixture pack
  → mismatch/outcome triage
  → mechanism RE/user intervention

근거:
- 포켓몬 전투 복제를 위한 핵심 실행 구조(TurnManager, Action/Trace 분리 로직)와 범용적 검증 구조(State Snapshot, Damage/Resource Delta IR)가 완성되었다.
- 실제 외부 코퍼스를 받아들이는 흐름이 HTML/Replay가 아닌 DB-log(CSV/JSON 등) 중심의 파이프라인으로 확고히 자리잡았다.
- 부족한 것은 핵심 복제 엔진 결함이라기보다 실제 코퍼스 확장과 제품 UI 이주/정리다.
- 현재의 Streamlit 환경은 최종 제품 UI가 아니라 검증 콘솔(Console)이며, 이를 대체하거나 분리하는 작업은 1차 목표의 필수 조건이 아니라 1차 종료 후 이주/2차 제품화 작업으로 둔다.

다음 액션:
1. 실제 DB 코퍼스(예: Gen 5 전투 로그)를 투입하여, 코퍼스가 요구하는 기능 꼬리를 RE 루프에 태우고 픽스처(manifest)를 살찌운다.
2. Mismatch/Outcome Triage 결과를 바탕으로 사용자가 직접 Mechanism을 수정하고 닫아내는 데모 시나리오를 가다듬는다.
3. 실증 결과가 안정권에 접어들면, 2차목표인 완전한 UI 이주, AI 컨트롤러, 다중 코퍼스 스케일아웃으로 진입한다.
