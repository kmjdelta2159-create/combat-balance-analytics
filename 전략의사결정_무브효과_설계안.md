# 전략 의사결정 모델 — 무브 효과 경로 + 최소 정책 설계안

전략 의사결정 모델로 진입하기 위한 정찰 결과와, 그로부터 도출한 범위 좁힌 설계다. 검토용 문서이며 코드 변경은 없다. 승인되면 Antigravity 프롬프트 빌더로 진입한다.

## 1. 정찰로 확정한 사실

`_act_move_select`(engine.py L380~L418)는 movepool의 각 무브에 대해 사용자 공식으로 기대 데미지를 계산하고(`_base * type_mult * stab_mult`), 그중 최대값 무브를 고른다. 타겟이 없으면 movepool[0], movepool이 없으면 current_move=None(단일 global 공식 경로)이다.

여기서 두 가지가 드러난다. 첫째, 위력 0인 무브(boost·setup·순수 회복 무브)는 `_base`가 0이라 데미지 무브를 절대 못 이기고 선택되지 않는다. 둘째, 더 근본적으로 현재 데이터 모델에는 **비-데미지 무브라는 개념 자체가 없다.** movepool의 무브는 power·category·type만 갖고, "효과"를 표현할 필드가 없다. 지금까지의 세 메커니즘(Leftovers·상태이상·Protean)은 모두 캐릭터의 아이템·특성에 붙는 패시브이지, 플레이어가 턴마다 고르는 무브가 아니다.

결론은 명확하다. 전략 의사결정 모델이 결정할 대상은 현재 "어떤 데미지 무브를 칠까"뿐이고, 그건 그리디가 이미 최적으로 푼다. **결정할 비-데미지 선택지가 없으므로, 전략 모델을 단독으로 지으면 빈 껍데기다.**

## 2. 정직한 선행 조건 — 무브 효과 경로

전략 모델이 의미를 가지려면, 무브가 데미지 외의 효과를 낼 수 있어야 한다. 그래야 "boost를 깔까, 때릴까", "상태이상을 걸까, 데미지를 줄까" 같은 진짜 결정이 생긴다.

토대는 이미 있다. engine.py L122~L180의 active_states + get_effective_stat가 누적 스탯 증감을 표현하고(PERMANENT 트리거로 영구 boost 가능), 방금 만든 ON_MOVE_USE hook이 무브 사용 시점을 잡는다. 무브에 `effect` 필드를 더하고, ON_MOVE_USE에서 그 효과를 사용자(또는 타겟)에게 적용하면 된다.

표준 예가 Calm Mind다. "위력 0, 사용 시 사용자의 SpAtk·SpDef를 +1단계(percent boost) 누적"인 무브. 이게 무브 효과 경로의 첫 PoC다.

## 3. 범위 좁힌 2단계 설계

### PR-E. 무브 효과 경로 (Calm Mind류 boost 무브)

무브 라이브러리 스키마에 선택적 `effect` 표현을 더한다. 예: `{"effect": "boost", "target_stat": "SpAtk", "mod_type": "percent", "value": 0.5, "scope": "self"}`. ON_MOVE_USE hook(또는 그 직후 전용 hook)에서, 선택된 무브에 effect가 있으면 active_states에 해당 상태를 부여한다(기존 `_track_state` 재사용). scope=self면 사용자, scope=target이면 타겟.

step2에 무브별 effect 입력 UI를 더하거나(무브 시스템 섹션 확장), 1차에는 game_config 기반 간이 매핑으로 둔다. 이건 설계 확정 시 정한다.

검증: boost 무브 사용 → 사용자 active_states에 boost 추가 → get_effective_stat이 반영 → 다음 데미지 계산에 boost된 스탯 사용. 하니스로 단위검증.

이 단계만으로는 그리디가 여전히 boost 무브를 안 고른다(위력 0). 그래서 PR-F가 필요하다. 단, PR-E는 독립적으로 "boost 무브를 강제 지정했을 때 효과가 적용됨"을 검증할 수 있다.

### PR-F. 최소 룰 기반 정책

`_act_move_select`의 그리디를 **플러그인 가능한 정책**으로 일반화한다. 기본 정책 = 현재 그리디(회귀 0). 룰 기반 정책 하나를 추가한다 — "setup/effect 무브가 movepool에 있고 사용자가 아직 그 boost를 안 깔았으면(active_states에 해당 마커 없음) 그 무브를 고르고, 아니면 그리디로 데미지." 사용자가 정책을 game_config로 선택한다(슬라이더 철학: 미설정 시 그리디 유지).

이게 전략 모델의 첫 진짜 결정이다 — "초반에 boost를 깔고 후반에 때린다"는 플레이 패턴을 재현. 룰 기반이며 ML이 아니다. 후속으로 조건을 확장(HP 임계·상대 위협도·턴 수)하거나 ML 정책으로 갈 수 있다.

검증: boost 무브 보유 캐릭터가 1턴에 boost를 깔고(active_states 부여 확인), 이후 턴엔 데미지 무브를 고르는지 하니스로 검증. 그리디 기본 경로 회귀 없음 확인.

## 4. 정직한 한계

이건 룰 기반 정책이지 학습된 정책이 아니다. 첫 PoC는 단일 boost 패턴만 다루고, 교체·메가 타이밍·Substitute 활용 같은 결정은 각 메커니즘이 "고를 수 있는 무브/행동"으로 표현된 뒤에야 정책에 들어온다. 교체는 여전히 전투 모델(bench) 부재로 막혀 있다. 정책이 정확도(62~67% 권역)에 주는 영향은 통제 측정 전이고, 합성 데이터 안 수치다. 외부 실제 데이터 수치는 미측정.

이 단계가 닫히면 도구의 위치는 "뼈대 + 정적 계산기 + 동적 메커니즘 3종 + 무브 효과 + 최소 전략 결정"이 된다. 시뮬레이터 정의(§1·§3-4: turn-by-turn 동적 메커니즘 + 전략 의사결정)에 처음으로 전략 축이 한 칸 들어가는 것이다. 여전히 부분 구현이며, 완전한 시뮬레이터와는 거리가 있음을 흐리지 않는다.

## 5. 다음 행동

이 설계 승인 여부를 묻는다. 승인되면 PR-E(무브 효과 경로)부터 정찰을 마저 하고(무브 라이브러리 스키마·step2 무브 입력 UI·active_states 부여 지점) 하니스 검증 후 Antigravity 프롬프트 빌더로 진입한다. PR-F(정책)는 PR-E 검증 후 같은 방식으로 잇는다. 각 단계 끝마다 보고 + 사용자 결정.
