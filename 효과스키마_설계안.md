# 효과-스키마 설계안 — 발동형 효과를 데이터로 (트리거×조건×효과×스코프)

> demand 카탈로그(`효과스키마_demand카탈로그.md`) §3에서 결정화. 1 battle이 아니라 2전투·
> 2세대·~10개 발동형 예시에서 공통 shape를 뽑았다. 효과-스키마는 (나)의 특성/도구/무브효과
> 질량을 *손코딩이 아니라 데이터*로 표현하는 keystone. 디스패처 코어는 프로토타입으로 검증됨.

## 0. 한 줄

엔진엔 이미 *특수 효과 적용기 4종*이 있다 — `_act_status_tick`(독/화상)·`_act_weather_chip`
(날씨)·`_apply_action_gate`(마비/혼란)·`_act_on_hit`(피격 패시브). 전부 "config 스펙 → 조건 →
자원 델타/행동차단 → 특정 페이즈"의 같은 구조다. 효과-스키마는 이를 **하나의 디스패처 +
데이터 테이블**로 일반화한다. 새 효과는 코드가 아니라 스키마 한 줄.

## 1. 통합 shape (프로토타입 검증됨)

```
effects = {
  "<이름>": {
    "trigger":   "<엔진 페이즈>",          # ON_HIT / ON_TURN_END / ON_MOVE_USE / ON_SWITCH ...
    "condition": {...},                    # (선택) weather/contact/target_attacking ...
    "effect":    {"type": "<효과>", ...},  # damage_frac / heal_frac / block_action / stat_mult ...
    "scope":     "self|target|attacker|slot",
    "source":    "ability|item|move",      # 출처(로그 매칭·UI용)
    "chance":    1.0                        # (선택) 확률 — stoch.roll_chance
  }, ...
}
```

검증(디스패처 프로토타입): 접촉 무브 ON_HIT → Rough Skin이 공격자 1/8 데미지 + flinch가
대상 행동차단; 비접촉이면 Rough Skin 미발동; rain일 때 Rain Dish 자힐 1/16(max 캡). shape가
§3 카탈로그를 덮음 확인.

## 2. 트리거 = 엔진 기존 페이즈 (신규 훅 불필요)

엔진이 이미 브로드캐스트하는 페이즈에 효과를 붙인다 — 신규 훅 없음:

| 트리거 | 엔진 페이즈 | 카탈로그 예 |
|---|---|---|
| on-hit / on-contact | `ON_HIT` | flinch, Rough Skin, Rocky Helmet, Trick |
| on-turn-end | `ON_TURN_END` / `ON_WEATHER_TICK` | Rain Dish, Leftovers(이미 보유) |
| on-attack | `ON_MOVE_USE`/후 | Life Orb(반동) |
| on-switch-in | `ON_SWITCH` | (Intimidate 등 후속) |
| on-action(선) | `_act_target_select` 앞 | Protect, Substitute |
| delayed | 스케줄 큐(신규) | Wish |

대부분 기존 페이즈에 떨어진다. *지연(Wish)·선행동(Protect/Substitute)*만 약간의 신규 표면.

## 3. 효과 타입 (자원/스탯 위 원자 연산)

기존 적용기가 쓰는 자원 델타 패턴(`res['current'] -= res['max']*frac`)을 그대로 재사용:

- `damage_frac {frac, of:maxhp}` — 스코프 대상에 maxHP 분수 데미지. (Rough Skin·Rocky Helmet·
  Life Orb 반동·해저드·날씨칩·상태틱 — *기존 4적용기가 이 한 타입의 특수화*.)
- `heal_frac {frac}` — 분수 회복(max 캡). (Rain Dish·Leftovers·Wish.)
- `block_action {duration}` — 이번 턴/지속 행동차단. (flinch·Protect — `_apply_action_gate`와
  동류.)
- `stat_mult {stat, mult}` — 항시 스탯 배율. (이미 PR4 효과레이어가 정적으로 처리; 발동형은
  여기로 통합 가능.)
- (후속) `create_substitute {frac}` — 대체 자원 생성(resource.py route_damage 흡수 — 갈래⑤),
  `swap_item` — Trick, `set_weather/set_hazard` — 이미 F3/F2(무브효과 kind).

## 4. 디스패처 (기존 4적용기 일반화)

각 트리거 페이즈에 등록되는 *단일* 액션 `_act_effect_dispatch(phase)`:
1. `game_config['mechanisms']['effects']`에서 `trigger==phase` 필터.
2. `condition` 평가(weather/contact/target_attacking…).
3. `chance` 있으면 `stoch.roll_chance`.
4. `scope`로 대상 결정(self=active_char, attacker=active_char, target=current_target).
5. `effect.type`별 자원/스탯 연산(위 원자).

이 디스패처가 서면 status_tick·weather_chip·action_gate는 *같은 디스패처의 데이터 케이스*가
된다 — 단, **지금 리팩터하지 않는다**(§7 경계).

## 5. §3 카탈로그 → 스키마 표현 (결정화 결과)

```
"flinch":       {trigger:ON_HIT, effect:{block_action, duration:this_turn}, scope:target, source:move}
"Rough Skin":   {trigger:ON_HIT, condition:{contact}, effect:{damage_frac,1/8}, scope:attacker, source:ability}
"Rocky Helmet": {trigger:ON_HIT, condition:{contact}, effect:{damage_frac,1/6}, scope:attacker, source:item}
"Life Orb":     {trigger:ON_MOVE_USE, effect:{damage_frac,1/10}, scope:self, source:item}
"Rain Dish":    {trigger:ON_TURN_END, condition:{weather:rain}, effect:{heal_frac,1/16}, scope:self, source:ability}
"Protect":      {trigger:on_action, effect:{block_incoming, this_turn}, scope:self, source:move}   # 후속
"Substitute":   {trigger:on_action, effect:{create_substitute,1/4}, scope:self, source:move}        # 후속(갈래⑤)
"Wish":         {trigger:delayed(2), effect:{heal_frac,1/2}, scope:slot, source:move}               # 후속(스케줄)
"Trick":        {trigger:ON_HIT, effect:{swap_item}, scope:both, source:move}                       # 후속
```

## 6. 첫 PR 범위 (verifiable·작게)

**디스패처 + 2 효과타입(damage_frac, block_action) + flinch + Rough Skin/Rocky Helmet.**
- 신규 엔진 액션 `_act_effect_dispatch`를 ON_HIT에 등록 + `effects` 스키마 읽기. 무브에
  `contact` 플래그(reference에 추가).
- 검증: **flinch가 Kecleon전 Fake Out `cant(flinch)` 이벤트를 재현**(대상 행동차단), **Rough
  Skin/Rocky Helmet이 gen5 접촉 env 데미지를 재현**(공격자 1/8·1/6 — 카탈로그 env에 실측 있음).
- 회귀 0: `effects` 미설정이면 디스패처 no-op. 기존 status_tick/weather/hazard 무변경.

이후 PR: heal_frac(Rain Dish·Leftovers 통합) → on_action(Protect) → create_substitute(갈래⑤)
→ delayed(Wish) → swap_item(Trick). 코퍼스 빈도순.

## 7. 정직한 경계

- **기존 F2/F3/G1/status_tick은 *지금 리팩터하지 않는다*.** 디스패처가 서도 기존 적용기는
  그대로 둔다(회귀 0 우선). "효과 적용기 4종을 디스패처 데이터로 흡수"하는 통합은 *의도된
  별도 리팩터*이지 이번 PR이 아니다. 새 효과만 스키마로.
- **shape 일반성은 포켓몬 2세대까지만 확인.** 교차-게임 가드(로드맵): trigger/condition/effect/
  scope가 *비-포켓몬* 스탯턴제 로그에서도 성립하는지는 비-포켓몬 로그 확보 후 굳히기 전 점검.
  지금 결정화는 *잠정*(포켓몬 demand 기반) — 비-포켓몬에서 새 트리거/효과타입이 나오면 확장.
- **복잡 효과는 단계적**: Substitute(자원 생성·route_damage 흡수)·Wish(스케줄)·Trick(상태
  교환)·Protect(피격 무효)는 새 효과타입이 필요 → 원자 2개(damage_frac/block_action)로 첫 PR을
  닫고 점증.
- **확률 효과**(Scald 30% 화상 등)는 `chance` + `stoch.roll_chance`(기존)로.
