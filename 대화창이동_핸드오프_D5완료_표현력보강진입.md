# 대화창 이동 핸드오프 — D5 완료 / 표현력 보강 진입 직전 (Pokemon 복제 1/4 정찰 중)

새 대화창에 이 문서를 그대로 붙여넣어라. 아래 컨텍스트를 숙지하고, 곧바로 §7의 즉시 다음
스텝부터 이어서 진행한다.

## 0. 너의 역할 (불변 제약)

너는 프로젝트 "Combat Balance Analytics"의 보조자다. 프로젝트 리드는 사용자(김민재)다.

프로젝트 코드 파일은 직접 수정하지 않는다. 너의 산출물은 코드 생성기 "Antigravity"에게
넘길 markdown 프롬프트다. 사용자가 그 프롬프트를 Antigravity에 전달하고, Antigravity가
실제 파일을 고친다.

예외는 검증용 테스트 하니스, 레퍼런스 구현, 합성 데이터 생성기, 검증 자산이다. 이것들은
프로젝트 코드가 아니라 검증 도구라 너가 직접 작성해도 된다. 샌드박스(`~/...` 또는
`outputs/`) 또는 검증 폴더(`검증_*/`)에서 작업한다.

검증 의무가 있다. Antigravity 산출물은 절대 설명만 믿지 마라. 항상 코드를 직접 읽어
검증한다 — Grep으로 마커 확인, 변경 구역 Read 정독, 라인 수 산술, 바이트 동등성(MD5),
클린룸 컴파일. Antigravity는 불완전·하드코딩 납품 전력이 있다.

프롬프트 작성 시 검증된 하니스에서 find/replace 블록을 프로그래매틱으로 추출·조립하고
라운드트립(쓰기→되읽기→블록 재추출→MD5 대조)으로 전사 오류를 0으로 만든다. 이전
프롬프트들(`Phase8d_채널매핑_프롬프트.md`, `WeightDrivenUI_프롬프트.md` 등)이 모범 양식
이다.

말투는 가독성을 우선한다. 압축적 노트 스타일(짧은 명사형, 화살표 남발, 표 위주)이
아니라 완전한 문장으로 쓴 prose를 기본으로 한다. 표는 시간·일자 같은 정렬된 정보에만
쓴다.

**시간 추정 금지** (이번 대화에서 명시적으로 정해진 규칙). "1주", "5분", "이번 작업은
N시간" 같은 숫자를 던지지 마라. 사용자가 진행 상황을 보고 조절한다. 너는 작업의
*범위·위험·예상 단계 수*만 보고하고, 시간은 사용자 영역이다.

## 1. 프로젝트 개요 — 최종 목표

전투로그 기반 범용 턴제 전투 시뮬레이터다.

최종 목표는 세 단계다. 첫째는 전투 로그를 역설계한다. 둘째는 전문가(사용자) 개입을
받아 게임의 전투 시스템을 복제한다. 셋째는 복제본을 밸런스 분석용으로 최적화한다.

핵심 원리는 연속 슬라이더다. 대상은 턴제 구조 게임이다. 대중적인 형태일수록 시뮬레이터
의 역설계 비중이 높고, 비대중적일수록 사용자 개입 비중이 높다. 역설계와 개입은 독립된
두 기둥이 아니라 하나의 연속 슬라이더의 양 끝이다.

조건이 있다. 이 세상에 없는 게임이라도 턴제 구조를 채용했다면 사용자 개입 비중 조절
만으로 복제 가능해야 한다. 슬라이더는 정밀도의 연속이지 매체의 카테고리 점프
(form → Python class)가 아니다.

정직한 스코프는 *스탯 기반 턴제 게임*이다(JRPG, 가챠, SRPG, 덱빌더). 비-스탯 정보 게임
(체스, 2048 등)은 *도구 한계가 아니라 의도된 도메인 선택 영역 밖*. 이 구분은 이전
대화에서 사용자와 명시적으로 합의됐다.

**모든 결정은 최종 목표 달성을 기준으로 한다. 그 과정에서 현재 시연하려는 게임 복제에
사용되는 기능을 먼저 구현하는 것뿐.** (이번 대화에서 사용자가 두 번 강조한 원칙. 시연
narrative·UI 정리·cosmetic은 *진짜 복제 능력 보강*보다 우선순위가 낮다.)

## 2. 아키텍처 핵심

### 2-1. 현재 구조

Streamlit SPA 위저드 구조다. Step 1 업로드, Step 2 시스템 정의, Step 3 불일치, Step 4
프로파일링, Step 6 대시보드 순으로 흐른다.

전투 엔진 `modules/engine.py`는 UI 독립이며 순수 Python이다. 액션 파이프라인은 9단계다.
PASSIVE_START, STAT_CALC, MOVE_SELECT, TARGET_SELECT, DAMAGE_CALC, ELEMENT_MULT,
CRIT_CALC, APPLY_DAMAGE, ON_HIT, DEATH_CHECK.

3-Layer 설계다. L1 Universal Core가 약 80%, L2 Pluggable Modules가 약 82%, L3 Game
Plugins가 약 15%로 대부분 부재다.

`game_config` dict가 L3의 데이터 척추다. 현재 키는 `categories`, `type_table`,
`type_columns`, `stab_factor`, `channels`다.

### 2-2. 가중치 기반 동적 상태 주도 UI (D4~D5에서 추가됨)

`modules/ui_registry.py`에 4 톱니바퀴 아키텍처가 들어가 있다. State Emitter는 D7~D8에서
engine 통합 예정이고 현재는 `build_mock_state_from_log()`가 mock 제공. Registry schema 8
필드(key, render_func, component_type, is_fixed, layout_hint, category, default_weight,
min_log_columns). Dynamic Layout Manager(`render_dynamic_dashboard`)와 가중치 패널
(`render_weight_panel`)이 메인 화면 expander로 표시.

**중요**: D4 설계안에서 가중치 패널을 사이드바에 두자고 했으나, main.py L42의
`[data-testid="stHeader"] { display: none !important; }` 정책이 *Streamlit 사이드바 토글까지
함께 숨겨서* 사용자가 사이드바를 열 방법이 없다. 그래서 D5-hotfix로 *메인 화면
expander*로 이동했다. 기존 main.py L92 "사이드바는 상태 확인 전용" 정책 유지.

**시연 가치 평가**: D4~D5의 동적 UI 자체는 *시연 핵심이 아니라 trajectory 인프라*다.
이번 대화에서 사용자가 명확히 했다 — "시연일에는 그냥 특정 게임 몇 개만 제대로 담아내는
걸 보여주면 되는 거고 데드라인 전까지 특정 게임만 담아낼 수 있을 정도로만 완성하면 되는
거임. 시연용으로 쓰지도 않을 기능 덕지덕지 붙이는 게 아니라". 즉 D5 dashboard 자체는
*그대로 두고 시연 화면에서는 노출 안 함* 정책이며, 데드라인 안의 우선순위는 *진짜 게임
복제 능력*이다.

## 5. 핵심 방법론 주의사항 (업데이트)

### 5-1. Bash mount silent truncation — **새 변종 추가**

bash로 큰 워크스페이스 파일을 cp/cat하면 조용히 잘린다. engine.py ~890줄, step2 ~960줄,
step6 1349줄. wc, py_compile, diff가 거짓 에러를 낸다.

이전 변종 (직전 핸드오프에서 알려진 것):
- Edit 도구로 mount 경로에 큰 파일(~13KB)을 쓸 때 silent truncation (null bytes 삽입)
- Write 도구도 mount 영역에서 ~13KB 경계 silent truncation

**이번 대화에서 발견한 새 변종**:
- bash sandbox의 Python `open().read()`도 silent truncation 영향. step6 ~80KB 파일을
  Python으로 읽으면 *mid-statement에서 끊김*. 라인 수가 1322 (실제 1349). py_compile이
  SyntaxError 보고하는 *거짓 실패*.

**규칙**:
- 워크스페이스 파일은 *Read 도구*로 읽는다. bash와 bash 안의 Python 모두 금지.
- bash py_compile/wc/sed/tail/Python read가 *알 수 없는 에러*를 내면 *Read 도구로 진실
  확인*. Read와 bash가 다른 답을 주면 Read가 진실이다.
- 검증은 Grep 마커와 변경 구역 Read 정독으로 한다.
- 샌드박스 자체 파일(`~/pkmn_regen/...`, `outputs/`)은 bash로 다뤄도 안전. 단 *그 안에서
  큰 mount 파일을 Python으로 읽는 건 위험*.
- 디렉터리 목록(`ls`)이나 소형 파일(<10KB)은 bash OK.
- 큰 파일 작성은 bash heredoc(`cat > path << 'HSEND'`) 우회.
- Write 도구로 큰 파일을 mount에 쓸 때 truncation 위험이면 bash heredoc 우회.

### 5-2. 검증 워크플로 (Antigravity 납품 후)

먼저 Glob 또는 bash `ls`로 신규 파일 존재 확인. 다음 Grep으로 새 마커와 옛 마커
(negative grep) 모두 확인.

라인 수 산술 — 이전 라인 수 + (replace 줄수 − find 줄수) = 새 라인 수가 정확히 일치
하는지 본다. *단 bash wc -l가 거짓 실패할 수 있음*. 의심되면 Read 도구로 끝부분 확인.

Phase 마커들이 동일한 정수 시프트를 받았는지 본다. 곁가지 수정 0건의 산술 증명.

변경 블록을 Read 정독해서 프롬프트의 바꾸기 블록과 바이트 동등성(MD5)을 본다.

클린룸 컨텍스트 py_compile은 *bash 거짓 실패 가능성 인지하고* 본다. SyntaxError 보고
시 line 번호의 Read 정독으로 진실 확인.

### 5-3. 프롬프트 작성 워크플로 — **디자인 정책 정찰 추가**

변경 사이트를 Grep과 Read로 정확한 컨텍스트 확정한다.

**(새 단계 — 이번 대화의 D5 실수로 추가)** *변경하지 않을 영역의 디자인 정책도
