# 대화창 이동 핸드오프 — 동적 메커니즘 인프라 진입 직전

새 대화창에 이 문서를 그대로 붙여넣어라. 아래 컨텍스트를 숙지하고, 곧바로 §6의 즉시 다음 스텝부터 이어서 진행한다.

## 0. 너의 역할 (불변 제약)

너는 프로젝트 "Combat Balance Analytics"의 보조자다. 프로젝트 리드는 사용자(김민재)다.

프로젝트 코드 파일은 직접 수정하지 않는다. 너의 산출물은 코드 생성기 "Antigravity"에게 넘길 markdown 프롬프트다. 사용자가 그 프롬프트를 Antigravity에 전달하고, Antigravity가 실제 파일을 고친다.

예외는 검증용 테스트 하니스, 레퍼런스 구현, 합성 데이터 생성기, 검증 자산이다. 이것들은 프로젝트 코드가 아니라 검증 도구라 너가 직접 작성해도 된다. 샌드박스(`~/...` 또는 `outputs/`) 또는 검증 폴더(`검증_*/`)에서 작업한다.

검증 의무가 있다. Antigravity 산출물은 절대 설명만 믿지 마라. 항상 코드를 직접 읽어 검증한다 — Grep으로 마커 확인, 변경 구역 Read 정독, 라인 수 산술, 바이트 동등성(MD5), 클린룸 컴파일. Antigravity는 불완전·하드코딩 납품 전력이 있다.

프롬프트 작성 시 검증된 하니스에서 find/replace 블록을 프로그래매틱으로 추출·조립하고 라운드트립(쓰기→되읽기→블록 재추출→MD5 대조)으로 전사 오류를 0으로 만든다. 이전 프롬프트들(`Pokemon복제1-4_무브공급_프롬프트.md`, `Pokemon복제1-4_Validator후속보강_프롬프트.md`, `Pokemon복제1-4_Step3후속보강_프롬프트.md`)이 모범 양식이다.

말투는 가독성을 우선한다. 압축적 노트 스타일 금지. 완전한 문장의 prose를 기본으로 한다. 표는 시간·일자 같은 정렬된 정보에만 쓴다.

시간 추정 금지. "1주", "5분", "이번 작업은 N시간" 같은 숫자를 던지지 마라. 사용자가 진행 상황을 보고 조절한다. 너는 작업의 범위·위험·예상 단계 수만 보고하고, 시간은 사용자 영역이다.

**시연 관련 일체 언급 금지**. 사용자가 직전 대화에서 명시적으로 강하게 짚었다. 시연일·데모·평가자·시연 narrative·시연용 화장 같은 표현 모두 금지. 본 보조자의 작업 영역은 도구의 진짜 능력 보강이지 시연용 보여주기가 아니다. 사용자가 명시적으로 그쪽을 요청하지 않는 한 그 영역으로 가지 마라.

도구 한계를 화장하지 마라. 정직성 우선. 정적 차원만 작동하면 그렇게 말하고, 동적 차원이 부재하면 그렇게 말한다. "Pokemon 복제 가능" 같은 강한 표현으로 도구의 위치를 흐리지 마라. 정확한 표현은 "Pokemon의 정적 데미지 모델 일부 복제"다.

## 1. 프로젝트 개요 — 최종 목표

전투로그 기반 범용 턴제 전투 시뮬레이터다.

최종 목표는 세 단계다. 첫째는 전투 로그를 역설계한다. 둘째는 전문가(사용자) 개입을 받아 게임의 전투 시스템을 복제한다. 셋째는 복제본을 밸런스 분석용으로 최적화한다.

핵심 원리는 연속 슬라이더다. 대상은 턴제 구조 게임이다. 대중적인 형태일수록 시뮬레이터의 역설계 비중이 높고, 비대중적일수록 사용자 개입 비중이 높다. 역설계와 개입은 독립된 두 기둥이 아니라 하나의 연속 슬라이더의 양 끝이다.

조건이 있다. 이 세상에 없는 게임이라도 턴제 구조를 채용했다면 사용자 개입 비중 조절만으로 복제 가능해야 한다. 슬라이더는 정밀도의 연속이지 매체의 카테고리 점프(form → Python class)가 아니다.

정직한 스코프는 스탯 기반 턴제 게임이다(JRPG, 가챠, SRPG, 덱빌더). 비-스탯 정보 게임(체스, 2048 등)은 도구 한계가 아니라 의도된 도메인 선택 영역 밖.

### 시뮬레이터의 정의 (이번 대화에서 확립)

시뮬레이터는 계산기와 다르다. 시뮬레이터는 두 팀이 주어졌을 때 turn-by-turn으로 전투의 흐름 자체를 재구성해서 승부와 그 사이의 동작을 재현하는 것이다. 데미지 한 번을 정확히 계산하는 것은 계산기다.

"Pokemon을 시뮬레이터로 복제한다"가 무엇을 의미하는지를 정확히 알아야 한다. 그건 무브 위력·타입 상성·STAB·P/S 라우팅 같은 정적 데미지 모델 + Protean(매 무브마다 타입 갱신)·메가 진화(전투 중 stat·타입·특성 교체)·우선도 무브(Speed 우회 액션 순서)·Substitute(보조 HP 풀)·Calm Mind(누적 boost)·Leftovers(턴 종료 % 회복)·Trace(특성 복사)·상태이상(화상·마비·독·잠) 같은 동적 메커니즘 + 그 위에서 작동하는 전략 의사결정 모델(언제 교체할지·언제 메가를 쓸지·언제 Substitute를 깔지·boost 누적 시점) 전부를 포함한다.

이 정의를 흐리지 마라. "Pokemon 데미지 모델 복제"는 계산기 수준이지 시뮬레이터 수준이 아니다.

## 2. 아키텍처 핵심

### 2-1. 현재 구조

Streamlit SPA 위저드 구조다. Step 1 업로드, Step 2 시스템 정의, Step 3 불일치, Step 4 프로파일링, Step 5 대시보드 순으로 흐른다. **파일명과 indicator 번호가 어긋난 매핑이 있으니 main.py L9~L13의 import를 진실 출처로 본다.**

전투 엔진 `modules/engine.py`는 UI 독립이며 순수 Python이다. 액션 파이프라인은 9단계다. PASSIVE_START, STAT_CALC, MOVE_SELECT, TARGET_SELECT, DAMAGE_CALC, ELEMENT_MULT, CRIT_CALC, APPLY_DAMAGE, ON_HIT, DEATH_CHECK.

3-Layer 설계다. L1 Universal Core가 약 80%, L2 Pluggable Modules가 약 82%, L3 Game Plugins가 약 15%로 대부분 부재다.

`game_config` dict가 L3의 데이터 척추다. 현재 키는 `categories`, `type_table`, `type_columns`, `stab_factor`, `channels`다.

### 2-2. 가중치 기반 동적 상태 주도 UI

`modules/ui_registry.py`에 4 톱니바퀴 아키텍처가 들어가 있다. State Emitter는 engine 통합 예정이고 현재는 `build_mock_state_from_log()`가 mock 제공.

### 2-3. 9단계 파이프라인의 정직한 한계 (이번 대화에서 확립)

9단계 파이프라인은 "한 캐릭터가 한 무브로 한 대상을 친다"는 가장 단순한 사이클을 잡고 있다. 그 위에 사용자 공식과 상성표가 얹힌다. 정적 데미지 계산은 이 구조로 잘 작동한다.

그러나 동적 메커니즘은 자리가 없다. 메가 진화로 전투 중 stat·타입·특성이 교체되거나, Protean으로 매 무브마다 캐릭터 타입이 갱신되거나, 우선도 무브가 Speed 비교를 우회하거나, Substitute가 데미지 흡수 보조 HP 풀로 들어가거나, Calm Mind가 영구 boost를 누적하거나, Leftovers가 턴 종료 % 회복을 하거나, 상태이상이 데미지·턴 진행을 바꾸는 메커니즘 — 이런 동작들이 9단계 어디에도 hook이 없다.

이게 본 phase 작업의 출발점이다. 9단계 파이프라인에 동적 hook을 끼우고, 캐릭터 instance에 동적 상태 필드를 추가하고, 표준 메커니즘 라이브러리를 채우고, 사용자가 폼으로 메커니즘을 부착할 수 있는 입력 UI를 만든다.

### 2-4. 기존 동적 상태 인프라

engine.py L122~L228에 이미 Event-Driven State Machine이 부분적으로 구현되어 있다. `active_states` 리스트에 buff/debuff state object 저장. `get_effective_stat`이 active_states 증감치 합산. `_notify_event`가 expire_trigger 매칭으로 카운트 차감 + 제거. `_PHASE_TO_EVENT` 매핑이 9단계 phase 키 → 이벤트 이름.

이 기존 인프라가 표준 메커니즘 라이브러리의 토대로 활용 가능한지가 본 phase 첫 정찰의 주된 관심사다. 활용 가능하면 그 위에 라이브러리를 얹고, 부족하면 ON_TURN_START·ON_TURN_END·ON_SWITCH 같은 누락 hook을 추가한다.

## 3. 현재 진행 상황

### 3-1. Phase 1~9b, 8c-α, 8d, D1~D5 (완료)

Phase 1~7은 업로드·매핑·SR·LR·결측치·검증 프레임. Phase 8a 무브 엔티티화, Phase 8b N-type 상성표·STAB, Phase 9 (μ,λ)-진화 전략 최적화, Phase 9b 멀티코어 핫픽스, Phase 8c-α per-battle backtest, Phase 8d 기믹 채널 명시 매핑. 모두 완료·검증.

D1~D5는 가중치 기반 동적 상태 주도 UI 인프라 구축. 완료·검증.

### 3-2. Pokemon 복제 1/4 + Validator 후속 + Step3 후속 (완료)

Pokemon 복제 4 항목 plan 중 1번 무브 데이터 공급 인프라가 완료됐다. 후속 누락을 두 번의 보강 PR로 잡았다.

`Pokemon복제1-4_무브공급_프롬프트.md` — step2 Move System 섹션(L294~L321)에 attack_log 업로드 expander + `df_for_moves` 결정 로직. +21줄. 납품·검증 완료. Pokemon attack_log 8000행에서 37개 무브 자동 추출 확인.

`Pokemon복제1-4_Validator후속보강_프롬프트.md` — 직전 PR이 Move System 섹션만 풀고 Live Formula Validator(L228~L236)는 메인 df만 봐서 `move_power` 변수가 환경에 안 들어와 NoneType 에러로 적용 버튼이 disabled됐다. `has_move_data` 우선순위 결정으로 attack_log_df도 보게 보강. +9줄. 납품·검증 완료.

`Pokemon복제1-4_Step3후속보강_프롬프트.md` — Step 3 Discrepancy(step5_discrepancy.py)도 같은 무브 변수 주입 누락. `eval_formula = formula.lower()` unique 라인 직전에 `eval_df` 컬럼 broadcast 보강. step5_discrepancy.py +17줄. 납품·검증 완료.

세 PR 모두 라운드트립 MD5 byte-equal 통과, Streamlit 재시작 후 동작 확인.

Pokemon 복제 plan의 나머지 항목(18타입 상성표·STAB·P/S 라우팅)은 모두 사용자가 step2에서 정적 데이터를 채우면 작동하는 것이고 코드 변경 0건이다. 사용자가 카테고리 라우팅 + 타입 컬럼 + STAB 1.5 + 공식 `move_power * offense / defense`로 채워 백테스트한 결과의 정확도 추이는 다음과 같다.

공식 `attack` (라우팅 미설정, 무브 시스템 비활성) 59.8%. 공식 `move_power * attack / defense` (라우팅 미설정) 62.8%. 공식 `move_power * attack / defense` (HP/HP 라우팅, 사용자 isolation 실험) 61.0%. 공식 `move_power * offense / defense` (Pokemon 매핑 라우팅 + STAB 1.5 + type_table 1.0 균일) 67.4%.

기준선: 71.6%는 무브/타입 도입 전 Pokemon best-effort, 95.8%는 Pokemon 이론적 천장(난수 4.2%), 50%는 동전 던지기다.

추이가 단조 증가했다. 비대칭(Ally vs Enemy 정확도 비대칭)도 라우팅·STAB 입력으로 회복됐다(이전 68.1% vs 57.2% → 66.9% vs 67.9%).

67.4%로 묶인 원인은 type_table 19×19가 1.0 균일이라 18타입 상성 신호 부재. 사용자가 type_table을 채우면 95% 권역 도약이 기대된다. **그러나 95% 권역은 합성 데이터(우리 ref가 같은 단순화 모델로 생성한) 안의 수치다. 실제 외부 데이터에 대고는 다른 측정이 필요하다.**

### 3-3. 외부 데이터 적용성 검증 결과 (이번 대화의 핵심 발견)

사용자가 외부 실제 게임 로그를 가져왔다. 두 형식이다.

`OUMonotype-2014-01-29-kdarewolf-onox.html`은 Pokemon Showdown 공식 replay다. `<script type="text/plain" class="battle-log-data">` 안에 PS battle protocol 텍스트가 들어 있다. 파이프 구분 이벤트 스트림 형식이다 — `|move|p1a: May Day Parade|Fake Out|p2a: AMagicalFox`, `|-damage|...`, `|switch|...`, `|turn|N`, `|faint|...`.

`제목 없는 스프레드시트 - 시트1.csv`는 사용자가 위 HTML을 AI에게 시켜 CSV로 변환한 결과다. 5 컬럼(Turn/Action/Subject/Target/Details) × 81 행. long-format 이벤트 단위.

우리 도구의 처리 시도 결과 네 가지 격차가 식별됐다.

격차 1은 Step 1 거부다. file_uploader가 `type=["csv"]`만 받아 HTML이 즉시 거부된다.

격차 2는 Step 2 스키마 불일치다. AI 변환 CSV가 long-format이고 우리 도구는 wide-format(stat이 컬럼)을 기대한다. `detect_move_columns`가 `Move_Power` 같은 컬럼 hint를 못 찾고, base_stats 추출 실패(HP는 "324/324" 같이 Details 안의 텍스트), gimmicks 추출 실패. 사용자가 컬럼 매핑 시도해도 어느 컬럼도 우리 스키마에 안 맞는다. 적용 버튼이 disabled.

격차 3은 메타정보 lookup 부재다. 변환을 어떻게 하든 PS replay엔 무브 위력·타입·카테고리가 이름만 있고 메타정보는 PS movedex JSON에서 별도 lookup 필요. 캐릭터 base stat도 PS pokedex JSON에서 lookup 필요. 우리 도구엔 lookup 계층 자체가 없다.

격차 4는 동적 메커니즘 표현 부재다. 위 변환·lookup을 모두 해도 9단계 파이프라인에 자리가 없는 메커니즘들은 시뮬이 재현 못 한다. 첫 7턴만 봐도 Protean, 메가 진화(Alakazam-Mega), Trace, 우선도 무브(Sucker Punch/Shadow Sneak/Fake Out), Substitute, Calm Mind, Leftovers가 한꺼번에 등장한다. 이 중 어느 것도 우리 game_config의 type_table·stab_factor·categories·channels로 표현 안 된다.

변환은 격차 1~3의 절반을 메운다(어댑터 PR 가능). 격차 4는 변환으로 안 풀린다. 9단계 파이프라인의 메커니즘 인코딩 자체를 확장해야 한다.

### 3-4. 현재 도구의 정직한 위치 (이번 대화에서 확립)

시뮬레이터의 뼈대는 있다. 9단계 파이프라인이 turn-by-turn 진행되고, turn_manager가 액션 순서를 정렬하고, MOVE_SELECT → DAMAGE_CALC → APPLY_DAMAGE → DEATH_CHECK가 순환한다. 한 게임을 turn-by-turn으로 시뮬레이션하는 골격은 있다.

그 안에 채워진 게 빈약하다. 무브 선택은 movepool에서 그리디(기대 데미지 최대)로 고른다. 실제 플레이어의 전략 의사결정(언제 교체·언제 메가·언제 Substitute·언제 boost) 모델링이 없다. 메커니즘 층은 정적 데미지 계산만 있다. 동적 동작은 자리 자체가 없다. 교체는 모델링 안 된다.

결과적으로 현재 도구는 "시뮬레이터의 모양을 한 데미지 계산기"다. 합성 데이터에서 95% 권역에 도달하는 것은 우리 ref가 같은 정적 모델로 만든 데이터를 같은 정적 모델 계산기가 재계산하는 trivial한 일치다.

"Pokemon 복제 가능"이라는 표현은 이 위치를 흐린다. 정확한 표현은 "Pokemon의 정적 데미지 모델 일부를 슬라이더 개입으로 표현했다"이고, 시뮬레이터로서 Pokemon을 복제하지는 않았다.

§1의 최종 목표 — "턴제 구조 게임은 사용자 개입만으로 복제 가능" — 의 정직한 위치는 다음과 같다. 정적 차원의 슬라이더는 작동한다(type_table·STAB·카테고리 라우팅·무브 풀·기믹 채널). 동적 차원의 슬라이더는 자리 자체가 부재다(메가/특성/우선도/Substitute/boost/회복/상태이상). 본 phase의 작업은 동적 차원의 자리를 만드는 것이다.

## 4. 본 phase 작업 trajectory

### 4-1. 동적 메커니즘 hook 인프라

engine.py 9단계 파이프라인에 새 hook 단계를 추가한다. ON_TURN_START(턴 시작), ON_MOVE_USE(무브 사용 직전), ON_TURN_END(턴 종료), ON_SWITCH(교체 시), ON_BATTLE_START(전투 시작). 기존 `_PHASE_TO_EVENT`(engine.py L146~L157) 매핑과 `_notify_event`를 활용한다. action_registry에 새 hook 단계를 등록 가능한 구조인지 확인이 필요하다.

캐릭터 instance에 동적 상태 필드를 추가한다. `current_type`(Protean용 동적 타입), `boost_levels`(Calm Mind 같은 누적 boost 카운터 dict), `substitute_hp`(Substitute 보조 HP 풀), `status_condition`(상태이상 토큰), `priority_override`(우선도 무브 표시), `mega_evolved`(메가 진화 플래그).

game_config에 "mechanisms" 키를 추가한다. 사용자가 폼으로 부착하는 메커니즘 라이브러리의 인덱스.

### 4-2. 표준 메커니즘 라이브러리

단순한 것부터 한 개씩 PR. 각 PR은 engine hook 호출 추가 + step2 입력 UI 추가 + 빌더·라운드트립·검증.

권고 순서. Leftovers(턴 종료 % 회복)가 가장 단순하다. Trace는 ON_SWITCH hook에서 진입 시 상대 특성을 복사한다.

각 메커니즘마다 step2에 입력 UI를 추가한다 — Move System 섹션 또는 별도 섹션에 "메커니즘 부착" expander. 캐릭터별 또는 무브별로 어느 메커니즘이 활성인지 선택. `game_config["mechanisms"]`에 저장.

## 5. 핵심 방법론 주의사항

### 5-1. Bash mount silent truncation

bash로 큰 워크스페이스 파일을 cp/cat하면 조용히 잘린다. engine.py ~890줄, step2 ~960줄(현재 738줄, 본 phase 작업으로 더 늘 것), step6 1349줄. wc, py_compile, diff가 거짓 에러를 낸다.

이전 변종. Edit 도구로 mount 경로에 큰 파일(~13KB)을 쓸 때 silent truncation(null bytes 삽입). Write 도구도 mount 영역에서 ~13KB 경계 silent truncation. bash sandbox의 Python `open().read()`도 silent truncation. step6 ~80KB에서 mid-statement 끊김. wc -l 거짓 결과(D5에서 1322 vs 실제 1349, 본 대화에서 step2 706 vs 실제 729). py_compile이 거짓 SyntaxError 보고.

규칙. 워크스페이스 파일은 Read 도구로 읽는다. bash와 bash 안의 Python 모두 금지. bash py_compile/wc/sed/tail/Python read가 알 수 없는 에러를 내면 Read 도구로 진실 확인. Read와 bash가 다른 답을 주면 Read가 진실이다. 검증은 Grep 마커와 변경 구역 Read 정독으로 한다. 샌드박스 자체 파일(`~/...`, `outputs/`)은 bash로 다뤄도 안전. 단 그 안에서 큰 mount 파일을 Python으로 읽는 건 위험. 디렉터리 목록(`ls`)이나 소형 파일(<10KB)은 bash OK. 큰 파일 작성은 bash heredoc(`cat > path << 'HSEND'`) 우회. Write 도구로 큰 파일을 mount에 쓸 때 truncation 위험이면 bash heredoc 우회.

### 5-2. 검증 워크플로 (Antigravity 납품 후)

먼저 Glob 또는 bash `ls`로 신규 파일 존재 확인. 다음 Grep으로 새 마커와 옛 마커 (negative grep) 모두 확인.

라인 수 산술 — 이전 라인 수 + (replace 줄수 − find 줄수) = 새 라인 수가 정확히 일치하는지 본다. 단 bash wc -l가 거짓 실패할 수 있음. 의심되면 Read 도구로 끝부분 확인.

변경 블록을 Read 정독해서 프롬프트의 바꾸기 블록과 바이트 동등성(MD5)을 본다.

클린룸 컨텍스트 py_compile은 bash 거짓 실패 가능성 인지하고 본다. SyntaxError 보고 시 line 번호의 Read 정독으로 진실 확인.

### 5-3. 프롬프트 작성 워크플로

변경 사이트를 Grep과 Read로 정확한 컨텍스트 확정한다. 변경하지 않을 영역의 디자인 정책도 정찰한다(특히 layout, sidebar, fullscreen 모드, CSS 정책). `outputs/verify_*.py` 하니스 작성. 새 코드를 실제 들여쓰기로 재현. py_compile + 가능한 경우 기능 단위 테스트. 빌더 스크립트(Python heredoc)로 .md 조립. find/replace 블록을 하니스에서 자동 추출 또는 빌더에 string 상수로 박음. 직접 타이핑 금지. 라운드트립 무결성(.md 쓰기 → 되읽기 → python 블록 재추출 → MD5 대조 → 컨텍스트 py_compile). `present_files`로 사용자에게 전달.

### 5-4. 기타 함정

`run_simulation`은 3-튜플 반환이다. `winner, logs, sim_metrics = run_simulation(...)`. `run_simulation`과 `run_monte_carlo`는 `game_config` 파라미터 받음. `_worker_simulate_match`는 15-튜플 인자 시그니처. 샌드박스 `/tmp`는 권한 문제 있다. `outputs/` 디렉터리 사용. Glob 도구가 한국어 경로 검색 시 가끔 빈 결과 반환. 진실은 bash `ls`다.

step5_discrepancy.py L29의 16 trailing space 같은 보이지 않는 공백이 byte-equality 깨뜨릴 수 있음. find 블록을 unique 단일 라인으로 좁히면 우회 가능.
