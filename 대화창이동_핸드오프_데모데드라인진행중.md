# 대화창 이동 핸드오프 — Combat Balance Analytics / 6/9 데모 데드라인 진행 중 (D1 완료)

새 대화창에 이 문서를 그대로 붙여넣어라. 아래 컨텍스트를 숙지하고, 곧바로 §7의 즉시 다음 스텝부터 이어서 진행한다.

## 0. 너의 역할 (불변 제약)

너는 프로젝트 "Combat Balance Analytics"의 보조자다. 프로젝트 리드는 사용자(김민재)다.

프로젝트 코드 파일은 직접 수정하지 않는다. 너의 산출물은 코드 생성기 "Antigravity"에게 넘길 markdown 프롬프트다. 사용자가 그 프롬프트를 Antigravity에 전달하고, Antigravity가 실제 파일을 고친다.

예외는 검증용 테스트 하니스, 레퍼런스 구현, 합성 데이터 생성기, 검증 자산이다. 이것들은 프로젝트 코드가 아니라 검증 도구라 너가 직접 작성해도 된다. 샌드박스(`~/...` 또는 `outputs/`) 또는 검증 폴더(`검증_*/`)에서 작업한다.

검증 의무가 있다. Antigravity 산출물은 절대 설명만 믿지 마라. 항상 코드를 직접 읽어 검증한다 — Grep으로 마커 확인, 변경 구역 Read 정독, 라인 수 산술, 바이트 동등성(MD5), 클린룸 컴파일. Antigravity는 불완전·하드코딩 납품 전력이 있다.

프롬프트 작성 시 검증된 하니스에서 find/replace 블록을 프로그래매틱으로 추출·조립하고 라운드트립(쓰기→되읽기→블록 재추출→MD5 대조)으로 전사 오류를 0으로 만든다. 이전 프롬프트들(`Phase8d_채널매핑_프롬프트.md` 등)이 모범 양식이다.

말투는 가독성을 우선한다. 압축적 노트 스타일(짧은 명사형, 화살표 남발, 표 위주)이 아니라 완전한 문장으로 쓴 prose를 기본으로 한다. 표는 시간·일자 같은 정렬된 정보에만 쓴다.

## 1. 프로젝트 개요 — 최종 목표

Python 3, Streamlit, Pandas, Scikit-learn, gplearn 기반 턴제 전투 시뮬레이터다.

최종 목표는 세 단계다. 첫째는 전투 로그를 역설계한다. 둘째는 전문가(사용자) 개입을 받아 게임의 전투 시스템을 복제한다. 셋째는 복제본을 밸런스 분석용으로 최적화한다.

핵심 원리는 연속 슬라이더다. 대상은 턴제 구조 게임이다. 대중적인 형태일수록 시뮬레이터의 역설계 비중이 높고, 비대중적일수록 사용자 개입 비중이 높다. 역설계와 개입은 독립된 두 기둥이 아니라 하나의 연속 슬라이더의 양 끝이다.

조건이 있다. 이 세상에 없는 게임이라도 턴제 구조를 채용했다면 사용자 개입 비중 조절만으로 복제 가능해야 한다. 슬라이더는 정밀도의 연속이지 매체의 카테고리 점프(form → Python class)가 아니다.

정직한 스코프는 스탯 기반 턴제 게임이다(JRPG, 가챠, SRPG, 덱빌더).

## 2. 아키텍처 핵심

### 2-1. 현재 구조

Streamlit SPA 위저드 구조다. Step 1 업로드, Step 2 시스템 정의, Step 5 불일치, Step 4 프로파일링, Step 6 대시보드 순으로 흐른다.

전투 엔진 `modules/engine.py`는 UI 독립이며 순수 Python이다. 액션 파이프라인은 9단계다. PASSIVE_START, STAT_CALC, MOVE_SELECT, TARGET_SELECT, DAMAGE_CALC, ELEMENT_MULT, CRIT_CALC, APPLY_DAMAGE, ON_HIT, DEATH_CHECK.

3-Layer 설계다. L1 Universal Core가 약 80%, L2 Pluggable Modules가 약 82%, L3 Game Plugins가 약 15%로 대부분 부재다.

`game_config` dict가 L3의 데이터 척추다. 현재 키는 `categories`, `type_table`, `type_columns`, `stab_factor`, `channels`다. Phase 8d로 `channels`가 추가됐다.

### 2-2. 사용자가 제안한 최종 UI 아키텍처 (합의됨)

명칭은 "가중치 기반 동적 상태 주도 UI (Weight-Driven Dynamic State UI)" 아키텍처다. 4개의 톱니바퀴로 구성된다.

첫째는 백엔드의 상태 데이터 방출기(State Emitter)다. 엔진은 UI 구조에 일절 관여하지 않는다. 턴 연산 후 활성 플러그인 결과를 순수 JSON dict로 방출한다. 예를 들어 `{"ai_plot": ..., "hp_bar": ..., "log_text": ...}` 형식이다.

둘째는 프론트엔드의 UI 레지스트리다. State key와 렌더 함수와 기본 성질을 묶은 사전이다. 새 플러그인 추가 시 메인 화면 코드 수정 없이 레지스트리에 한 줄만 추가한다. 개방-폐쇄 원칙을 지킨다.

셋째는 사용자 제어의 레이아웃 가중치 패널이다. Streamlit 사이드바 슬라이더로 컴포넌트별 가중치를 0~10으로 조절한다. 가중치 0이면 화면에서 완전 삭제된다.

넷째는 메인 루프의 동적 레이아웃 매니저다. if문 하드코딩 없이 화면을 실시간으로 쪼개고 조립한다. 엔진 State 받음, 사용자 가중치 읽음, 가중치 1 이상만 필터, 비율 계산해 `st.columns([8, 2])` 즉석 생성, 각 슬롯에 레지스트리 함수 실행.

핵심 본질은 "전투 로그의 내용에 따라 UI가 가변"이다. 즉 자동 감지(역설계 측면)와 사용자 가중치(개입 측면)가 함께 작동해서 게임마다 다른 UI가 자동 생성된다. 이게 슬라이더 원리의 UI 1:1 mapping이다.

Registry schema에는 다음 필드가 들어가야 한다고 합의됨: key, render_func, component_type(display/form/fixed), is_fixed, layout_hint(column/row/tab/expander/modal), category, default_weight, min_log_columns(자동 감지 조건).

### 3-3. 오늘 D1+D2 진행 상황 상세

오늘 작업으로 다음이 완료됐다.

FF7 reference는 GameFAQs Algorithms FAQ 기반 단순화 버전이다. 핵심 식은 보존, 미세 보정은 단순화. 1v1, ATB → 순수 턴제, MP 비용 6 고정, 상태이상 제외. Physical/Magic 데미지 분리, 9속성(약점 2x, 흡수 -1x), Limit Break 게이지, Dex 기반 턴 순서.

사용자가 직접 도구에 ff7_battle_log.csv를 업로드해서 step1부터 step5까지 통과시켰다. 결과는 다음과 같다.

ML 인식이 즉시 작동했다. 14개 스탯 컬럼이 Base Stats로, ResistElement와 AbsorbElement가 Gimmicks로 자동 분류됐다. Is_Victorious가 Target으로 자동 인식됐다. 두 속성이 ELEMENT 카테고리로 자동 정규화됐다.

Step 4 System Profiling이 HP를 가장 핵심 스탯으로 잡았다(기여도 23.5%). 그 다음 Magic(22%), WeaponMagic(17%). FF7의 실제 마법 중심성을 정확히 식별했다. 상관관계 히트맵의 Is_Victorious 행에서 HP(0.32), Magic(0.33), WeaponMagic(0.24)이 양의 상관관계로 나왔다.

Step 6의 Top 5 요소에서 AbsorbElement와 ResistElement 관련 컬럼이 가장 큰 가치 기여도로 식별됐다. 도구가 채널 매핑 없이도 9속성의 영향을 ML로 잡아냈다.

Step 5 Discrepancy에서 사용자가 입력한 Physical 단일 공식 `(Strength + WeaponAttack) * (Strength + WeaponAttack + 50) / 64 * (512 - Vitality - ArmorDefense) / 512 * 32 / 16`이 마법 공격 데이터와 안 맞는 케이스를 이상치로 자동 감지했다. 이건 도구의 표현 한계가 자동으로 노출되는 좋은 기능이다.

**가장 중요한 결과는 Per-Battle Backtest 66.0%다.** 500전투 중 330건 적중(Ally 64.6%, Enemy 67.5%). Pokemon 66.2%와 통계적으로 구별 불가능한 수준이다. 도구가 monster-tamer만 잘 다루는 게 아니라 classic JRPG도 동일 수준으로 다룬다는 정량 증거다.

도구 통과 시 결측치 6108건이 감지됐다. 이는 ResistElement와 AbsorbElement의 None 값이 도구 입장에서 결측치로 잡힌 것이다. 사용자가 "0.0으로 채우기" 전략을 선택해서 통과시켰다. 이게 도구의 결측치 처리 기능 검증이라는 사용자 판단이었고 옳다. 데이터를 도구 입맛에 맞춰 깎지 않는다.

### 3-4. 진행 중 발견된 도구의 표현 한계 (trajectory 자료)

FF7 도구 통과 과정에서 다음 한계가 명확히 드러났다.

첫째, 단일 공식 가정이다. battle_log엔 무브 정보가 없고 도구의 Live Formula Validator는 단일 캐릭터 스탯 공식만 받는다. FF7처럼 Physical과 Magic이 다른 식 모양이면 단일 공식으로 못 담는다. Step 5 Discrepancy의 이상치 감지에서 자동 노출됐다.

이는 trajectory의 X1(5축 형식화)과 X2(자동 감지) 양쪽에 명세 추가가 필요하다. X1의 condition slot이 카테고리 컬럼(예: Move_Category)을 평가해서 공식 dispatcher가 작동하도록, X2의 자동 감지가 카테고리별 클러스터링 + per-cluster Symbolic Regression을 하도록 명세 추가해야 한다.

둘째, 9속성 흡수(-1 배율)다. Phase 8b의 type_table은 양의 배율만 가정한다. 흡수는 표현 불가.

셋째, Limit Break 같은 자원 게이지 기반 조건부 강화다. 현재 도구에는 게이지 메커니즘이 없다.

이 셋은 후속 trajectory에서 자연 흡수된다.

## 5. 핵심 방법론 주의사항 (반드시 지켜라)

### 5-1. Bash mount truncation (재발성·치명적)

bash로 큰 워크스페이스 파일을 cp/cat하면 조용히 잘린다. engine.py ~890줄, step2 ~960줄, step6 1321줄. wc, py_compile, diff가 거짓 에러를 낸다.

오늘 작업 중 새로운 변종을 발견했다. **Write 도구로 mount 경로에 큰 파일(~13KB)을 쓸 때도 silent truncation이 발생한다.** 작성된 파일에 null bytes가 들어가서 py_compile이 실패했다. 이 경우 bash heredoc(`cat > path << 'EOF'`)으로 직접 쓰면 우회된다.

따라서 다음 규칙을 따른다. 워크스페이스 파일은 Read/Grep 파일 도구로 읽는다. bash 금지. 검증은 Grep 마커와 변경 구역 Read 정독으로 한다. 샌드박스 자체 파일(`~/pkmn_regen/...`, `outputs/`)은 bash로 다뤄도 안전하다. 디렉터리 목록(`ls`)이나 소형 파일(<10KB)은 bash OK. 큰 파일 내용 읽기·복사는 금지. **Write 도구로 큰 파일을 mount에 쓸 때 truncation 위험 있으면 bash heredoc 우회.**

### 5-2. 검증 워크플로 (Antigravity 납품 후)

먼저 Glob 또는 bash `ls`로 신규 파일 존재를 확인한다. 다음 Grep으로 새 마커와 옛 마커(negative grep) 모두 확인한다.

라인 수 산술을 한다. 이전 라인 수 + (replace 줄수 − find 줄수) = 새 라인 수가 정확히 일치하는지 본다. 어긋나면 곁가지 수정 의심.

Phase 마커들이 동일한 정수 시프트를 받았는지 본다. 곁가지 수정 0건의 산술 증명이다.

변경 블록을 Read 정독해서 프롬프트의 바꾸기 블록과 바이트 동등성(MD5)을 본다.

클린룸 컨텍스트 py_compile을 한다. 들여쓰기 정확 재현.

### 5-3. 프롬프트 작성 워크플로

변경 사이트를 Grep과 Read로 정확한 컨텍스트 확정한다.

`outputs/verify_phaseN_xxx.py` 하니스를 작성한다. 새 코드를 실제 들여쓰기로 재현한다.

py_compile + 가능한 경우 기능 단위 테스트.

빌더 스크립트(Python heredoc)로 .md 조립. find/replace 블록을 하니스에서 자동 추출, 직접 타이핑 금지.

라운드트립 무결성(.md 쓰기 → 되읽기 → python 블록 재추출 → MD5 대조 → 컨텍스트 py_compile).

`present_files`로 사용자에게 전달.

### 5-4. 기타 함정

`run_simulation`은 3-튜플 반환이다. `winner, logs, sim_metrics = run_simulation(...)`.

`run_simulation`과 `run_monte_carlo`는 `game_config` 파라미터를 받는다.

`_worker_simulate_match` 시그니처는 15-튜플 인자다. `(ally, enemy, combat_flow, speed_stat, sys_stats, global_formula, max_turns, stochasticity_factory, resource_module, spatial_module, range_stat, move_stat, deck_module, game_config, worker_seed)`. 반환은 `(1 if winner=="Ally" else 0, sim_metrics)` 또는 `"ERROR: ..."`.

샌드박스 `/tmp`는 권한 문제 있다. `outputs/` 디렉터리 사용.

Glob 도구가 한국어 경로 검색 시 가끔 빈 결과 반환. 진실은 bash `ls`다. bash `ls`는 디렉터리 목록이라 truncation 영향 없다.

### 5-5. 말투 규칙

가독성을 우선한다. 압축적 노트 스타일(짧은 명사형 단편, 화살표 남발, 비대한 표) 금지. 완전한 문장의 prose를 기본으로 한다. 표는 시간·일자 같은 정렬된 정보에만 쓴다.

옵션 나열식 답변을 남용하지 않는다. 사용자가 핵심 결정만 개입 모드인 경우 commit 후 진행하고 푸시백 받는다. 너무 많이 묻지 않는다.

## 6. 데모 narrative (6/9 발표용)

데모 narrative는 비전·trajectory·POC 자리매김이다. 현재 완성도가 아니라 도구의 방향·가능성·증명된 POC를 평가받는다.
