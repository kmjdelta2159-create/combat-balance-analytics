# 교체(switch) 모델 설계안 — 전투 모델 재설계

## 0. 이 문서의 위치

직전 phase(동적 메커니즘 PR-A~G) 완료 후, 핸드오프 §5가 "최종목표 기준 남은 최대 구조적
격차"로 지목한 교체(switch) 모델에 착수한다. 사용자가 "여러 갈래가 있으면 항상 최종목표를
기준으로 선택하라"고 결정 위임했고, 그 기준으로 교체 모델이 선택됐다 — 교체가 없으면
Pokemon 전투 흐름 전체가 재현되지 않고, Trace(진입 시 특성 복사)·교체 전략·메가 진화
타이밍 결정의 substrate 자체가 없기 때문이다.

이 문서는 정찰 결과와 설계 결정, PR 분할을 담는다. 사용자 검토 후에야 Antigravity 프롬프트
빌더로 넘어간다. 본 보조자는 프로젝트 코드를 직접 수정하지 않는다.

### 0-1. 지배 원칙 (사용자 재강조)

최종목표는 세 단계다 — 전투 로그 역설계, 전문가 개입을 받은 전투 시스템 복제, 복제본의
밸런스 분석 최적화. 핵심 제약은 "이 세상에 없는 게임이라도 턴제 구조를 채용했다면 *사용자
개입 비중 조절만으로* 복제 가능해야 한다"이다. 역설계와 개입은 하나의 연속 슬라이더의 양
끝이지 매체의 카테고리 점프가 아니다.

따라서 교체는 "Pokemon 기능"으로 엔진에 박지 않는다. 교체는 *턴제 구조 게임이면 사용자
개입으로 켜고 파라미터화할 수 있는 범용 1급 능력* — "액티브/예비(reserve) 회전"이라는
도메인 중립 프리미티브다. 코드에 게임 이름이나 특정 게임 전용 규칙 분기를 두지 않는다
(D5-hotfix2에서 게임 preset 버튼을 없앤 그 원칙). 회전의 활성·파라미터·진입 효과는 전부
game_config와 기존 메커니즘 부착 UI를 통한 개입으로만 정해진다. 미설정이면 현행 전원-동시
동작 그대로(회귀 0).

## 1. 현재 전투 모델의 정직한 구조 (정찰 결과)

정찰은 `engine.py`(run_simulation 및 액션 함수), `turn_manager.py`, `win_condition.py`,
`per_battle_backtest.py`를 Read 도구로 정독해 확정했다. 핵심은 다음과 같다.

현재 모델은 "전원 동시 필드" 팀 전투다. Pokemon식 "1마리 액티브 + 나머지 벤치, 교체로
교대"가 아니라, 한 팀의 모든 유닛이 동시에 필드에 올라 매 라운드 전원이 행동한다. 구조적
근거는 세 곳이다.

첫째, 참가자 초기화다. `run_simulation`(engine.py L894~L915)이 ally_instances와
enemy_instances의 *모든* 원소를 단일 평면 리스트 `participants`로 만든다. 각 원소에
`id`(A1, A2.../E1, E2...)와 `team`("Ally"/"Enemy")만 붙는다. 액티브/벤치 구분도, 필드 슬롯
개념도 없다.

둘째, 턴 루프다. `SequentialTurnManager.run`(turn_manager.py L110~L142)이 매 라운드
participants를 speed_stat 기준 정렬한 뒤(L113~L115), 살아있는 *모든* 참가자가 한 번씩
행동한다(L117 `for active_char in participants:`). 벤치 개념이 없으니 "이번 턴에 누가
필드에 있는가"라는 게이팅이 없다.

셋째, 타겟팅이다. `_act_target_select`(engine.py L310~L311)는 타겟 후보를 "상대 팀에서
살아있는 *전원*"으로 잡는다. `_act_move`의 이동 대상도 동일(L359~L361). 즉 한 유닛이 상대
팀 전체를 상대로 타겟을 고른다.

승패 판정은 `ResourceDepletion`(win_condition.py L36~L44)이다. HP가 살아있는 참가자들의
team 집합이 1개 이하로 줄면 종료한다. 이건 "한 팀 전멸"이 곧 패배라는 뜻이고, 액티브/벤치를
구분하더라도 벤치 유닛이 participants에 남아 있는 한 그대로 재사용 가능하다.

턴 실행기 `StandardTurnExecutor`(turn_manager.py L25~L63)는 pre-target / per-target /
post-target 3슬롯이다. 한 유닛의 한 턴은 "pre 액션들 → 타겟별 per 액션들 → post 액션들"
순서로 고정돼 있고, 무브 선택은 `_act_move_select`(engine.py L380~L431)가 movepool에서
그리디(기대 데미지 최대) + setup_first 정책 분기로 한다. 즉 한 턴 = 반드시 "공격 1회"이며,
"이번 턴엔 공격 대신 교체한다"는 *상호배타적 액션 선택*의 자리가 없다.

동적 hook은 ON_TURN_END·ON_STATUS_TICK(post-level)과 ON_MOVE_USE·ON_MOVE_EFFECT
(target-level)이 있다(engine.py L120, 등록 L758 부근). **ON_SWITCH hook은 없다.** 직전
핸드오프들이 ON_SWITCH를 "계획"으로 언급했으나 코드에는 미구현이다.

백테스트 데이터 표현은 `build_battles`(per_battle_backtest.py L73~L104)다. 한 전투의 N행을
절반으로 잘라 앞 N/2 = Ally 전원, 뒤 N/2 = Enemy 전원으로 만든다. 즉 로그가 "교체"라는
사건을 표현할 자리가 데이터 형식 자체에 없다.

## 2. 최종목표 기준 격차

시뮬레이터의 정의(핸드오프에서 확립)는 turn-by-turn 흐름 재구성 + 동적 메커니즘 + 전략
의사결정이다. Pokemon을 시뮬레이터로 복제하려면 교체가 1급 액션이어야 한다. 현재 구조로는
다음이 불가능하다.

한 유닛이 자기 턴에 공격 대신 교체를 선택하는 것. 액티브가 쓰러지면 벤치에서 다음 유닛을
내보내는 것(강제 교체). 진입 시 효과(Trace 특성 복사, 입장 해저드, Intimidate)를 거는 것.
"불리한 매치업이면 교체한다" 같은 전략 결정. 그리고 이 모두가 막혀 있으면 메가 진화 타이밍
결정도 올라설 토대가 없다.

이건 메커니즘 1종 추가가 아니라 *전투 모델의 구조 변경*이다. 그래서 작은 PR 여러 개로
쪼개되, 첫 PR이 "액티브/벤치 구분 + 강제 교체"라는 골격을 세우고 그 위에 자발적 교체 →
ON_SWITCH hook → 전략 정책 순으로 얹는다.

## 3. 설계 결정 (forks) + 권고

세 갈래에 결정이 필요하다. 최종목표 기준으로 권고안을 제시하되, 사용자 검토에서 뒤집을 수
있다.

### 결정 A — 액티브 모델의 일반성

§0-1 원칙에 따라 설계는 도메인 중립이어야 한다. 팀별 액티브 수 `active_count`를 설정값으로
두어 싱글(1), 더블(2), 전원-동시(active_count = 팀 크기, 현행)를 *한 메커니즘으로* 표현한다.
어느 값도 특정 게임 이름과 묶이지 않는다 — 사용자가 개입으로 고르는 연속 파라미터다.

싱글(active_count=1)은 *설계 가정이 아니라 첫 검증 슬라이스*일 뿐이다. 골격이 가장 단순하고,
사용자가 가져온 외부 실제 데이터가 마침 OU 싱글이라 라이브 검증 대상과 맞아서다. 더블·
N-액티브는 슬롯 수만 늘리면 같은 코드 경로로 동작해야 한다.

권고: `active_count` 설정값 도입(범용, 게임명 분기 0). 첫 PR은 싱글로 검증. active_count =
팀 크기일 때 *현행 전원-동시 동작과 완전 동일*해 회귀 0.

### 결정 B — 기존 평면 participants 모델과의 통합 방식

선택지 둘. (1) 평면 participants 리스트를 유지하되 각 참가자에 `on_field`(bool)와 슬롯
인덱스를 붙이고, 턴 루프가 on_field=False인 유닛을 건너뛰게 한다. 교체는 플래그를 토글한다.
win_condition·hook·targeting은 "on_field 추가 필터"만 더하면 된다. (2) 진짜 Team 객체(로스터
+ 액티브 슬롯)를 도입한다. 더 깨끗하지만 run_simulation·turn_manager·per_battle_backtest의
여러 호출부를 동시에 갈아엎어야 한다.

권고: (1) 플래그 기반 점진안. Antigravity 한 번에 큰 변경은 위험하다는 운영 원칙과, 작은
단계로 쪼개 검증한다는 방법론에 부합한다. active_count = 팀 크기면 전원 on_field가 되어 회귀
0. 더블/일반 액티브로의 확장도 슬롯 수만 늘리면 된다.

### 결정 C — 백테스트 데이터 표현 (검증 경로)

교체 모델의 가치는 *구조적*이다(Trace·메가·전략의 토대). 그런데 `build_battles`는 교체를
표현할 자리가 없고, 합성 ref도 교체 로그를 생성하지 않는다. 합성 데이터를 교체까지 표현하도록
재설계해 거기에 대고 정확도를 재는 것은, 우리 ref가 만든 단순화를 우리 계산기가 재계산하는
순환이라 신호가 약하다.

권고: 백테스트 통합은 *분리해서 뒤로* 미룬다. 첫 검증 경로는 (a) 손으로 짠 싱글 시나리오
(액티브 1 + 벤치 1, 액티브가 쓰러지면 벤치 등장)에서 라이브 단일 전투 로그로 강제 교체
발화 확인, (b) 자발적 교체 정책이 붙으면 같은 방식으로 교체 액션 발화 확인이다. 합성 ref
확장이나 외부 PS replay 어댑터(핸드오프 §4-4)는 별도 결정 항목으로 남긴다.

## 4. 표준 설계 (권고안 A=싱글 PoC / B=플래그 / C=라이브 검증 기준)

### 4-1. 참가자 필드 상태

`run_simulation` 참가자 초기화(engine.py L894~L915)에서 각 참가자에 `on_field`(bool)와
`field_slot`(int) 또는 팀 내 로스터 순서 `roster_idx`를 부여한다. 팀별로 앞에서부터
active_count명을 on_field=True, 나머지를 False로 둔다. game_config에 `active_count`(기본:
팀 크기 → 회귀 0) 키를 추가한다.

### 4-2. 턴 루프 게이팅

`SequentialTurnManager.run`(turn_manager.py L117)의 행동 루프가 on_field=True인 유닛만
턴을 갖도록 게이팅한다. 정렬(L113)도 on_field 유닛만 대상으로 한다. active_count = 팀
크기면 전원 on_field라 현행과 동일.

### 4-3. 액티브 사망 시 처리 (설정 규칙, 하드코딩 아님)

액티브 유닛이 쓰러졌을 때의 처리는 엔진에 박지 않고 game_config의 규칙으로 둔다 —
`on_active_faint`: `"replace_from_reserve"`(예비에서 자동 등장) / `"team_loss"`(즉시 팀 패배)
/ `"none"`(아무 일 없음). 이 한 규칙으로 회전 있는 게임, "쓰러지면 끝"인 게임, 회전 없는
게임을 모두 표현한다. 기본값은 현행 동작 보존(예비 슬롯이 없으면 전원-동시라 이 규칙이
무동작 → 회귀 0).

`"replace_from_reserve"`일 때: 라운드 종료 또는 유닛 사망 직후, 어떤 팀의 on_field 유닛이
전멸했지만 예비에 살아있는 유닛이 남아 있으면 다음 예비 유닛을 on_field로 승격한다. 턴을
소모하지 않는 교대다. 위치는 win_condition.check 직전(turn_manager.py L129 부근) — 승패 판정
전에 "예비가 남았으면 아직 안 짐"을 반영해야 ResourceDepletion이 오판하지 않는다. 즉 사망
처리 규칙 적용 후 ResourceDepletion 판정.

### 4-4. ON_SWITCH hook

새 hook 키 ON_SWITCH를 등록한다. 강제 교체와 자발적 교체 양쪽에서 새로 들어온 유닛에 대해
브로드캐스트한다. 첫 구현은 no-op 슬롯(부착된 메커니즘 없으면 아무 일 없음)으로 두고, Trace
같은 진입 메커니즘은 이 hook 위에 후속 PR로 얹는다. `_PHASE_TO_EVENT`·`_broadcast_phase_event`
기존 패턴(engine.py L221 등) 재사용.

### 4-5. 타겟팅 제한

`_act_target_select`(engine.py L310~L311)와 `_act_move`(L359~L361)의 상대 후보를 "상대 팀
전원"에서 "상대 팀 *on_field* 전원"으로 좁힌다. active_count=1 싱글이면 자연히 단일 상대.
active_count = 팀 크기면 전원 on_field라 현행 동일(회귀 0).

### 4-6. 자발적 교체 액션 + 전략 정책 (후속 PR)

한 유닛의 턴에 "공격 vs 교체"를 고르는 분기. `_act_move_select`(engine.py L380~) 또는 그
앞단에 정책 hook을 둬서, 정책이 "교체"를 반환하면 per-target 공격 슬롯을 건너뛰고 교체
액션을 실행한다. 첫 정책은 룰 기반 최소형(예: 현재 매치업 타입 배율이 불리하면 1회 교체).
직전 phase의 setup_first 정책 분기(engine.py L388~L399)가 "정책이 무브 선택을 가로채는"
선례라 같은 자리·같은 양식으로 확장 가능하다.

## 5. PR 분할 (작은 단계, 각 PR 납품·검증·보고)

PR-S1. 필드 상태 골격 — `active_count` 설정 + on_field/roster 부여(engine.py 참가자 초기화)
+ 턴 루프 on_field 게이팅(turn_manager) + `on_active_faint` 규칙 처리 + win_condition 순서.
자발적 교체 없음. active_count·on_active_faint 미설정 시 회귀 0. **검증: 라이브 싱글
시나리오(on_active_faint="replace_from_reserve")에서 액티브 사망 시 예비 등장 로그.**

PR-S2. 타겟팅 on_field 제한 — `_act_target_select`·`_act_move`. **검증: 싱글에서 상대 액티브
1마리만 타겟, 전원-동시 설정에서 현행 동일.**

PR-S3. ON_SWITCH hook 인프라 — hook 등록 + 강제/자발 교체 시 브로드캐스트(no-op 슬롯).
**검증: 교체 발생 시 hook 발화(부착 메커니즘 없으면 무동작).**

PR-S4. 자발적 교체 액션 + 최소 룰 정책 — `_act_move_select` 앞단 정책 분기 + 교체 액션.
**검증: 불리 매치업 시나리오에서 교체 액션 발화.**
