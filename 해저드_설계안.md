# 해저드(진입 데미지) 설계안 — 진입 hook 위 두 번째 메커니즘

## 0. 이 문서의 위치

S-트랙(Trace + 진입 즉시 타이밍, S6~S8) 라이브 실증 후, "다음" 갈래로 **해저드(Stealth
Rock류 진입 데미지)** 를 고른다. 방금 만든 진입 hook `_fire_switch_in` 위에 두 번째 진입형
메커니즘을 얹어 진입 효과 계열을 확장한다. 본 보조자는 코드를 직접 수정하지 않는다 — 정찰 →
설계안 → 사용자 검토 → 작은 PR 분할 → Antigravity 프롬프트 → 검증.

## 1. 현재 구조 (정찰 결과, Read 기준)

진입 hook. `_fire_switch_in(char, participants, game_config, add_log)`(engine, S7)이 교체 진입
즉시 호출된다 — 자발적 교체(`_maybe_voluntary_switch`)·강제 교체(turn_manager `_resolve_faint`
콜백) 양쪽에서. 현재 진입 효과(`_apply_switch_in_effects`: Trace) + 진입 이벤트 + 로그를 부른다.
**초기 on_field 배치(run_simulation "1b")는 _fire_switch_in을 호출하지 않는다** → 리드(첫 액티브)는
해저드를 안 맞고, **교체로 들어온 유닛만** 맞는다(Pokemon 해저드와 동일).

데미지 적용. engine은 `apply_delta(char, -dmg)`(resource.py)를 import한다. 퍼센트 데미지의 기존
모델은 `_act_status_tick` — `res = char['resources'][vital]`에서 `res["current"] = max(0, current
- max*percent)`. 해저드도 같은 percent-of-max로 진입 유닛에 적용하면 된다(HP=PRIMARY_RESOURCE
기본).

메커니즘 config 패턴. `game_config['mechanisms'][name]` = spec dict. Leftovers/상태이상/Protean/
Trace 모두 `gimmick_col`/`match_value`로 부착 판정. 단 해저드는 "캐릭터 부착"이 아니라 "필드(팀)
조건"이라 부착 모델이 다르다 — 이게 §2의 핵심 결정.

## 2. 핵심 결정 — 해저드를 어떻게 설치/저장하나

해저드는 캐릭터 속성이 아니라 *팀 진영(field)의 조건*이고, 본래 *전투 중 무브로 설치*된다.
이 둘을 어떻게 모델링하느냐로 두 갈래가 갈린다.

(A) **정적 해저드 (config 상수).** `game_config['mechanisms']['hazard']` = {team_scope, percent}.
지정 진영으로 *교체 진입하는* 유닛이 매번 percent 데미지를 받는다. 설치 무브·필드 상태 없음 —
`_fire_switch_in`에서 그 진영 진입이면 데미지. **장점**: 필드 상태 substrate 불필요, 리드 면제가
자연히 떨어짐(_fire_switch_in은 교체에서만 호출), 회귀 0 쉬움(미설정 no-op), 진입 hook 직접
재사용. **단점**: "무브로 설치하는 셋업 다이내믹"이 없다(상시 진입세).

(B) **무브-설치 동적 해저드.** 무브가 상대 진영에 해저드를 *설치*하고(필드 상태), 그 진영
진입자가 데미지를 받으며, 청소 무브로 제거. **장점**: 충실(셋업 전략 재현). **단점**: 엔진에
*필드(팀) 레벨 상태* substrate가 없다 — 현재 상태는 전부 participant의 active_states에 산다.
저장 위치를 새로 정해야 한다(아래), 무브→해저드 설치 배선 필요, 병렬 sim에서 상태 공유 주의.

(B)의 필드 상태 저장 후보: ① `game_config`에 런타임 sub-dict(`_field_state`) 추가 — 간단하나
game_config는 본래 정적 config라 병렬 sim에서 복사/공유 여부 확인 필요(미확인). ② 별도 battle-
state 객체를 turn 루프에 스레딩 — 큰 리팩터(그런 객체가 없음). 둘 다 비자명.

## 3. 권고 — H1(정적) 먼저, H2(동적)는 필드-상태 결정과 함께 후속

S6(효과)→S7(타이밍)을 나눠 위험을 격리한 것과 같은 철학으로, **정적 해저드(H1)를 먼저**
넣는다 — 진입 데미지 경로를 hook 위에서 실증하고(낮은 위험·필드 상태 불필요), 무브-설치
동적화(H2)는 필드-상태 substrate를 정식으로 정할 때 의도적으로 들어간다. H1만으로도 "교체로
들어오는 유닛이 진입세를 맞아 교체 남발이 억제된다"는 진입 데미지의 핵심 효과는 실증된다.

## 4. H1 정적 해저드 — 목표 구조

`_apply_switch_in_effects`처럼 ctx 비의존 헬퍼 `_apply_entry_hazard(char, participants,
game_config, add_log)`를 만들고 `_fire_switch_in`이 (Trace 다음에) 호출한다.

```
spec = game_config['mechanisms']['hazard']   # 예: {"team": "Enemy", "percent": 0.125}
```
- `team`: 해저드가 깔린 진영("Ally"/"Enemy"/"both"). char의 team이 거기 해당하면 발동.
- `percent`: vital 자원(HP) max의 비율 데미지. (flat 데미지는 후속 옵션.)
- 적용: `_act_status_tick`과 동일하게 `res["current"] = max(0, current - max*percent)`, 로그
  `-> [Phase: ON_SWITCH] {id} 진입 데미지 {n} (Hazard)`.
- 미설정/team 불일치/자원 없음 시 no-op → 회귀 0.

호출 순서: `_fire_switch_in`에서 Trace(`_apply_switch_in_effects`) → 해저드(`_apply_entry_hazard`)
→ 진입 이벤트 → 진입 로그. (타입 복사 후 데미지가 자연스러움.)

## 5. 정직한 한계 / 엣지

(a) **진입 KO.** 해저드 데미지로 진입 유닛이 그 자리에서 사망할 수 있다. _fire_switch_in은
_resolve_faint 도중 호출되므로, 진입 즉사 시 그 사망은 다음 _resolve_faint 사이클(다음 행동
후) 또는 win_condition에서 정리된다 — 즉시 연쇄 교체는 아니다. H1에선 이 근사를 수용하고
로그로 드러낸다(즉사 시 "진입 데미지"로 0이 됨). 연쇄 즉시 교체는 후속.

(b) **정적 = 상시.** H1은 전투 내내 진입세가 켜져 있다. 무브로 켜고 끄는 다이내믹은 H2.

(c) **team_scope 의미.** 게임 중립상 "어느 진영에 해저드가 깔렸나"를 config가 직접 지정한다
(무브 설치가 없으므로). 사용자 개입으로 "Enemy 진영에 해저드" 식으로 설정.

(d) 멀티 액티브와 무관(진입 단위 처리). 초기 리드는 면제(설계 의도).

## 6. PR 분할

PR-H1. **정적 진입 해저드.** `_apply_entry_hazard` 추가 + `_fire_switch_in`이 호출. engine 한
파일, find/replace 1건(헬퍼 추가 + _fire_switch_in 한 줄 호출 삽입). 회귀 0: hazard 미설정 no-op.
하니스로 team 매칭·percent 데미지·미설정 no-op·자원 없음 검증.

PR-H2(후속, 선택). **무브-설치 동적 해저드.** 필드 상태 substrate 결정(§2-B) + 무브 효과로
해저드 설치/청소 + 진입 체크가 필드 상태를 읽도록 변경. 필드-상태 저장 방식은 별도 설계·검토.

PR-H3. **해저드 부착 UI.** step2 메커니즘 expander에 해저드 체크박스 + team_scope + percent
(Trace·Protean 동일 양식, `_mech_cfg["hazard"]`). 저장은 기존 `_gc["mechanisms"]` 배선 재사용.

## 7. 사용자 결정 대기 항목

권고: PR-H1(정적 진입 해저드) → 라이브 확인 → PR-H3(UI)로 사용자 개입 경로. 동적(H2)은
필드-상태 substrate를 정식으로 정할 때. 승인하면 PR-H1 빌더로 진입한다(find/replace 자동
추출·하니스·납품 후 Read·Grep). H1만으로 충분한지, 처음부터 H2(동적)를 노릴지는 사용자 선호.
