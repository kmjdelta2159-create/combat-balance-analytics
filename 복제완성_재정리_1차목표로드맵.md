# 1차목표 로드맵 (확정형) — 역설계→사용자수정 루프 기준

> 갱신: 이번 대화에서 1차목표 정의·조직원리·순서가 확정됨. 이전판(존재 체크리스트 →
> 검증 루프)을 대체한다. `복제완성_Trajectory_6갈래.md`는 *능력 목록(inventory)*으로 강등돼
> 이 문서에 재료로 먹힌다.

## 0. 확정된 것 (결정 완료)

- **1차목표 = (나) 트레이스-리플레이 충실도.** 전용 시뮬레이터가 없는 게임의 *전투 로그*를
  넣으면(로그는 항상 있다), 엔진이 *로그에 찍힌 행동을 그대로 먹고*(AI는 스코프 밖 — 로그가
  무엇을 했는지 이미 알려줌) *관측된 결과를 수치까지 재현*. 데미지·상성·상태·기절·교체가
  이벤트별 일치하면 합격. "통제된 전투만"이 아니라 "임의의 실제 전투"가 타깃.
- **싱글 먼저, 더블 보류.** 갈래④(멀티 액티브)는 시일 보고 사용자가 나중에 결정.
- **스코프 경계.** 스탯기반 턴제(JRPG·가챠·SRPG·덱빌더). 비-스탯 정보게임(체스·2048)은
  의도된 도메인 밖. AI 의사결정 복제 제외.

## 1. 조직 원리 — 역설계→사용자수정 루프 (이게 척추)

복제 구조는 **로그 역설계(자동) → 사용자 수정(개입)**이고, 둘은 *같은 표현* 위 한 슬라이더의
양 끝이다(대중적일수록 RE 비중↑, 비대중일수록 개입 비중↑). 로드맵은 이 루프를 중심에 둔다:

```
로그 → [역설계 자동추출] → 효과-스키마에 후보 복제본 → [사용자 수정 UI] → diff(로그 대조) → 반복
        (detection/regression           (공유 표현·키스톤)     (step2 매핑UI 연장)   (잔차 측정)
         를 메커니즘 축으로 연장)
```

- **diff는 최상위가 아니라 루프의 *계기*다.** 충실도를 측정하고, RE가 못 잡은 걸 사용자에게
  가리킨다.
- **효과-스키마는 키스톤이다.** 역설계가 채우고 사용자가 고치는 *같은 객체*. 슬라이더를
  메커니즘 축에서 물리적으로 구현하는 표현.
- **기존 척추를 재사용·연장한다.** detection.py(자동감지)·symbolic_regression(식 역설계)·
  step2 매핑UI(사용자 수정)가 이미 *수치·스탯 축*의 역설계→수정 루프다. 1차의 진짜 일은 이
  검증된 척추를 *메커니즘 축*까지 연장하는 것이고, 그러려면 메커니즘이 *쓰여질 대상*(스키마)
  으로 존재해야 한다.
- **엔지니어링의 몫은 둘로 하향.** (1) 스키마 *표현력*(언어 자체 — 드물게, 코퍼스가 "표현
  불가"를 가리킬 때만 확장), (2) RE 검출기 + 수정 UI(도구). 특정 게임의 *채움*은 엔지니어가
  아니라 RE+사용자가 한다.

## 2. 확정된 방향 (4점 + 가드)

1. **방향 맞다** — 역설계→수정 루프, 스키마 키스톤, 엔지니어링은 표현력+도구로 하향. (확신)
2. **스키마는 앞이 아니라 코퍼스에서 키운다** — 6개 원시만 보고 미리 설계하면 성급한 추상화
   (표현력 부족 → 엔지니어링 회귀, 또는 과표현 → 코드-위장). 실례에서 결정화.
   **가드(필수): 포켓몬에서 키우되, 추상을 *굳히기 전에* 구조가 다른 게임 로그 한둘로
   일반성 검증.** 포켓몬 하나로만 키우면 포켓몬 모양으로 굳어 다음 사용자(다른 게임)에게
   핵심 가치가 깨진다. 과적합 가드 = *교차-게임 로그 다양성.*
3. **메커니즘 자동RE는 보너스** — 수치 RE(회귀)는 입증, 메커니즘 RE(효과 이벤트 패턴 마이닝)
   는 미해결 연구. 제품 가치를 여기 걸지 않는다. 신뢰는 *스키마 + 사용자 수정 + 수치 RE*.
4. **첫 발은 A(하니스) + 코퍼스** — 측정과 실례가 스키마를 잘 설계하기 위한 전제. 불확실한
   조각(스키마)을 데이터가 쌓인 뒤로 미루는 순서. 첫 코퍼스는 1차 목표대로 포켓몬.

## 3. 지켜야 할 핵심 가치

전용 시뮬레이터가 없는 게임의 시뮬레이터를 만들 때, 맨땅에 헤딩보다 **10%라도 사용자 부담을
던다.** 로그가 항상 있으므로 이 경감은 세 출처의 합이다: **(a) RE가 로그에서 자동 추출하는
몫**(항상 0보다 큼) + **(b) 실행 substrate 공짜**(턴루프·자원·데미지 파이프라인·결정론 —
이미 입증, step2에서 로그로 게임 저작 가능) + **(c) 수정이 코딩보다 가벼움**(스키마 form,
단 코드-위장이 아닐 때).

- **가드**: 위 2-가드(교차-게임 로그)와 *수정 표면이 코드-위장이 아닐 것*(슬라이더는 정밀도
  연속이지 매체 점프 아님).
- **정직한 경계**: RE도 못 잡고 스키마로도 표현 안 되는 메커니즘 꼬리는 사용자가 거의 맨땅이고,
  매체 점프(플러그인 탈출구)를 부른다. 10%는 *흔한 substrate+표준 메커니즘*에 견고하고
  괴상한 꼬리엔 무조건이 아니다.

## 4. 단계 (코퍼스 구동)

- **A. 트레이스-리플레이 하니스 (첫 발).** 쇼다운 HTML(`<script class="battle-log-data">` 내
  `|`-프로토콜) → 정규화 트레이스(행동+관측결과+HP delta+플래그). 엔진 "트레이스 모드"(AI
  대신 관측 행동 주입) + 이벤트별 diff. 측정 토대이자 루프의 계기. 첫 마일스톤: *구현된
  메커니즘만 쓰는* 단순 전투로 데미지 코어를 못박는다(diff ≈ 0).
  - 숨은 스탯 역산: 로그에 EV/IV/성격 없음 → 데미지 식·종족값·관측 데미지에서 *back-solve*.
    이게 슬라이더의 역설계 끝이 실 데이터에서 도는 자리. 1급 서브시스템.
- **B. 효과-스키마.** 기존 원시(hazard/weather/status-gate/accuracy/multihit/boost)를
  `트리거×조건×효과×스코프×지속×확률` over 스탯·자원으로 접기. 회귀0(기존 config 재표현).
  코퍼스에서 키우고, **굳히기 전 교차-게임 로그로 일반성 점검**(2-가드).
- **C(흡수됨). 레퍼런스 데이터 — lazy, 코퍼스 구동.** "전체 도감"이 아니다. 로그가
  관측해주는 것(사용 무브·발동 도구/특성·상성결과·크리·최대HP)은 *답안지*. 엔진 독립 계산에
  필요한 것만 게으르게: 전역 상수(타입표·성격표·스탯식 — 작음) + 등장분 종족값/무브 한 줄씩
  + 등장 특성·도구(스키마 데이터). 배울거리·미등장 종족 불필요. D에 흡수.
- **D. 코퍼스 구동 커버리지 루프.** 실 싱글 리플레이 N개 → 트레이스 리플레이 →
  **diff 실패 = 미구현/미표현 메커니즘 신호 → 시스템 RE-추론 시도 → 사용자 확인/수정.**
  코퍼스가 커버리지 경계를 정의. (엔지니어 코딩 큐가 아니라 RE+사용자 루프.)
- **E. 남은 구조 갈래는 코퍼스가 부를 때.** 사전 목록이 아니라 *코퍼스 빈도순*으로 끌려옴.

### 5′. 마일스톤 갱신 — 효과 디스패처(키스톤) 복원 + 환경/무브 레이어 (2026-06)

**스키마 vs 데이터 분리(2-가드의 표적)**: `EFFECTS`/`CONTACT_MOVES`/`ABILITY_TYPE_IMMUNITY`·
타입표·스탯식은 *임의 gen5 전투에 일반*. `SETS` 12종만 *이 한 판에 역설계된 per-battle 데이터*.
→ **다음 게이트 = 교차검증(2-가드)**: 아직 맞춘 적 없는 held-out 리플레이에 일반 스키마를 던져,
divergence를 (ㄱ)일반 스키마는 맞는데 SET 데이터만 결측 / (ㄴ)스키마가 표현 못 하는 신규 메커니즘
/ (ㄷ)롤 노이즈로 분해한다. (ㄴ)의 수 = 과적합 평결. 한 판 잔여 폴리싱보다 이게 1차목표의 천장을
직접 잰다. (설계: `교차검증_설계안.md`.)

## 6. 첫 황금표준 메모 (업로드된 Kecleon전)

`OUMonotype-2014-...-kdarewolf-onox.html` 전체 파싱 결과: 양쪽 HP가 정확값(213/292 등 —
수치 검증 이상적), 닉네임→종족 해석됨, 이벤트 ~30종 표준. 단 *첫 코어 검증용으론 난이도 높음*
— **Substitute(⑤)+flinch+Recover+이중 Protean**을 끌어옴. → 이 전투는 코어 검증용이 아니라

## 8. 한 줄 요약

1차목표 = 임의의 실제 싱글 전투를 트레이스-리플레이로 수치까지 재현. 조직원리 = 역설계→
사용자수정 루프(diff=계기, 스키마=키스톤, 기존 detection/regression/매핑UI를 메커니즘 축으로
연장, 엔지니어링은 표현력+도구로 하향). 순서 = A(하니스)+포켓몬 코퍼스 → B(스키마, 굳히기 전
교차-게임 점검) → D(코퍼스 구동 커버리지, 데이터 lazy) → E(코퍼스가 부르는 구조 갈래).
메커니즘 자동RE는 보너스. 핵심 가치(맨땅보다 10%↓)는 RE추출+공짜substrate+가벼운수정에 살고,
교차-게임 로그·비-코드 수정표면으로 지킨다.
