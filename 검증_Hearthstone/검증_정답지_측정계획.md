# 검증 마일스톤 — Hearthstone式 카드 기반 덱빌더 (정답지 + 측정 계획)

## 0. 이 실험이 무엇인가

도구(전투로그 역설계 + 전문가 개입 → 복제 → 최적화)가 **monster-tamer(Pokemon)와
classic JRPG(FF7)를 넘어, 본질이 다른 장르인 card-based deckbuilder에서도 작동하는가**
를 검증한다. Pokemon은 페어드 엔티티 전투(크리처가 곧 단위), FF7은 1v1 캐릭터+자원
게이지, Hearthstone은 *덱이 엔티티이고 카드가 행동인* 게임 — 도구의 "entity =
numerical stats" 가정과 가장 멀리 떨어진 장르.

방식 — **known-answer 통제 실험** (Pokemon·FF7과 동일 패턴): Hearthstone(Blizzard 2014)
전투 규칙의 공개 사양을 *극단적으로 단순화한* 참조 구현(`hs_ref.py`)에서 로그를 생성.
도구는 *로그만* 보고 규칙을 역설계해야 한다. 정답을 우리가 알고 있으므로 회수 충실도를
정밀 채점할 수 있다.

본 reference는 100% 정확한 Hearthstone 시뮬레이터가 아니다 — **공식 룰 기반 핵심
메커니즘만 추린 단순화 버전**. 미니언과 영웅의 데미지 교환, 마나 자원, 카드의
Attack·HP·ManaCost 세 속성만 다룬다. 키워드(taunt, divine shield, charge, stealth,
freeze 등)와 카드 텍스트의 복잡한 효과(소환, 변신, 카드 드로우 트리거, deathrattle,
battlecry)는 *모두 deferred*. 단순화 가정은 §7에 일괄 명시.

이 실험의 진짜 질문은 두 가지다. 첫째, 도구의 "엔티티-퍼-로우" 데이터 모델이 *덱
프로파일을 엔티티로 간주*하는 추상화로 카드 게임을 어느 정도 다룰 수 있는가. 둘째,
Pokemon 66.2%·FF7 66.0%의 백테스트 정확도가 *장르가 본질적으로 다른* Hearthstone에서
얼마까지 유지되는가.

## 1. 정답 (Ground Truth) — 도구가 회수해야 할 규칙

정확한 코드 정의는 `hs_ref.py`. 요약:

### 1-1. 미니언-vs-미니언 트레이드 공식 (핵심 데미지 메커니즘)

```
A가 B를 공격하면:
  B.HP -= A.Attack
  A.HP -= B.Attack         (방어자도 반격 — 양쪽 동시 데미지)
  A.HP <= 0 → A 사망 (보드에서 제거)
  B.HP <= 0 → B 사망
```

Hearthstone의 전투는 *상호 트레이드*가 본질이다. Pokemon/FF7과 가장 다른 점은 이 곳:
방어자가 자동 반격하고, 데미지가 비선형 공식이 아니라 *공격력 그대로*다.

### 1-2. 영웅 공격 (헤로 데미지)

```
미니언 M이 적 영웅 H를 공격:
  H.HP -= M.Attack
  영웅은 반격하지 않는다 (무기 없는 단순화).
승리 = 적 영웅 HP <= 0
```

### 1-3. 스펠 데미지 카드

```
플레이어가 데미지 스펠 S를 캐스트:
  target.HP -= S.DamageAmount
  target ∈ {적 영웅, 적 미니언 (그리디 — 가장 위협적인 적 미니언)}
```

### 1-4. 카드 구성

```
미니언 카드: (ManaCost ∈ [1, 10], Attack ∈ [0, 12], HP ∈ [1, 12])
스펠 카드(데미지): (ManaCost ∈ [1, 8], DamageAmount ∈ [1, 10])
힐 카드: (ManaCost ∈ [1, 6], HealAmount ∈ [2, 10])

카드의 stat 합계는 ManaCost와 양의 상관관계:
  미니언 vanilla curve: Attack + HP ≈ 2 · ManaCost + 1
  스펠 vanilla curve: DamageAmount ≈ ManaCost + 1
이는 Hearthstone의 공식 vanilla baseline.
```

### 1-5. 턴·자원 규칙

```
Mana 크리스털 = min(turn_number, 10)
  매 턴 mana 풀이 turn_number만큼 회복 (1턴=1, 2턴=2, ..., 10턴 이후 항상 10)
영웅 HP 시작 = 30
한 턴에 mana 한도 내에서 카드 N장 플레이 (그리디 AI — ManaCost 큰 것부터)
턴 종료 후 자기 보드 미니언이 적 미니언/영웅 공격
보드 미니언 한도 = 7 (8번째부터 못 냄)
매 턴 덱에서 1장 드로우 (시작 핸드 4장)
```

### 1-6. AI 정책

```
플레이 단계 — mana 내림차순 그리디. 핸드 카드 중 ManaCost ≤ 남은 mana 인 것 중
  가장 비싼 것부터 낸다.
공격 단계 — 보드 미니언 내림차순 (ManaCost 큰 미니언부터 행동).
  타겟 선택 — 적 미니언이 보드에 있으면 가장 비싼 미니언과 트레이드, 없으면 영웅 공격.
승패 — 적 영웅 HP ≤ 0 이면 즉시 승. 50턴 초과 시 영웅 HP% 높은 쪽 승.
```

### 1-7. 덱 구성

30장 카드, 매 게임 시작 시 셔플. archetype별 카드 분포:

- **Aggro** — 저코스트 미니언 70%, 데미지 스펠 25%, 힐 5%
- **Midrange** — 중코스트 미니언 60%, 데미지 스펠 25%, 힐 15%
- **Control** — 고코스트 미니언 50%, 데미지 스펠 20%, 힐 30%

각 archetype의 mana curve와 stat 합계가 다르다 — 도구가 archetype 신호를 잡을지 보는
포인트.

## 2. 로그 파일 (도구 입력)

### `hs_battle_log.csv` — 5000행 (2500전투 × 2)

플레이어-per-전투. 각 행이 한 플레이어의 *덱 프로파일*과 그 게임 승패. 컬럼:

```
HeroHP, DeckAvgAttack, DeckAvgHP, DeckAvgManaCost,
DeckMinionCount, DeckSpellCount, DeckHealCount,
DeckTotalStatPower, DeckTempo,
StartHandSize, Archetype, Is_Victorious
```

- **HeroHP** — 30 고정. 도구가 *상수 컬럼*을 어떻게 처리하나 보는 포인트
- **DeckAvgAttack / DeckAvgHP / DeckAvgManaCost** — 덱 30장의 평균 (미니언만 대상)
- **DeckMinionCount + DeckSpellCount + DeckHealCount = 30** (제약)
- **DeckTotalStatPower** — 덱 전체 (Attack + HP) 합 (미니언만)
- **DeckTempo** — 평균 ManaCost의 역수 × 10 (낮을수록 느린 덱)
- **StartHandSize** — 4 고정
- **Archetype** — `Aggro` / `Midrange` / `Control` (카테고리)

→ Step 2 ML + Step 6 시뮬레이션 + Validation Loop 용. 의도된 도전 — 카드 게임의 진짜
엔티티는 "덱"이지 "유닛"이 아니므로, 이 행 각각이 *집계 통계*다. 도구가 집계 통계를
엔티티 stats처럼 처리해서 승률을 예측할 수 있는가?

### `hs_attack_log.csv` — 8000행

per-attack. 한 미니언-vs-미니언 트레이드 또는 스펠 캐스트:

```
Atk_Attack, Atk_HP, Atk_ManaCost,
Def_Attack, Def_HP, Def_ManaCost,
Card_Category, Damage
```

- **Card_Category** ∈ {Minion, Spell}. Minion이면 트레이드 공격, Spell이면 직접 데미지
- **Damage** — 방어자가 받은 데미지. *정답은 양 카테고리 모두 `Damage = Atk_Attack`*.
  Spell 행에서는 `Atk_Attack` 컬럼이 그 스펠의 DamageAmount를 담는다(의도적 인코딩 통일).
  Spell 행의 `Atk_HP` 컬럼은 의미 없는 노이즈(0 또는 임의값), `Atk_ManaCost`는 카드 가치
  지표지만 데미지와는 무관 — SR이 이 두 컬럼을 무시해야 정답에 도달한다.

→ Phase 7 Symbolic Regression 용. 정답은 *극단적으로 단순*: `Damage = Atk_Attack`.
방어자 stat이 데미지 식에 안 들어간다 (Pokemon·FF7과 정반대). SR이 이 단순성을 회수할
지가 핵심 측정.

1. `hs_battle_log.csv` 업로드 → Step 2
2. 스탯 `HeroHP, DeckAvgAttack, DeckAvgHP, DeckAvgManaCost, DeckMinionCount,
   DeckSpellCount, DeckHealCount, DeckTotalStatPower, DeckTempo, StartHandSize` /
   기믹 `Archetype` / 타겟 `Is_Victorious`

   탐사용)
4. 데미지 공식 시도 — `Damage = Atk_Attack` 단순 공식을 엔진 공식창에 입력 가능한가?
5. Step 2 프로파일링 → Step 6 시뮬레이션 → Validation Loop
6. 각 단계 화면 캡처 공유

### Run C — Per-battle backtest

1. `hs_battle_log.csv`를 Step 6 시뮬레이션 로그 탭의 🔬 전투별 백테스트 섹션에 입력
2. Pokemon 66.2% · FF7 66.0% 대비 Hearthstone은 어느 수준?

## 5. 사전 예측 (실행 전 기록 — 나중에 실제와 대조)

- **Phase 7 SR (Run A)** — 정답이 단순(`Damage = Atk_Attack`). SR이 회수하기 *쉬워야*
  한다. Card_Category 분기 없이도 단일 변수 회귀. **R² > 0.9 예상**. Pokemon·FF7과 정반대
  방향의 극단(너무 단순해서 회수). 단 Card_Category=Spell 행은 `Atk_HP·Atk_ManaCost`가
  무의미한 노이즈로 들어가 R²를 약간 깎을 수 있다.
- **ML (A)** — 10개 숫자 스탯 + Archetype 카테고리 → 승률. **DeckTotalStatPower·
  DeckAvgAttack·DeckMinionCount**가 양의 중요도. **HeroHP·StartHandSize**는 상수라
  0 중요도(*도구가 상수 컬럼을 어떻게 다루나*가 부가 관찰 포인트). **DeckTempo**는
  archetype에 따라 비선형
  관계 — ML이 선형으로만 잡으면 약한 신호. **DeckHealCount**는 Aggro 상대 음의 상관관계지만
  ML이 archetype 상호작용을 못 잡으면 무신호.
- **Archetype 채널 (Run B)** — Pokemon·FF7의 element 채널과 자연스러운 매핑이 없다. 도구의
  channel 매핑이 카드 게임에 *부적합한 추상화*임이 드러날 것. → **표현 불가 항목으로
  기록**.
- **표현 불가 (C)** — 카드 드로우 동역학(매 턴 1장), 핸드 관리(드는 카드의 운),
  마나 곡선(턴별 사용 가능 카드 폭), 보드 상태(트레이드 누적), 모든 키워드(taunt 등),
  영웅 능력. **이 모두가 deferred로 표시되고 X6(카드/덱 엔진 확장) trajectory의
  존재 근거가 된다**.
- **Damage 공식 (B)** — `Damage = Attack`은 엔진 공식창에 *너무 단순해서 표현 가능*하다.
  Atk 스탯 하나만 받는 공식 입력. 사용자 개입은 가벼움.
- **Per-battle backtest (D)** — 표현 불가 항목들 때문에 시뮬이 Hearthstone의 동역학을
  충실히 재현 못 함. 덱 프로파일이 엔티티 stats로 들어가서 시뮬레이션은 *집계 통계 기반
  스탯 비교*로 변질. 본질이 다른 추상화 위에서 돌아가는 셈. **per-battle backtest 50~60%
  예상** (Pokemon·FF7의 66% 대비 약 10%p 낮음). 50% 이상이면 도구의 robustness 증명에
  충분 (랜덤 50% 대비 유의미).

**예측 종합:** 도구는 *덱 stat 강도→승률 경향*은 부분 회수. 데미지 공식은 너무 단순해서
SR이 거의 완벽 회수. 카드 게임 동역학의 본질(핸드·마나 곡선·보드 상태·드로우 운)은
구조적으로 표현 불가 — *이게 X6 카드/덱 엔진 확장이 필요한 정확한 증거*. 슬라이더 원리
(사용자 개입 비중 증가)로 archetype 가중치 정도는 시연 가능하지만, 카드 게임의 진짜
복잡성은 데드라인 이후 X6에서 다룬다.

## 6. 결과 → 결정

- Per-battle backtest 일치율(D)이 50%+ → 도구의 *장르 일반화 능력*이 monster-tamer와
  JRPG를 넘어 deckbuilder까지 도달한다는 증거. 데모 narrative의 핵심 증거.
- Per-battle backtest 60%+ → 카드 게임도 entity-as-deck-aggregate 추상화로 의미 있게
  작동. 강한 증거.
- 표현 불가 목록(C)이 풍부 → **X6(카드/덱 엔진 확장) trajectory의 정당화 자료**. 후속
  trajectory 문서의 "왜 X6가 필요한가" 섹션 채움.
- SR이 `Damage = Atk_Attack` 회수 → SR이 *단순 공식에서도* 작동한다는 sanity check.
  Pokemon·FF7에서 R² 0.5~0.75였던 대비 단순 케이스의 정답 회수 능력 검증.

도구를 돌리고 각 단계 화면을 공유하면 채점표를 함께 채운다.

## 7. 단순화 가정 (Hearthstone 원작 대비)

본 reference는 다음을 단순화했다:

1. **1v1 영웅 대결**: 원작도 1v1. 변동 없음.
2. **키워드 전면 deferred**: taunt, divine shield, charge, rush, stealth, freeze,
   windfury, lifesteal, poisonous, reborn, immune 등 *모든 키워드 제외*. 단순 stat
   트레이드만.
3. **카드 텍스트 효과 전면 deferred**: battlecry, deathrattle, end-of-turn, start-of-turn,
   on-damage 트리거, 소환, 변신, 카드 드로우 트리거 *모두 제외*. 카드는 (Atk, HP,
   ManaCost) 세 숫자 그대로.
4. **클래스/영웅 능력 제외**: Mage Fireblast, Warrior Armor Up 같은 2-mana hero power
   제외. 영웅은 HP=30 빈 슬레이트.
5. **영웅 무기 제외**: 영웅은 직접 공격하지 않는다. 무기 장착 메커니즘 없음.
6. **마나 가속 메커니즘 제외**: Coin, Innervate, Wild Growth 같은 ramp 카드 없음. 마나는
   *순수히 turn_number*.
7. **드로우는 매 턴 1장만**: 카드 드로우 효과 카드(Arcane Intellect 등) deferred. 핸드
   고갈 시 fatigue 데미지 대신 *그냥 카드 못 냄*.
8. **보드 한도 7**: 원작과 동일. 8번째 미니언은 못 낸다.
9. **시작 핸드 4장, mulligan 없음**: 원작은 선공 3장 + mulligan, 후공 4장 + Coin. 본
   reference는 양쪽 4장 고정.
10. **Fatigue 제외**: 덱 떨어지면 그냥 카드 못 뽑음. 50턴 초과 시 HP% 비교로 판정.
11. **그리디 AI**: 플레이는 ManaCost 내림차순, 공격은 ManaCost 큰 미니언부터, 타겟은
    적 미니언 우선 (없으면 영웅). 실제 플레이의 전략적 깊이는 *전부 제거*.
12. **스펠 종류 단순화**: 데미지 스펠과 힐 스펠 두 종류만. AOE, transform, polymorph,
    silence, secret 등 모든 효과형 스펠 deferred.
13. **양방향 트레이드만**: 미니언 공격 시 양쪽이 동시에 데미지. Hearthstone 원작과
    일치 (이 점은 그대로 유지 — 핵심 메커니즘).
14. **클래스 카드 풀 통합**: 원작은 9개 클래스가 각자 카드 풀. 본 reference는 단일
    카드 풀에서 archetype 분포만 적용.

이 단순화는 의도적. 100% Hearthstone 시뮬레이터가 목적이 아니라, **도구가 card-based
deckbuilder 클래스 메커니즘(덱-as-엔티티 + 카드-as-액션 + 양방향 트레이드 + mana 자원)을
reverse-engineer 할 수 있는가**를 known-answer로 검증하는 게 목적.

검증의 타당성은 단순화의 충실도가 아니라 known-answer가 명확히 정의되었는가에 달려 있다.

가장 정직한 자기 비판: 이 단순화는 *Hearthstone에서 흥미로운 거의 모든 것*을 제거했다.
키워드와 카드 텍스트 효과가 Hearthstone의 본질이지만, 그것은 X6(카드/덱 엔진 확장)의
영역. 본 reference는 *도구가 카드 게임의 가장 단순한 골격이라도 다룰 수 있는가*만 검증.
표현 불가 항목(C)이 길어지는 것이 *정확히 의도된 결과*이며 X6 trajectory의 정량적 정당화
자료가 된다.
