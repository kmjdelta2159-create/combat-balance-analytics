# 검증 마일스톤 — Hearthstone式 카드 기반 덱빌더 · 결과 채점표

`검증_정답지_측정계획.md`의 측정 계획에 대한 실측 결과. 사용자가 도구를 라이브로 통과
시켜 측정. 도구 엔진 `modules/engine.py`는 무수정으로 사용.

---

## 0. 한 줄 요약

도구의 Per-Battle Backtest 충실도는 **47.2%**(236/500)다. Pokemon 66.2%·FF7 66.0%와
약 **19%p 격차**이며 동전 던지기(50%) 이하 영역. 그러나 이 결과 자체가 *trajectory
X6(카드/덱 엔진 확장)의 정량 정당화*다 — 도구의 장르 일반화 능력은 monster-tamer와
classic JRPG에 도달하고, card-based deckbuilder에는 도달하지 못함을 *정직하게 자기
측정*했다. 약점이 아니라 trajectory의 정당화 자료다.

추가 신규 발견: **ML 단일 매치업 예측 82.1% vs 시뮬 백테스트 47.2%의 35%p 간극**이
핵심 진단. 카드 게임에서는 도구의 ML과 시뮬 사이 다리가 끊어진다.

## 1. 실행 메타

| 항목 | 내용 |
|---|---|
| 입력 로그 | `hs_battle_log.csv` 5000행 · `hs_attack_log.csv` 8000행 (hs_ref.py 생성) |
| 실행 방식 | 사용자 라이브 — 도구의 Step 1~6 전체 흐름 |
| 데미지 공식 | `DeckAvgAttack` (스펙 §1-1 양방향 트레이드의 덱 집계 환원) |
| Type Effectiveness | 3×3 Aggro/Control/Midrange 매트릭스 — 모두 1.0 (스펙에 archetype 상성 없음) |
| STAB | 1.0 (스펙에 STAB 메커니즘 없음) |
| 결측치 처리 | 자동 — hs_ref.py가 결측 0건 보장 |
| 백테스트 표본 | 500전투 (도구 기본값) |

### 1-1. 흥미로운 자동 동작 — Archetype 채널 미스매핑

도구가 Step 2에서 *자동으로* `Archetype` 컬럼을 ELEMENT 카테고리로 분류하고 3×3 type
effectiveness 매트릭스를 생성했다. 우리 ref엔 archetype 상성이 없으므로 1.0 그대로
두는 것이 옳지만, 이는 *도구의 channel 매핑이 카드 게임에 부적합한 추상화임을 자동
노출*한 좋은 사례. element 채널 자체가 monster-tamer/JRPG 가정의 산물이고, deckbuilder
엔 element 같은 직교 카테고리가 없다. 도구는 이 미스매치를 *모르고* fallback으로 channel
배선을 만들어버린다 — 사용자가 1.0 유지를 선택해야만 spec과 일치.

## 2. 채점표

### A — 역설계 자동 회수

| 경로 | 결과 | 판정 |
|---|---|---|
| ML 가치 기여도 (Step 5 Top 5) | 1위 DeckAvgAttack · 2위 Archetype_Aggro · 3위 DeckMinionCount · 4위 Archetype_Midrange · 5위 DeckAvgHP | **성공** — spec §5 예측대로 |
| 예측 승률 (Logistic Regression 단일 매치업) | 82.1% (Ally 강한 라인업 vs Enemy 빈) | **성공** — 강한 ML 신호 |
| 시뮬레이션 백테스트 | 47.2% (집계 통계로는 동역학 재현 불가) | **실패** |

ML이 *집계 통계로부터 승패를 예측*하는 능력은 강하지만, *그 ML 신호를 엔진 시뮬레이션
으로 변환*하는 다리가 카드 게임에서 무너진다. 82.1%(ML)와 47.2%(시뮬)의 **35%p 간극**이
핵심 진단이며, Pokemon·FF7에서는 관찰되지 않은 신규 실패 모드다.

### B — 개입 필요량

| 개입 | 정도 | 비고 |
|---|---|---|
| 컬럼 매핑 (스탯/기믹/타겟) | 자동 — 사용자 개입 없음 | ML이 12개 컬럼 모두 정확히 분류 |
| 데미지 공식 직접 작성 | 경 — `DeckAvgAttack` 한 변수 | spec §1-1과 직접 일치, 극단 단순 |
| Type Effectiveness | 경 — 1.0 기본값 유지 | 도구 자동 생성 매트릭스 그대로 |
| Channel 매핑 | 자동 (단 *부적합한 추상화*) | §1-1 참조 |
| 결측치 처리 | 자동 — 0건 | hs_ref.py가 보장 |

슬라이딩 스케일 상 Hearthstone은 **저개입 끝단**. 사용자가 데미지 공식 한 줄과 클릭
몇 번 외 사실상 개입 없음. *그런데도 47.2%* — Pokemon의 "고개입에도 71.6% 천장"과
정반대 모드의 실패. 개입을 늘려도 해소되지 않을 종류의 한계.

### C — 표현 불가 (★ X6 trajectory의 정량 정당화)

| # | 항목 | 왜 불가능한가 |
|---|---|---|
| 1 | 카드 드로우 동역학 | battle_log 한 행은 *정적 덱 프로파일*. 매 턴 드로우의 운 요소가 집계로 압축 시 사라짐 |
| 2 | 핸드 관리 | 시작 4장 + 매 턴 1장. *어떤 카드가 언제 손에 들어오는가*의 sequencing이 결과를 좌우 |
| 3 | 마나 곡선 동역학 | 턴별 사용 가능 mana = min(turn, 10). 저코스트 vs 고코스트의 시점 매치가 핵심 |
| 4 | 보드 상태 누적 | 매 턴 트레이드 누적이 다음 턴 상태 결정. Markov chain. 도구는 stateless aggregate만 봄 |
| 5 | 키워드 시스템 전체 | taunt·divine shield·charge·freeze 등. spec에서 deferred 명시 — 본 검증엔 없음이지만 실제 카드 게임의 본질 |
| 6 | 카드 텍스트 트리거 효과 | battlecry·deathrattle·on-hit 등. 마찬가지로 deferred. 실제 게임의 50% 이상 차지 |
| 7 | 양방향 트레이드 비대칭 | A→B 데미지 = A.Attack, A 데미지 = B.Attack. *두 stat이 동시에* 결정. 도구는 한 공식만 입력 가능 |
| 8 | 영웅 HP=30 고정의 의미 | 모든 행에서 30 상수 → ML에서 0 중요도. 그러나 시뮬에서는 영웅 HP가 *승리 임계*. 상수의 의미가 시뮬로 전달 안 됨 |

**공통 뿌리**: 카드 게임의 본질은 *시퀀스 결정 게임* — 결과는 한 줄의 "덱 stat 비교"가
아니라 *턴별 의사결정 시퀀스의 누적 확률*이다. 도구의 entity-vs-entity 시뮬은 *덱을 한
entity로 간주*하는 추상화에서 멈춘다. **이것이 X6(카드/덱 엔진 확장) trajectory의 정확한
영역이다.**

### D — 시뮬 일치율 (충실도)

| 측정 | 값 |
|---|---|
| Per-Battle Backtest 전체 일치율 | **47.2%** (236/500) |
| ─ Ally 승 정확도 | 59.8% (실제 Ally 승 241건 중 적중) |
| ─ Enemy/무승부 정확도 | 35.5% (실제 비-Ally 승 259건 중 적중) |
| 도구 ML 단일 매치업 예측 | 82.1% — *ML은 작동, 시뮬이 무너짐* |
| 비교: Pokemon Per-Battle Backtest | 66.2% (1655/2500) |
| 비교: FF7 Per-Battle Backtest | 66.0% (330/500) |

Ally 59.8% vs Enemy 35.5%의 **24.3%p 비대칭**이 결정적. 도구가 *공격적 강점 덱*(주로
Aggro)은 잘 식별하지만 *방어/회복 중심 덱*(Control)은 못 식별한다는 명확한 신호. 우리
ref 시뮬에서 Control이 44.1%로 약했고, 도구가 그 약함을 학습했지만 *왜 약한지의
동역학*은 못 잡았기에 단순 stat 비교에 의존하면서 비대칭이 발생. 한 줄 결론: 도구는
"강한 덱이 이긴다"는 모델을 학습했지만 "조건부로 강한 덱이 이긴다"는 모델을 학습하지
못했다.

## 3. 핵심 발견 — ML/시뮬 간극

```
 47.2% ──────────── 시뮬 백테스트 (집계 통계 → 동역학 재현 시도)
   │  +34.9%p ← 카드 게임 동역학을 집계 통계로 압축 시 손실
 82.1% ──────────── ML 단일 매치업 예측 (집계 통계 → 직접 회귀)
   │  + (상한) ← 진짜 카드 게임 충실 시뮬은 더 풍부한 정보 필요
100.0%
```

Pokemon은 "엔진이 typeless 천장(75.3%)까지 잘 쓰고 그 위로 못 올라간다"는 *한계의
위치가 명확*했다. Hearthstone은 "ML은 강한데 시뮬은 동전 던지기 이하"라는 *질적으로
다른 종류의 실패*다. 카드 게임에서는 *집계 → ML*은 작동해도 *집계 → 시뮬*은 작동하지
않는다. ML이 잡은 상관관계 신호를 엔진이 동역학으로 재현할 수 없다. 이 간극을 메우는
것이 X6의 정확한 작업이다.

## 5. 결정 — trajectory에 미치는 함의

정답지 §6 분기 판정:

- **Per-Battle Backtest 50% 이상**? 47.2%로 **하단 약간 벗어남**. 도구의 robustness가
  monster-tamer·JRPG에서 멈춘다는 정직한 측정. 50%를 넘지 못한 것은 *집계 통계로 카드
  동역학을 재현하려는 시도 자체의 구조적 실패*이며, 단순 공식 보정으로 해결 불가.
- **표현 불가 목록(C)이 풍부**? 8개 항목 확인. **X6(카드/덱 엔진 확장) trajectory의 정량
  정당화 자료 풍부**. 이 중 1~4는 deckbuilder 본질이고, 5~6은 실제 Hearthstone의 50% 이상
  콘텐츠이며, 7~8은 도구 추상화 자체의 카드 게임 부적합.
- **신규 발견: ML/시뮬 간극** → 단순히 "메커니즘 추가"가 아니라 *엔진 추상화 자체의
  변경*(entity-vs-entity → sequence-of-decisions)이 필요함을 시사. X6의 범위가 처음
  예상보다 깊다.

**데모 narrative 함의**: 세 게임의 일관된 측정(Pokemon 66.2%, FF7 66.0%, Hearthstone
47.2%)이 *도구의 장르 경계*를 정량으로 그려준다. 47.2%는 약점이 아니라 *trajectory의
정당화*. "동일 도구가 두 장르에서 66%대 정확도, 한 장르에서 47%로 떨어진다. 떨어진
지점이 정확히 X6이 다룰 영역"이라는 프레이밍이 데모에 강력하다. 도구가 *자기 한계를
정직하게 측정할 수 있다*는 메타 능력 자체가 검증 결과다.

## 6. X6 trajectory 우선순위 영향

본 검증이 X6의 *영역 정의*를 구체화한다:

1. **턴별 시퀀스 결정 시뮬** — 핸드·보드·덱 상태 머신. entity-vs-entity 모델 폐기 또는
   확장이 필요. 가장 큰 작업.
2. **마나 곡선 동역학** — 턴별 사용 가능 자원 vs 카드 cost. 자원 시스템 (X1의 5축 중
   condition + modifier)을 카드 게임 맥락으로 특화.
3. **카드 드로우 확률 모델** — 시작 핸드 + 매 턴 드로우의 분산. 결정론 시뮬에서 확률
   시뮬로의 전환.
4. **양방향 트레이드 공식 슬롯** — 두 stat이 동시에 결정되는 데미지 식. 현재 엔진의 단일
   global_damage_formula 슬롯의 한계.

X6는 단순히 "카드 메커니즘 추가"가 아니라 *엔진의 시뮬레이션 모델 자체*를 entity-vs-entity
에서 sequence-of-decisions로 확장하는 작업이며, 본 검증이 그 범위와 정당화에 결정적
자료를 제공한다.

## 7. 다음 검증 작업 (후속)

데드라인 데모 후 X6 진입 시 본 검증 자산이 *retest 기준*이 된다:

- 동일 hs_battle_log.csv·hs_attack_log.csv를 X6 적용 후 도구에 통과시켜 Per-Battle
  Backtest 재측정.
- 목표: 47.2% → 60%+ 진입. 60% 이상이면 X6가 카드 게임 도메인을 도구의 적용 범위로
  편입시켰다는 증거.
- 추가 회귀 보호: Pokemon·FF7 backtest가 66.2%·66.0%에서 떨어지지 않는지 동시 확인.
  X6 변경이 기존 도메인을 깨지 않아야.
