# 풀배틀 트레이스 리플레이 설계안 (A 하니스 완성 — 루프 닫기)

> A 하니스의 마지막·최대 조각. 지금까지는 *per-event*(깨끗한 데미지 1건씩) 검증이었다.
> 이건 엔진을 *전투 전체* 동안 로그대로 구동해, 효과·해결·턴순서·교체·상태진화를 *한 번에*
> 실 로그와 대조한다. "복제"의 골드 스탠다드. 고립 핵심은 클린룸 검증됨, 전체 실행은 앱 사이드.

## 0. 검증 제약 (먼저 — 작업 방식이 바뀜)

**engine.py가 74KB(~1595줄)를 넘어 샌드박스 bash/python이 잘라 읽는다** — import/실행 불가
(E1/E2 적용으로 길어진 결과; Read/Grep은 진실 출처라 정상). 따라서:
- **전체 실행 검증은 사용자가 앱에서** 한다(엔진 온전한 환경). 로그를 붙여주면 함께 점검.
- **고립된 신규 조각**(트레이스→행동큐·상태스냅샷·divergence·셋업 빌더)은 작은 모듈이라
  클린룸/standalone 검증 가능 — 아래 §4 검증됨.
- 엔진 수정 PR은 FIND/REPLACE 앵커를 Read/Grep으로 잡고, 적용 후 *앱에서* 동작 확인.

## 1. 목적·구조

로그의 행동 시퀀스를 엔진에 주입해 전투를 재생하고, 라운드별로 엔진 상태를 로그와 대조한다.

```
trace ──▶ [셋업] 양팀 participants(reference 스탯/세트/효과) + game_config
      ──▶ [행동 주입] 트레이스 구동 selector(MOVE/TARGET) + 자발교체 주입
      ──▶ [엔진 턴 루프 실행] (기존 SequentialTurnManager — 모든 페이즈·효과 발동)
      ──▶ [상태 비교] 라운드별 엔진 HP/상태 vs 로그 스냅샷 → divergence 리포트
```

per-event와 달리 엔진의 *턴 스케줄러·전 페이즈*를 그대로 쓴다 — 그래서 효과(E1/E2 디스패처)·
날씨·해저드·상태·부스트가 *실제 전투 흐름에서* 검증된다.

## 2. 네 하위 문제

**(a) 셋업 — trace → 전투 구성.** 양팀 팀프리뷰(meta.teams) + 등장 세트 → reference로
participants(스탯·ability/item·resources) + game_config(formula·type_table·mechanisms.effects).
PR3/PR4 재사용 + 약간 확장(양팀 전체·resources 필드).

**(b) 행동 주입 — 로그대로 행동.** 트레이스 구동 selector를 MOVE_SELECT/TARGET_SELECT에
오버라이드 등록(A설계안 §3) — 계산 대신 *현재 라운드 그 유닛의 로그 행동* 반환. **자발교체
(이 로그 24회)**는 신규 표면: 엔진 교체모델이 *기절 구동*만 있으니, 로그의 voluntary switch를
"행동" 으로 주입하는 교체 액션이 필요(교체모델 내부 — 갈래 관련).

**(c) 턴순서 — 엔진 스케줄 vs 로그.** 엔진은 속도로 정렬·우선도 재정렬. 두 길:
- *검증형*: 엔진이 스케줄한 순서를 로그 순서와 대조(우선도·속도·타이가 맞는지 = 턴순서도 검증
  대상). 불일치 시 divergence.
- *강제형*: 로그 순서를 그대로 강제 주입(순서 검증 포기, 결과만 검증).
권고는 *검증형*(턴순서 정확도도 복제의 일부) — 단 우선도 무브·Hidden Power 타입·속도 타이가
얽혀 초기엔 강제형으로 결과부터 닫고 검증형으로 올리는 단계적 접근.

**(d) 상태 비교 — divergence 리포트.** 라운드별 엔진 상태(HP/status/weather/faint) vs 로그
스냅샷. 허용구간(데미지 롤) 내면 일치, 벗어나면 divergence + 원인 추정(미모델 효과·세트 prior·
턴순서). per-event 잔차 리포트의 전투 버전.

## 3. 정직한 난점

- **자발교체 24회**(이 로그) — 엔진에 voluntary switch 주입 표면이 없다(신규). 교체모델 수술.
- **턴순서 매칭** — 우선도 브래킷·속도 타이·Hidden Power 타입(숨김)·Protean 타입변경이 순서/
  상성에 얽힘. 초기엔 강제형으로 우회.
- **상태 진화 누적** — 부스트·날씨 지속(gen5 무한 vs gen6 5턴)·맹독 누진·Substitute가 라운드를
  넘어 누적 → 한 곳 틀리면 이후 divergence 연쇄. 그래서 *라운드별* 비교로 첫 divergence 지점을
  격리하는 게 중요.
- **검증이 앱 사이드** — 전체 실행을 내가 못 돌린다(엔진 truncation). 사용자 앱 실행 로그로 대조.

## 4. 클린룸 검증됨 (고립 조각)

gen5 로그로 standalone 확인:
- **행동 큐**: 60개(move 36 + 자발교체 24) — 엔진이 따라야 할 시퀀스 추출 정상.
- **상태 스냅샷**: 28턴 — 라운드별 HP 비교 기준 추출 정상.
- **divergence**: 순수 함수, 모의 −5 오차를 정확히 포착.
이 셋은 엔진 없이 돌아가는 *비교 인프라*라 클린룸으로 완결 — 앱 실행분과 대조할 기준이 선다.

## 5. PR 분할

- **PR-B1 (고립·클린룸 가능)**: 트레이스 → 행동큐·상태스냅샷·divergence 모듈(`fullbattle_diff.py`).
  비교 인프라. 엔진 무관 — 지금 검증·납품 가능.
- **PR-B2 (reference 확장)**: 셋업 빌더 — trace 양팀 → participants(+resources·ability/item) +
  game_config(+effects). reference_gen6 확장. 클린룸 검증.
- **PR-B3 (엔진)**: 트레이스 구동 selector(MOVE/TARGET 오버라이드) + 자발교체 주입. 엔진
  FIND/REPLACE. *앱에서* 동작 확인.
- **PR-B4 (통합)**: 전체 run + 라운드별 divergence 리포트. 앱 실행, 결과 함께 점검.

권고 순서: B1(비교 인프라, 지금 가능) → B2(셋업) → B3(엔진 주입) → B4(통합·앱 검증).

## 6. payoff
한 번 서면 *임의의 실 전투*를 넣어 효과·해결·턴순서·교체를 라운드별로 대조하는 골드 스탠다드가
선다. 잔차/divergence가 미모델 메커니즘·세트를 라운드 단위로 지목 → 역설계→수정 루프가 전투
스케일로 작동. 1차목표("실 전투를 수치까지 재현")의 합격 시험 그 자체.
