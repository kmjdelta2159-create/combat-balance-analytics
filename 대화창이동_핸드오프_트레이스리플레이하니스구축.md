# 대화창 이동 핸드오프 — 트레이스-리플레이 A 하니스 구축 + 효과-스키마 진입 + 풀배틀 설계까지

새 대화창에 이 문서를 붙여넣고, 직전 핸드오프(`대화창이동_핸드오프_날씨상태이상명중다중hit완료.md`)와 함께 읽어라. 본 문서는 그 위에서 **1차목표를 재정의하고 트레이스-리플레이 검증 하니스를 새로 구축**한 결과를 기록한다. 역할·제약·방법론은 직전 핸드오프 그대로다: **코드 직접 수정 금지·Antigravity 프롬프트 산출·검증 의무·시간 추정 금지·시연 언급 금지·도구 위치 과대표현 금지·말투 prose 우선·갈래는 항상 최종목표 기준으로 사용자에게 묻고 결정받기.**

## 0. 이번 phase의 큰 그림 — 1차목표 재정의 + 새 검증 축

직전까지는 "6갈래(명중/필드/상태이상/멀티/Substitute/진입KO)를 닫으면 Pokemon 복제"라는 *메커니즘 존재 체크리스트*였다. 이번 phase에서 사용자와의 긴 전략 대화로 **이 프레임을 갈아엎었다.** 핵심 합의(전부 사용자 결정):

* **최종목표(사용자 진술)**: 전투로그 기반 범용 턴제 전투 시뮬레이터. ①로그 역설계 → ②전문가(사용자) 개입으로 전투 시스템 복제 → ③밸런스 분석용 최적화. **연속 슬라이더**(대중적 게임일수록 역설계 비중↑, 비대중일수록 사용자 개입 비중↑ — 둘은 한 슬라이더의 양 끝). 조건: 전용 시뮬레이터가 없는 게임이라도 *전투 로그만 있으면* 개입 비중 조절로 복제 가능. 슬라이더는 정밀도의 연속이지 매체 점프(form→Python class)가 아니다. 스코프: 스탯기반 턴제(JRPG·가챠·SRPG·덱빌더), 비-스탯(체스·2048)은 의도된 도메인 밖.
* **1차목표 = (나) 트레이스-리플레이 충실도**: "6갈래 닫기"가 아니라 **"임의의 실제 싱글 전투(쇼다운 리플레이)를 넣으면 엔진이 관측 행동을 그대로 먹고 결과를 수치까지 재현"**. AI 의사결정 복제는 스코프 밖(로그가 행동을 이미 알려줌 → 주입). 싱글 먼저, 더블 보류(사용자 추후 결정).
* **조직 원리 = 역설계→사용자수정 루프**: diff는 계기, **효과-스키마가 키스톤**(역설계가 채우고 사용자가 고치는 같은 객체). 기존 척추(detection.py 자동감지·symbolic_regression 식 역설계·step2 매핑UI 사용자수정)를 메커니즘 축까지 *연장*. 엔지니어링 몫은 둘로 하향 — (1) 스키마 *표현력*(언어), (2) RE 검출기+수정 UI(도구). 특정 게임의 *채움*은 RE+사용자.
* **메커니즘 자동RE는 보너스**, 신뢰는 스키마+사용자수정+수치RE. **스키마는 코퍼스에서 키운다**(성급한 추상화 금지) + **교차-게임 가드**(비-포켓몬 로그로 굳히기 전 일반성 점검).
* **세대 구분 = 레퍼런스 데이터 교체일 뿐**(하니스 세대중립). 세대 *감지*는 안 짓는다(로그가 `|gen|N` 선언 + 전문가 사용자가 앎). "룰셋 감지"는 틀린 프레임 — 선언 읽기 + 수치RE + 사용자 공급.
* **핵심 가치**: 전용 시뮬레이터 없는 게임을 복제할 때 맨땅보다 ≥10% 부담 경감 — RE 자동추출 + 공짜 실행 substrate + 코딩보다 가벼운 수정. (substrate는 이미 입증. 잔여 꼬리는 RE불가+스키마불가 영역.)

재정리 로드맵: `복제완성_재정리_1차목표로드맵.md`(이번 phase에 (나) 기준 확정형으로 전면 갱신). `복제완성_Trajectory_6갈래.md`는 *능력 목록*으로 강등.

## 1. 구축한 것 — 트레이스-리플레이 A 하니스 (PR1~4, E1~E2)

전부 한글 프롬프트 .md 직접 작성 + 되읽어 추출 + 클린룸/standalone 검증. **사용자 적용 완료분**과 **신규 모듈**:

* **PR1 — `modules/showdown_trace.py` (신규, 적용됨)**: 쇼다운 HTML/raw 프로토콜 → 정규화 트레이스 IR. `parse_replay`(meta·nick2species·events) + `clean_damage_events`. **핵심: `-damage`를 `[from]` 유무로 분리**(없음=무브 직격/검증타깃, 있음=부수=날씨/해저드/상태/도구). 발효 날씨·공격자 상태를 move `context`에 스탬프(깨끗한 이벤트 필터). HP 상태 접미사(tox/brn) 파싱. 검증: Gen6전 깨끗한 직격 9·Gen5전 3(날씨 일찍 발효).
* **PR2 — `modules/trace_replay.py` (신규, 적용됨)**: `TraceStochasticity`(관측 hit/crit 재생, StochasticityModule 상속) + per-event 해결 하니스. **턴 스케줄러 우회 — 각 (공격자·무브·방어자) ctx로 해결 3페이즈(DAMAGE_CALC→ELEMENT_MULT→CRIT_CALC)만 구동, dmg 대조.** selector 오버라이드 불필요(ctx 직접 세팅). 데미지 롤(0.85~1.0) 미관측 → 허용구간 [0.85R,R] 판정.
* **PR3 — `modules/reference_gen6.py` (신규, 적용됨)**: gen6 종족값·무브·타입표 + `DAMAGE_FORMULA`(엔진 언어, base=`floor(floor(42*power*offense/defense)/50)+2`, ELEMENT_MULT가 STAB×타입, CRIT가 ×1.5) + `make_char`(종족값+표준세트 prior로 스탯) + `residual_report`. **부스트 추적(트레이스 boost 누적)·KO 하한 인지·종족값 자가검증(관측 maxHP)**. 결과(Kecleon전): **match 4 / ko_lowerbound 2 / residual 3**.
* **PR4 — `reference_gen6` 효과 레이어 (FIND/REPLACE, 적용됨)**: `ABILITIES`/`ITEMS`(항시형 정적 스탯 배율) + `SETS` + `apply_effects`. **Diggersby Huge Power(×2)로 U-turn 잔차 폐쇄**(관측 192 ∈ 식 [188,222]). 역설계→수정 루프 한 사이클 완주(잔차 지목→보정 인코딩→폐쇄). Eviolite는 EV-도구 모호성으로 미할당(정의만).
* **PR-E1 — `engine.py` 효과 디스패처 (FIND/REPLACE, 적용됨)**: `_act_on_hit` 끝에 `_act_effect_dispatch(ctx,"ON_HIT")` 호출 + 디스패처 4함수(`_eff_get_res`/`_eff_scope`/`_eff_cond_ok`/`_act_effect_dispatch`). 발동형 효과를 *데이터*(`mechanisms['effects']`)로 적용. 효과타입 `damage_frac`, 조건 `contact`. **검증: Garchomp(Rough Skin+Rocky Helmet)가 Hippowdon 접촉 피격 → 420→368→298 관측 정확 재현(−52=1/8, −70=1/6).**
* **PR-E2 — `engine.py` 디스패처 확장 (FIND/REPLACE, 적용됨)**: `heal_frac` 효과타입 + `weather` 조건 + `_act_turn_end_heal` 맨 앞에 `_act_effect_dispatch(ctx,"ON_TURN_END")`. **검증: Rain Dish 비일 때 +24(1/16), 모래 미발동, 풀피 캡.**

**임계점 도달**: 디스패처가 2 효과타입(damage_frac·heal_frac) × 2 트리거(ON_HIT·ON_TURN_END) × 2 조건(contact·weather)을 갖춰, **새 효과가 코드 0줄·데이터만으로 적용**된다(예: Life Orb 반동 = ON_HIT에서 공격자 item 자동 점검 → 스키마 한 줄, 엔진 무변경. standalone 검증됨).

* **PR-A2 — `step2_system_definition.py` move_props UI (프롬프트 작성, 적용 여부 미확인)**: 이번 phase 시작 시 직전 갈래(명중/다중hit) UI 경로 닫으려 작성. 전략 재정의로 초점이 트레이스-리플레이로 이동하며 적용 확인 안 됨. `명중률다중hit_PR-A2_UI_프롬프트.md`. 필요 시 적용 점검.

## 3. 효과-스키마 (키스톤, 설계+첫 두 PR)

* **demand 카탈로그** (`효과스키마_demand카탈로그.md`): 2전투에서 발동형 메커니즘 추출 → 셋으로 분류. **엔진 이미 커버**(regen·Protean·sandstorm·status틱·status부여+게이팅), **Tier-3 갭**(SR 타입스케일·맹독 누진), **미구현 발동형 클러스터**(flinch·Rough Skin·Rocky Helmet·Life Orb·Rain Dish·Wish·Protect·Substitute·Trick — ~10개가 *공통 shape*). 1battle 아닌 **2전투·2세대**에서 shape 반복 확인 → 결정화 정당화.
* **설계안** (`효과스키마_설계안.md`): 통합 shape = **트리거 × 조건 × 효과 × 스코프 × 출처 × 확률**. 트리거=엔진 기존 페이즈(ON_HIT/ON_TURN_END/ON_MOVE_USE/ON_SWITCH). 효과타입=자원/스탯 원자(damage_frac·heal_frac·block_action·stat_mult·후속 create_substitute/swap_item). **디스패처가 기존 4적용기(status_tick·weather_chip·action_gate·on_hit)를 일반화** — 단 기존은 *지금 리팩터 안 함*(회귀0 우선, 통합은 의도된 후속). 첫 인코딩: flinch·Rough Skin. **교차-게임 가드: shape는 포켓몬 2세대까지만 확인, 비-포켓몬 로그로 굳히기 전 점검(잠정 결정화).**
* PR-E1·E2 위 §1. **다음 발동형**: flinch(block_action+action-gate 통합 — 무브가 효과소유자라 무브-소스 차원 필요, 덜 깨끗), Life Orb(코드0줄·데이터), Protect/Substitute/Wish/Trick(새 효과타입, 빈도순).

## 4. ★중요 운영 변화 — 엔진 truncation 하드 블록

**engine.py가 74KB(~1595줄)를 넘어 샌드박스 bash/python이 잘라 읽는다**(build_ctx 중간에서 끊김 → 미완성 문자열 SyntaxError). E1/E2 적용으로 길어진 결과. **`import modules.engine`/`ast.parse`가 이제 샌드박스에서 불가**(거짓 SyntaxError/AttributeError). 세션 초반엔 엔진이 짧아 import가 됐으나 지금은 안 된다.

* **Read/Grep이 진실 출처** — 함수 위치·완결성은 Read/Grep으로 확인(E1/E2 정상 적용 확인됨: `def _act_effect_dispatch` 모듈레벨 834~, build_ctx가 L1590으로 밀림). bash의 SyntaxError·import 실패는 **truncation 거짓양성**.
* **엔진 실행 검증은 이제 클린룸/standalone 또는 *사용자 앱***. 디스패처 등 고립 로직은 standalone 사본으로 검증(이번에 그렇게 함). 전체 엔진 실행이 필요한 검증은 사용자가 앱에서 돌려 로그 붙여주면 함께 점검.
* reference_gen6·trace_replay·showdown_trace·stochasticity는 작아서(<10KB) 샌드박스 정상. engine.py만 초과.
* (고려거리, 미결: engine.py 분할 리팩터로 74KB 미만 복귀? 큰 위험작업이라 보류.)

## 5. 정직한 위치 / trajectory

per-event 검증으로 "포켓몬 싱글 전투가 수치까지 재현되나"가 — 깨끗한 직격(match 4/KO하한 2/잔차 3) + env 스트림(38/38) 으로 *대부분 yes*. 잔차/갭은 발동형 클러스터(효과-스키마로 흡수 중) + Tier-3(SR 타입스케일·맹독 누진). 효과-스키마가 임계 넘어 **새 효과 = 데이터**. 단 **효과는 엔진 턴루프(ON_HIT/ON_TURN_END)에서 발동하는데 현 하니스는 per-event라, *실 전투 흐름* end-to-end 검증은 미완** → 풀배틀 리플레이 필요.

## 6. 다음 단계 — 사용자가 (b) 선택, PR-B1부터
