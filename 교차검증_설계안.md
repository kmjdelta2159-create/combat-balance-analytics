## 0. 왜 지금, 왜 한 판 폴리싱이 아니라 이것

한 골든 리플레이에 계속 맞추면 충실도 숫자는 오르지만 *그 숫자가 일반성을 보장하지 않는다*.
다음 사용자(다른 전투·다른 게임)에게 핵심 가치가 깨지는지는, 아직 맞춰본 적 없는 로그에 던져야
드러난다. 한 판 잔여 버킷(switch-in-turn·명중/Protect·세트정밀화)은 마진이 작고 점근적이며,
무엇보다 *같은 축*이다. 교차검증은 1차목표의 진짜 천장("임의 싱글")을 직접 잰다.

## 1. 스키마 vs 데이터 분리 (이 검증이 가능한 이유)

`reference_gen5.py`를 뜯으면 두 층이 깨끗이 갈린다:

- **일반 스키마(임의 gen5 전투에 적용)**: `EFFECTS`(Rough Skin·Iron Barbs·Rocky Helmet·
  Leftovers·Black Sludge·Rain Dish·sand·brn·tox[progressive]·Explosion·Self-Destruct·Trick)·
  `CONTACT_MOVES`·`ABILITY_TYPE_IMMUNITY`·`NATURES`·타입표·스탯식·디스패처(`_act_effect_dispatch`).
  이건 메커니즘 *언어*다 — 어느 gen5 전투에서나 같은 의미로 발동.
- **이 한 판에 역설계된 per-battle 데이터**: `SETS` 12종(이 leftiez 전투의 특정 포켓몬과 그
  특성·도구·EV). 다른 gen5 전투엔 *다른 포켓몬* → `SETS` 미커버.

하니스(`fullbattle_run.run_and_diff(trace, ref, ...)`)는 이미 레퍼런스 모듈에 대해 generic하다.
즉 일반 스키마를 그대로 두고 **새 trace + (필요 시) 새 SETS만** 갈아끼우면 held-out 실행이 된다.

## 2. 과적합 평결 루브릭 (divergence 3분해)

held-out 리플레이를 돌려 나온 per-event/라운드별 divergence를 셋으로 가른다:

- **(ㄱ) 일반 스키마는 맞는데 SET 데이터만 결측** — 그 유닛이 `SETS`에 없어 기본값으로 도는 탓.
  → *합격 신호.* 스키마는 일반화됐고, 비는 건 RE+사용자가 채울 per-battle 데이터층(로드맵 설계대로).
- **(ㄴ) 스키마가 표현 못 하는 신규 메커니즘** — `트리거×조건×효과×스코프`로 적을 수 없는 효과가
  로그에 관측됨(예: 미모델 무브효과·새 도구계열·필드 상호작용). → *과적합/표현력 부족 신호.*
  **(ㄴ)의 개수와 종류 = 평결의 핵심 수치.**
- **(ㄷ) 롤 노이즈(±15%)·세트 EV 미세차** — 데미지 굴림 미관측에서 오는 작은 hp 차이. → 무시.

판정: (ㄴ)가 0~소수면 일반 스키마는 견고 → 축을 일반화로 더 밀거나 데이터층 자동화로 진행.
(ㄴ)가 다수·다양하면 스키마가 gen5-leftiez 모양으로 굳고 있던 것 → 표현력 보강이 진짜 일.

## 3. 측정 메커니즘 (재사용)

- `run_b4.py` 패턴(데미지 delta 진단, 턴×유닛 셀, ★=구조적) + `fullbattle_run.format_report`
  (faint·status·큰 hp차 = 구조적 격리)를 **코퍼스 인자화**해 held-out HTML에 재사용.
- 엔진 풀런은 앱사이드(mnt가 수정 .py 절단). 클린룸으론 trace 파싱·SET 커버리지 점검·divergence
  분류 같은 *엔진 무관 순수분*만 선검증.
- 회귀0: 일반 스키마 무변경. held-out용으로 추가하는 건 trace + (선택) 새 SETS 데이터뿐.

## 4. 코퍼스 선택지 (사용자 결정 — 이게 다음 갈래)

| 선택 | 무엇 | 무엇을 검증 | 비용 |
|---|---|---|---|
| **A. 신선한 gen5 OU 싱글 리플레이** | 같은 gen·never-fit 1건 | gen5 일반 스키마가 *다른 포켓몬·세트*에 일반화되나. SET-결측 vs 신규메커니즘 분해가 가장 깨끗 | 낮음(리플레이 1건 확보 + SET 역산 또는 기본값) |
| **B. gen6 골든 스키마 포팅** | 이미 보유한 `OUMonotype-2014-...` | 세대 교차 + Substitute·Protean·Mega 신규메커니즘 | 높음(`reference_gen6`가 옛 백테스트 형태 → EFFECTS-스키마 포팅 + 신규 효과타입) |
| **C. 비-포켓몬 스탯턴제 로그** | JRPG/가챠/덱빌더 로그 | 로드맵의 *궁극* 2-가드(교차-게임). 스키마가 포켓몬 모양으로 굳었는지 | 중~높음(로그 소스 확보 필요·게임 상수 신규) |

**추천 = A.** 변수를 하나만 바꿔(전투만, 세대·게임 고정) 일반 스키마의 generalization을 가장
깨끗이 격리한다. B는 세대차와 신규메커니즘이 섞여 평결이 흐리고, C는 가장 강한 가드지만 로그
소스부터 정해야 한다. A로 스키마가 견고함을 1차 확인한 뒤 C로 올라가는 순서가 자연스럽다.

## 5. A 선택 시 실행 절차 (다음 산출 = run-프롬프트)

1. 신선한 gen5 OU 싱글 리플레이 HTML 1건 확보(사용자 제공 또는 fetch). 대타·변신·메가 적은
   *표준* 전투가 이상적(신규메커니즘 잡음 최소화 → 평결 깨끗).
2. `run_b4` 패턴을 코퍼스 인자화한 `run_xval.py`(앱사이드): 새 HTML + `reference_gen5` 일반
   스키마로 `run_and_diff` → 데미지 delta 진단 + format_report.
3. **SET 커버리지 리포트**: 새 전투 등장 종 중 `SETS` 미커버분 목록(이게 (ㄱ) 예상 출처).
4. divergence를 §2 루브릭으로 3분해 → (ㄴ) 신규메커니즘 후보 목록 = 평결.
